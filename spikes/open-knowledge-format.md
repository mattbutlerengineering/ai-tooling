# Spike: Open Knowledge Format (OKF) — what it is, and whether it fits this repo

**Issue:** #178 · **Date:** 2026-06-29 · **Outcome:** recommendation below (no adoption yet; optional follow-ups noted)

Answers the questions in #178: *What is the open knowledge format? How does it help? How should it be configured/used? What are the antipatterns? Why not just md?* — grounded in the published spec, and weighed against this repo's [markdown-as-source decision (ADR-0002)](../docs/adr/0002-catalog-source-of-truth.md).

## What it is

**OKF (Open Knowledge Format)** is a vendor-neutral open specification from **Google Cloud** (v0.1, June 2026) for representing a body of knowledge as **a directory of markdown files with YAML frontmatter** — authored by people, generated by agents, consumed by both, with no SDK or proprietary account required. It formalizes the "LLM-wiki" pattern into a portable interchange format. The spec lives in Google's public Knowledge Catalog repo.

Core model:

- **Knowledge bundle** — the unit of *distribution*: a self-contained directory tree of markdown (a repo, a tarball, or a subdir of a monorepo).
- **Concept** — the unit of *knowledge*: one markdown file (tangible asset like a table/API, or abstract idea like a metric/playbook). Its identity is its file path minus `.md` (`tables/users.md` → `tables/users`).
- **Frontmatter** — YAML delimited by `---`. Exactly one field is **required: `type`** (producer-chosen, unregistered, e.g. "BigQuery Table", "Playbook"). Recommended optional: `title`, `description`, `resource` (canonical URI), `tags`, `timestamp` (ISO 8601). Producers may add custom keys; consumers **must preserve unknown keys and tolerate unknown types**.
- **Body** — freeform markdown, conventional headings (`# Schema`, `# Examples`, `# Citations`).
- **Links** — markdown links (absolute `/path.md` recommended) are *untyped directed edges in a graph*; relationship meaning lives in the prose, not syntax. **Broken links are tolerated** (= docs not yet written).
- **Reserved files** — `index.md` (optional directory listing / progressive disclosure, no frontmatter) and `log.md` (optional human changelog by ISO date, separate from git).
- **Conformance** — a bundle conforms if every non-reserved `.md` has parseable frontmatter with a non-empty `type`. Consumers must **not** reject for missing optional fields, unknown types, broken links, or absent `index.md`. Version via `okf_version: "0.1"` in the root `index.md`.

Design principles: **minimally opinionated** (only `type` required), **producer/consumer independence** (swap tooling at either end), **format-not-platform** (no lock-in). It *references* domain schemas (Avro/Protobuf/OpenAPI) rather than replacing them.

## How it helps

The problem: AI-relevant knowledge is scattered across catalogs (proprietary APIs), wikis, shared drives, docstrings, and people's heads — so "every agent builder solves the same context-assembly problem from scratch, every catalog vendor reinvents the same data models." OKF gives agents one portable, human-readable, agent-parseable substrate for curated context:

- **Same file for humans and agents** — no translation layer; readable in any editor, parseable without an SDK.
- **Portable with context intact** — survives moving between systems/orgs (tarball, git, or filesystem mount).
- **Interoperable** — an open standard instead of N vendor-locked catalog schemas.
- **Version-control-native and indexable** — lives next to code; works with any search/visualization tool.

## How it should be configured & used

- **Author** markdown concept files directly; or have agents **generate** them via metadata/enrichment pipelines (e.g. an agent walks a database and emits a bundle).
- **Configure** by declaring `okf_version` in the root `index.md`, choosing a `type` vocabulary for your domain, and adopting the recommended frontmatter fields you actually need.
- **Structure** the directory by domain (producer-defined); add `index.md` for navigation and `log.md` for a curated changelog.
- **Cross-link** concepts with absolute markdown links; add `# Citations` for traceable claims.
- **Automate** with CI: validate that every concept has frontmatter + `type`; auto-generate `index.md` by scanning `title`/`description`.
- **Consume** by parsing frontmatter for metadata and walking links as a knowledge graph — gracefully ignoring unknown types/keys.

## Antipatterns

- **Treating it as a database/query engine.** OKF is a *format*, not a platform — no runtime, query layer, or auth. Suited to internal knowledge and enrichment pipelines, not real-time catalog queries.
- **Over-structuring.** Cramming complex/nested metadata into flat YAML fights the spec; reference an external schema (`resource` + Avro/Protobuf/OpenAPI) instead.
- **Rejecting non-perfect bundles.** Consumers that error on unknown types, missing optional fields, or broken links violate the spec's permissiveness and block incremental growth.
- **Documentation rot.** The flip side of "broken links tolerated": without discipline (or CI), link integrity and freshness decay. A curated `log.md` and `timestamp` discipline are the mitigations.
- **Registering/standardizing `type` prematurely.** Values are intentionally producer-chosen; a rigid global taxonomy defeats "minimally opinionated."

## Why not just md?

Because OKF **is** "just md" — plus a thin, deliberate contract that plain markdown lacks:

1. **A required `type` field** so a consumer can categorize a concept without parsing prose.
2. **Path-as-identity + bundle structure** so concepts are addressable and distributable as a unit.
3. **Link-as-graph-edge semantics** so a folder of notes becomes a traversable knowledge graph.
4. **Conformance rules** so independent producers and consumers can rely on a contract (the interop guarantee).

"Just md" gives you none of those guarantees — every consumer reinvents the metadata and linking conventions. OKF's bet is that the *minimum* convention on top of markdown (one required field + structure + permissive conformance) buys interoperability at almost zero authoring cost. The cost over plain md is small and mostly mechanical (frontmatter discipline); the benefit is portability and agent-readability.

## Relevance to this repo

This repo is *already* an OKF-shaped artifact — a directory of markdown consumed by agents — and [ADR-0002](../docs/adr/0002-catalog-source-of-truth.md) deliberately chose **markdown-as-source-with-validators** over a generated single source, for the same reasons OKF exists (markdown is the product; humans and agents read it directly; no build step). Concretely:

- **Already aligned:** memory files use `type` frontmatter; the repo has `index`-like entry docs (`README.md`, `CLAUDE.md`, `MEMORY.md`), changelog-ish discovery logs, citations in evals, and a validator suite (`make check`) that mirrors OKF's "CI validates frontmatter" idea.
- **Gap to conformance:** the heavyweight knowledge (evals, methodologies) carries metadata as **bold-key prose + the CATALOG table**, not YAML frontmatter with a `type` field. So the repo is OKF-*adjacent*, not OKF-*conformant*.
- **The "why not just md?" answer for us:** ADR-0002 already settled that markdown is the right substrate. OKF would add a portability/interop contract on top — valuable if we ever want other tools/agents to consume our evals as a standard bundle; low value if the audience stays this repo's own tooling.

## Recommendation

**Do not adopt OKF wholesale now — but it's a low-cost, high-alignment option worth tracking.** Concrete, optional follow-ups (each its own issue, none done here):

1. **Catalog OKF** as a `reference` entry (Reference category, peer to `ag-ui`/`openui`/`a2ui` — open interchange specs). Cheapest win; pure documentation.
2. **Conformance experiment (spike → eval):** add YAML frontmatter with a `type` field to one doc class (e.g. `evaluations/`) and a CI check, measuring authoring cost vs. the interop benefit. Reversible and additive — exactly the kind of incremental step ADR-0002 leaves the door open to.
3. **Defer** any bundle-wide restructuring; OKF v0.1 is explicitly "a starting point, not a finished standard," so wait for the spec to mature before deep investment.
