## What it does

A TypeScript CLI (from LangChain, v0.0.1, ~10 days old at eval time) that runs a DeepAgents documentation agent against your repo to generate and continuously maintain an **agent-facing wiki** in an `openwiki/` directory. Mechanism: `openwiki --init` spins up a `createDeepAgent` graph (LangGraph + SQLite checkpointer) with a `LocalShellBackend` rooted at your repo (virtual filesystem paths + real host shell for git), feeds it a ~130-line system prompt (`src/agent/prompt.ts`) plus a git-evidence block (`git status`, HEAD, recent `git log --name-status`), and lets it write `openwiki/quickstart.md` plus a small set of section pages. It then appends a fixed "## OpenWiki" reference section to top-level `AGENTS.md`/`CLAUDE.md` so coding agents read the wiki before exploring source. `openwiki --update` does an incremental refresh: it reads `openwiki/.last-update.json` (timestamp + `gitHead` + model), diffs `git log <lastHead>..HEAD`, and is prompted to make *surgical* edits with an explicit diff budget ("if fewer than ~5 source files changed, update at most 1-2 wiki pages"; no-op allowed). A shipped GitHub Action example (`examples/openwiki-update.yml`) runs `openwiki --update --print` daily and opens a PR via `peter-evans/create-pull-request`. Providers: OpenRouter (default, model `z-ai/glm-5.2`), Baseten, Fireworks, OpenAI, Anthropic; credentials saved to `~/.openwiki/.env` (mode 0600).

## How we tested it

**Evidence:** RUN

Hands-on smoke test of the CLI plus a full source read; **the core doc-generation loop was not run hands-on — it needs an LLM API key** (none available in this environment: no `ANTHROPIC_API_KEY`, no `~/.openwiki/.env`), so no wiki output or token/latency metrics were measured here. What we *did* run and observe:

1. **Install resolves**: `npm view openwiki version dist-tags` → `0.0.1` (latest), MIT. Local install (`npm install openwiki --prefix .`) added 140 packages in 14s.
2. **`openwiki --help`** works and prints the banner (shows `provider: OpenRouter`, `model: z-ai/glm-5.2`, "LangSmith tracing disabled") plus usage for `--init`, `--update`, `-p/--print`, `--modelId`. Quirk: **`--version` is not a flag** ("Unknown option: --version"); the version only appears in the banner.
3. **No-key gate**: in a throwaway git repo (one committed `math.js`), `openwiki -p "Summarize this repo"` and `openwiki --init` (stdin closed) both fail fast with exit code 1 and a clear message: `OPENROUTER_API_KEY is required for non-interactive runs. Run openwiki in an interactive terminal to save credentials.` Nothing was written — no `openwiki/` dir, no `AGENTS.md`, and `~/.openwiki` was not created. Clean failure behavior for CI.
4. **Source read** (shallow clone, ~6.6k LOC of TS across 9 files): prompt strategy in `src/agent/prompt.ts`, incremental-update mechanics in `src/agent/utils.ts` (SHA-256 content snapshot of `openwiki/` before/after the run — excluding `.last-update.json` — so metadata is only rewritten when doc content actually changed), provider/model fallback in `src/agent/index.ts` (OpenRouter route retries server errors through `openai/gpt-5.4-mini` → `anthropic/claude-sonnet-5`), credential handling in `src/env.ts` (0700 dir / 0600 file, key-hygiene diagnostics for whitespace/newlines/quotes).
5. **Dogfood evidence** (vendor's own output, not ours): the repo's own `openwiki/` was generated by the tool itself (`.last-update.json`: 2026-06-30, model `accounts/fireworks/models/glm-5p2`) — a 5-page wiki (quickstart + architecture/cli/agent/operations) whose quickstart accurately names every source file and includes a genuinely useful "Notes for future agents" section. Its `AGENTS.md` is exactly the prompt's fixed reference block.

```
npm view openwiki version dist-tags license          # 0.0.1, MIT — install command resolves
mkdir openwiki-install && npm install openwiki --prefix openwiki-install
./openwiki-install/node_modules/.bin/openwiki --help # banner + usage (real output captured)
cd toyrepo && git init && git commit ...             # throwaway repo
openwiki -p "Summarize this repo" </dev/null         # exit 1: OPENROUTER_API_KEY is required for non-interactive runs.
openwiki --init </dev/null                           # exit 1, same gate; no files written, ~/.openwiki untouched
git clone --depth 1 https://github.com/langchain-ai/openwiki  # source read
```

## What worked

- **CLI exercised end-to-end up to the key gate**: install, help, print-mode, and `--init` all behave sanely; the no-key failure is fast, exit-code-correct (1), and side-effect-free — good properties for the daily-CI use case.
- **The system prompt is unusually disciplined** for a doc generator: hard rules against inventing files/APIs, a subagent budget (1-2 for large repos), a temporary `_plan.md` that must be deleted before finishing, an update diff budget, explicit "updates may be a no-op", and a ban on formatting-only edits — directly targeting the doc-churn failure mode that makes scheduled doc PRs noisy.
- **Incremental update is git-anchored, not vibes-anchored**: `.last-update.json` records `gitHead`; the update prompt receives `git log <lastHead>..HEAD --name-status` and must build a "source change → docs affected → edit needed → why" impact plan before touching pages.
- **Agent-facing wiring is the real differentiator**: the fixed `## OpenWiki` section appended to `AGENTS.md`/`CLAUDE.md` (top-level only, idempotent, semantic-staleness check on updates) is what none of the overlapping doc generators do.
- **Credential hygiene is above average for a v0.0.1**: 0700/0600 perms on `~/.openwiki`, key previews truncated, diagnostics for pasted-key artifacts (whitespace, newlines, bracketed suffixes), and the prompt forbids the agent from reading `.env` files.

## What didn't work or surprised us

- **Could not verify output quality first-hand** — the generation loop needs an API key. The only wiki we inspected is the vendor's own dogfooded one (which is good, but it's their showcase).
- **Prompt/Action mismatch**: the system prompt requires updating root `AGENTS.md`/`CLAUDE.md` on every run, but the shipped Action (`examples/openwiki-update.yml`) uses `add-paths: openwiki` in `create-pull-request` — so any root-file edits the agent makes in CI are **silently dropped from the PR** (and left dirty in the runner). Local runs and CI runs produce different results.
- **The agent gets a real host shell — with one real mitigation**: deepagents' `LocalShellBackend.execute` is a bare `spawn(command, {shell: true})` with the user's full permissions ("no restrictions" per deepagents' own docstring); `virtualMode: true` only remaps *filesystem-tool* paths, and the cwd-rooting of shell commands is prompt-enforced only. The mitigation: openwiki passes neither `env` nor `inheritEnv`, and deepagents defaults `inheritEnv: false`, so the spawned shell gets an **empty environment** — the LLM's commands cannot read `OPENROUTER_API_KEY`/`LANGSMITH_API_KEY`. The residual CI risk is the checkout's `persist-credentials: true` token in `.git/config`, which shell commands can use (e.g. `git push`) regardless of env, in a workflow with `contents: write`.
- **Very young**: created 2026-06-22, v0.0.1, no tests in the repo (`checks.yml` is lint/format only). Default model is `z-ai/glm-5.2` via OpenRouter — the defaults reflect LangChain's provider partnerships more than a quality benchmark.
- **`--version` doesn't exist** as a flag; minor, but the kind of rough edge you'd expect at 0.0.1.
- **Their own daily-update PRs show mixed churn discipline** (observed via `gh pr list`, 2026-07-02): the repo dogfoods the scheduled Action (`.github/workflows/openwiki-update.yml`, not just the example), and of its "docs: update OpenWiki" PRs one merged empty (+0/−0), one is open empty, and the one substantive update (+70/−23) was closed unmerged — even the disciplined update prompt produced a refresh the maintainers rejected.

## Quality signals affected

| Signal | Impact | Evidence |
|--------|--------|----------|
| Correctness | + (unverified) | Prompt hard-grounds every claim in inspected sources/git and allows no-op updates; but we could not run generation to check hallucination rate ourselves. |
| Speed | + | A maintained `openwiki/quickstart.md` referenced from AGENTS.md/CLAUDE.md cuts per-session codebase re-exploration for agents — the stated goal ("make high-quality code changes with less source exploration"). |
| Maintainability | + | The whole design targets doc rot: gitHead-diffed incremental updates, surgical-edit rules, daily PR automation, content-hash change detection. |
| Safety | - / neutral | LLM gets host shell execution; daily CI run executes model-chosen commands under a `contents: write` token; mitigated by prompt rules and PR-gated (not direct-push) output. |
| Cost Efficiency | neutral | Update runs are scoped by git diff and a page budget, but a daily LLM agent run over a large repo is a recurring spend; no token metrics measured. |

## Verdict

**DEFER**

Promising and genuinely differentiated — agent-facing docs wired into AGENTS.md/CLAUDE.md with git-anchored daily maintenance is a niche none of our catalogued doc generators (deepwiki-rs, PocketFlow-Tutorial-Codebase-Knowledge, docmd, oo-component-documentation) covers — and the CLI surface we could exercise behaved well. But the core generation loop is unverified here (blocked on an LLM API key), the project is 10 days old at v0.0.1 with no tests, and the shipped Action silently drops the agent's root-file edits. Re-evaluate after running `openwiki --init` + one `--update` cycle with a real key on a mid-size repo (grade the wiki for hallucinations and the update for churn), or when the package passes ~v0.1 with tests.
