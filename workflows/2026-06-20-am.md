**Context Engineering**

**What it is:** A discipline for systematically designing the complete information environment an agent operates in — not just the prompt, but the memory architecture, retrieval pipeline, context compression strategy, and session state. In mid-2026 this is widely called "the defining AI productivity skill of 2026," a direct response to teams hitting ceilings on output quality despite model upgrades. The shift is architectural: memory is now a dedicated layer that extracts relevant facts at session end, stores them in a vector index keyed by entity/session/project, and injects semantically retrieved context at session start. Teams with tight context management consistently outperform teams with larger context windows.

**Pros:** Lifelong agent memory turns a frequently-visited codebase into a compounding asset — second and third sessions arrive with the right 10k tokens rather than a raw 100k dump. Retrieval-augmented context means the agent doesn't need to re-explore the repo on every session, which cuts both latency and cost. Session-start injection separates "remembering the project" from "filling the context window," making it possible to maintain state across /clear and model swaps. The pattern also enables overnight and async runs to continue where they left off rather than starting cold.

**Cons:** Retrieval quality depends entirely on chunking and indexing strategy — a naive strategy retrieves the wrong context and actively misleads the agent. Code isn't prose: most memory frameworks (dense embeddings, BM25) were designed for document retrieval and perform poorly on code structure and call graphs. Adding a memory layer introduces latency and architectural complexity that most teams underestimate. The dominant 2026 failure mode is bolting on a generic doc-retrieval system and calling it "agent memory" — the supermemory.ai memory bottleneck post documents this as the gap between what teams ship and what actually works.

**Suggestions:** OMEGA, mem0, claude-mem, SimpleMem, agentmemory, and engram (CATALOG.md → Memory & Context) each cover different points on the persistence/retrieval trade-off; OMEGA and mem0 offer the richest semantic search. For the compression side — keeping what's retrieved from bloating the context — headroom and context-mode (CATALOG.md → Memory & Context) report 60–98% token reductions on verbose tool output. claude-subconscious (CATALOG.md → Memory & Context) is the lightest-weight option for cross-session state without a separate memory server. This repo's WORKFLOW.md already calls out context management as an Implement-stage concern; the memory tools above operationalize it.

**Verification-Aware Planning (Goal-Check Loops)**

**What it is:** A planning discipline where each subtask in the agent's plan is accompanied by an explicit pass/fail criterion the agent evaluates before moving to the next step. The agent proceeds only when the current checkpoint passes, or halts and escalates when it fails. The 2026 agentic loop research names this as the structural difference between a basic "plan and execute" pattern and a production-reliable loop. Research from DataScienceDojo's agentic loops guide identifies four root causes of agentic loop failures — no hard stopping conditions, underspecified goals, context overflow, and missing cost controls — verification-aware planning directly addresses the first two. The pattern is sometimes called "goal-check loops" to distinguish it from the simpler ReAct loop where the agent only checks its own reasoning, not an external criterion.

**Pros:** Errors are caught at the subtask level rather than compounding through 50 subsequent steps. Each checkpoint passage becomes an audit trail entry: you can see exactly where an agent succeeded, stalled, or made a wrong turn. The planning discipline this enforces — every subtask needs a checkable criterion — independently improves task decomposition quality. Composes naturally with the SDD (Spec-Driven Development) pattern from the previous workflow log: acceptance criteria from the spec translate directly into verification checkpoints in the plan.

**Cons:** Writing good verification criteria is where the pattern breaks down in practice — vague criteria ("confirm tests pass") catch only obvious failures. Agents can game shallow criteria — deleting failing tests, modifying assertions, writing stubs that satisfy checks without doing the work. The agent that writes the plan and the agent that verifies it share the same blind spots, so systematic errors survive the loop. Adds non-trivial planning overhead that doesn't pay off for short or one-shot tasks.

**Suggestions:** claude-code-harness (CATALOG.md → Agent Harnesses) ships an autonomous Plan-Work-Review cycle with built-in gate logic. KARIMO (CATALOG.md → Agent Harnesses) provides phased execution — PRD → planning → implementation → review — that maps cleanly onto a verification-aware plan. For generating the criteria themselves, pair with the SDD pattern and use context7 (WORKFLOW.md → Implement) to keep API references in acceptance criteria current. The /workflows fan-out command can dispatch independent verification subagents, each checking a different checkpoint criterion without sharing context with the generator.

**Hybrid Plan-Execute-CI Workflow**

**What it is:** The emerging consensus "best default" agentic coding workflow: the developer reviews and approves a design plan in the IDE, the agent executes against it in a local sandbox, and CI green + human PR review are required gates before anything merges. Kilo.ai explicitly names this as the best default for 2026; Addy Osmani's workflow post describes a nearly identical loop. The pattern is a direct response to the "vibe coding" backlash — it captures AI implementation speed while preserving the infrastructure gates that protected codebases before AI. The critical insight: AI-generated PRs without review governance wait 4.6× longer in review queues even as time-to-PR drops by up to 58%, so the bottleneck has shifted from implementation to review.

**Pros:** Keeps the developer at the two high-leverage decision points — design approval and merge — while delegating the implementation and routine test loop to the agent. CI running on the agent's branch before human review means reviewers see a working branch, not a broken one. The gate is in the infrastructure (CI config, branch protection), not in the agent's system prompt, so it applies consistently regardless of which model or harness is in use. Pairs naturally with any of the other patterns in this file — context engineering makes the implementation phase faster; verification-aware planning makes CI more likely to pass first try.

**Cons:** Review becomes the new bottleneck — AI can produce PRs 58% faster than humans can review them. CI green doesn't imply correct behavior; agents can produce passing test suites that don't actually validate the intended behavior. Sandboxed local execution adds setup complexity that teams skip on quick tasks, creating inconsistency. Without explicit review governance (SLAs, assignment rules, queue limits), the AI speed gain evaporates in the review queue.

**Suggestions:** This is WORKFLOW.md's Verify → Review → Ship arc made explicit as a team norm. code-review-graph (CATALOG.md → Code Review & Quality) provides blast-radius analysis so reviewers know exactly what a change touches — critical for reviewing AI-generated PRs that often span more files than expected. design-council (CATALOG.md → Skills & Plugins) runs 11 role-specialized peer agents to pressure-test the design plan before the execution phase starts. superpowers (CATALOG.md → Agent Harnesses) ships a code review skill that can serve as a CI-adjacent automated first-pass, reducing the human review burden for the most common error classes.

**Human-in-the-Loop Trust Checkpoints**

**What it is:** Explicit approval gates embedded inside long-running agentic workflows, placed at trust boundaries rather than at fixed time intervals. A trust boundary is any point where the agent transitions from read to write, from local to remote, from reversible to irreversible, or from low-stakes to high-stakes. Common checkpoint placements: before any push to a shared branch, before deleting or migrating data, before invoking external APIs with side effects, and before any production-affecting action. The pattern emerged from the 2026 human-in-the-loop movement as agentic systems matured from demos into production workflows. Research confirms that trust boundaries defined in workflow config (AGENTS.md, tool permissions) rather than embedded in individual prompts apply consistently across tasks and survive harness updates.

**Pros:** Reduces the blast radius of errors dramatically — the agent can reason incorrectly through 50 read-only steps, and the human catches it before any damage is written. Checkpoint logs create a natural audit trail of human decisions, useful for post-mortems and compliance reviews. Developers report significantly higher trust in overnight and async runs when they know the agent will halt at write operations rather than proceeding autonomously. Boundary-based placement avoids the "every step requires approval" trap — the human only engages when the risk profile changes.

**Cons:** Checkpoints at the wrong granularity negate the automation benefit — too many pauses, and the developer is manually driving the agent. Human-in-the-loop requires available, attentive humans; the pattern fails for overnight or async runs where no one is watching. Defining trust boundaries upfront requires mapping the agent's complete action graph, which most teams skip. Agent behavior at a blocked checkpoint is harness-dependent — some loop forever, some exit silently, some spam notifications.

**Suggestions:** WORKFLOW.md's permission modes section covers the native Claude Code approval model. ECC and ruflo (CATALOG.md → Agent Harnesses) both ship configurable guardrail configurations with explicit stop points. For async overnight runs where a human isn't watching, substitute spec-driven success criteria (SDD pattern) for real-time checkpoints — the spec defines what "safe to proceed" means without requiring a human in the loop. The AskUserQuestion tool in Claude Code's remote SDK routes approval requests to a notification channel (phone push + email) rather than blocking a terminal, making async approval practical.

**Code Health as an AI Precondition**

**What it is:** Before deploying AI coding agents at scale, audit and raise codebase health to a minimum threshold before expecting reliable AI output. The insight, from CodeScene's agentic coding patterns research: AI performs best in well-structured, low-coupling code and degrades in tangled or undocumented codebases — not because the AI is confused, but because it follows local patterns and actively proliferates the existing problems. The practical consequence: teams deploying AI on high-debt codebases amplify the debt rather than escaping it. CodeScene's benchmarks recommend a Code Health score of at least 9.5/10 as the precondition for predictable AI-assisted development. Classic software engineering disciplines — design before coding, writing tests, maintaining standards — become more important, not less, when AI writes the code.

**Pros:** Clean-code investment pays double dividends: better human readability and materially better AI output quality on the same codebase. Low-coupling code is easier for agents to partition into independent tasks, directly enabling the parallel worktree pattern from the previous workflow log. Reduces the surface area for AI to proliferate debt — an agent working in a 9.5-health codebase has fewer bad local patterns to imitate. The investment is durable: the clean codebase continues to produce better AI output as models improve, compounding the return.

**Cons:** Raising code health before AI adoption delays adoption in competitive environments. The 9.5 threshold is CodeScene's proprietary metric and depends on their specific analysis tooling — not independently benchmarked. Legacy codebases may require rewrite rather than incremental health improvement, which is a different order of investment. Teams under pressure to ship with AI can't always afford the cleanup cycle first, creating a two-speed dynamic where teams with clean codebases pull ahead faster.

**Suggestions:** codescene (CATALOG.md → Code Review & Quality) provides Code Health scoring and hotspot analysis to identify which files are most likely to degrade AI output. code-review-graph and graphify (CATALOG.md → Code Understanding) give blast-radius visibility for targeted refactoring runs before agent deployment. Crucially, the cleanup itself can be an agentic task: an overnight run with a targeted refactoring spec (SDD pattern) and verification checkpoints can accelerate the health improvement phase, turning a chicken-and-egg problem into a bootstrapped loop. WORKFLOW.md's Reflect stage already calls for periodic code health audits — position that audit as the trigger for each new AI adoption wave.
