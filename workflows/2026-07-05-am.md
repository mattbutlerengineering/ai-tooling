**Instruction-File Standardization: AGENTS.md as Cross-Tool Convention Infrastructure**

**What it is:** Through mid-2026 the ecosystem of AI coding tools has converged on a shared instruction-file convention — a markdown file placed in the repository root that encodes project rules, architecture constraints, coding standards, and task-execution preferences for agent consumption. AGENTS.md, originally a community-driven open format, is now stewarded by the Agentic AI Foundation under the Linux Foundation and natively read by Claude Code, OpenAI Codex CLI, Cursor, Aider, Devin, GitHub Copilot, Gemini CLI, Windsurf, and Amazon Q as of early 2026. The convergence of eight-plus major tools onto a single file format has transformed what was previously a per-tool configuration concern — each harness had its own convention (CLAUDE.md for Claude Code, cursor rules for Cursor, Copilot instructions for VS Code) — into a cross-tool infrastructure standard. The operating pattern teams are adopting: every repository carries a well-maintained instruction file that specifies what agents should do, what they must not touch, which test commands constitute acceptance, and which architectural decisions are settled and should not be relitigated. When an agent makes a mistake, the instruction file is updated to prevent recurrence — turning the file into persistent institutional memory that accumulates across sessions and across the team, rather than requiring each developer to re-discover the same constraints in each new session.

The empirical finding from the 2026 evaluation thread (cited in HN discussion of AGENTS.md effectiveness research) is counterintuitive in an environment where AI automation is otherwise assumed to dominate: human-curated instruction files consistently outperform LLM-generated files by roughly 4 percentage points on task success rate across agents tested. The instruction file is the artifact where human engineering judgment has the highest per-hour leverage on agent output quality, because it encodes tacit knowledge, prior decisions, and constraint boundaries that agents cannot infer from code inspection alone. A model reading well-maintained project rules starts each session with the accumulated context of all prior sessions; a model without one reconstructs that context from scratch (and often gets it wrong) every time.

**Pros:** The cross-tool standardization creates a compounding effect: a well-maintained AGENTS.md improves output quality across every tool the team uses, not just the primary harness. A team that has invested in a thorough instruction file gets the same architectural constraint enforcement and coding standard compliance from Claude Code sessions, Copilot inline completions, and PR-level review agents without per-tool reconfiguration. The instruction file also functions as a team alignment artifact: writing explicit rules about what agents should and should not do surfaces disagreements about coding standards, architecture constraints, and delegation scope that would otherwise emerge as inconsistent agent behavior across developers. The maintenance loop — update the file when an agent makes a correctable mistake — means each session starts with the lessons of all prior sessions rather than requiring repeated correction of the same failure modes.

**Cons:** Cross-tool standardization introduces a shared-file drift risk: an instruction file written for one tool's context model may not translate correctly to another tool's interpretation of the same text. AGENTS.md is a flat markdown file with no schema enforcement, so the same instruction may be parsed differently by Claude Code (which reads it as a session context supplement) and Copilot (which extracts subsets for different inline-completion contexts). A team maintaining one file for eight tools is betting that behavioral overlap across tools is sufficient for the file to be useful to all without per-tool tuning — sometimes correct, sometimes producing subtle per-tool divergence that is hard to diagnose. The maintenance burden is also real: a comprehensive instruction file written at project inception becomes stale as the architecture evolves, and stale rules are worse than absent rules because they give agents false constraints to optimize against. The 4-percentage-point advantage of human-curated files disappears if the human-curated file is six months out of date.

**Suggestions:** CATALOG.md's Skills & Plugins category includes the instruction-file authoring and maintenance tools that operationalize this pattern at team scale; the Agent Harnesses category covers the harnesses that consume instruction files and how their parsing behaviors differ. This repo's own CLAUDE.md is a live example: it encodes the catalog format, category taxonomy, integrity gate commands, skill references, and harness conventions that Claude Code and opencode agents need to operate correctly in this codebase, and it is updated when agent behavior deviates from intent. The JetBrains and Augment Code guides (linked in Sources) are the most complete published how-tos for structuring instruction files for maximum cross-tool effectiveness. For teams starting: the minimum viable instruction file has four sections — (1) project description and settled architecture decisions; (2) coding standards not captured by linter rules; (3) which paths or files require human review before agent modification; (4) the exact test commands that constitute acceptance for any change — and should be treated as a living document updated after any agent session that produces a correctable mistake.

**Issue-Quality Task Framing: Structuring Prompts as Engineering Tickets**

**What it is:** The 2026 practitioner consensus, articulated in the Stack Overflow shared coding guidelines post, the Addy Osmani and GoGloby workflow analyses, and Anthropic's Agentic Coding Trends report, is that task framing quality is the highest-leverage variable in agentic coding output — higher than model selection, prompt style, or tool choice. The differentiating pattern is simple in description but consistently underinvested in practice: tasks are written like GitHub issue tickets, not like chat messages. A chat-style prompt ("refactor the auth module to be more maintainable") gives an agent a direction without a destination; an issue-quality prompt gives the agent background (what the auth module currently does and why it needs changing), a goal (what done looks like in behavioral terms), acceptance criteria (specific, verifiable conditions the output must satisfy), constraints (what must not change, which APIs must remain stable, what the test coverage floor is), and test commands (the exact commands the human will run to verify completion). The structural difference is the difference between asking an agent to "do something useful in this direction" versus "reach this specific destination by this route and prove arrival with this check" — and the output quality difference is proportionally large. Strong agentic workflows across the 2026 analyses share three traits: tasks are written with explicit background, goal, acceptance criteria, constraints, and test commands; permissions are managed like production systems with least privilege and audit logs; and results are reviewed like PRs from junior engineers rather than trusted output.

**Pros:** Issue-quality task framing is the intervention with the most consistent quality improvement across model types, task types, and tool configurations because it directly addresses the most common agentic failure mode: an agent completing the wrong task confidently. A well-framed task with explicit acceptance criteria converts the exit condition from a model-internal heuristic judgment ("this seems done") to a mechanical check ("these specific tests pass, these specific files are unchanged, this specific API contract is honored") — producing output that is either correct or verifiably wrong, eliminating the failure mode of plausible-but-subtly-wrong output that survives code review and manifests in production. The shared conventions pattern extends this further: when task framing standards are documented and applied consistently across a team, every developer produces tasks the same way, which means agents receive consistently structured inputs across all developers and can be evaluated and improved systematically rather than per-developer.

**Cons:** Issue-quality task framing requires more upfront investment than casual prompting, and the cost is paid before any implementation output is visible — which creates adoption friction for teams under delivery pressure. For exploratory tasks where the goal itself is uncertain, the full issue-ticket structure adds process overhead without proportional benefit: acceptance criteria cannot be specified before the exploratory task surfaces what acceptance looks like. The discipline also requires thinking precisely about what is wanted before asking for it, which is harder than it sounds — many developers discover during task framing that they do not have a clear specification for the task they thought they understood, and the friction of writing acceptance criteria is actually the friction of thinking the problem through. That is valuable friction, but not zero-cost. Teams that apply full issue-ticket structure to all agent interactions also find the overhead disproportionate for small, well-understood tasks (a single-function edit with an obvious acceptance criterion does not need a full ticket structure), and calibrating when to use full versus lightweight framing requires experimentation.

**Suggestions:** WORKFLOW.md's Plan stage is structurally aligned with issue-quality task framing: the Plan stage's output is a task description, and the issue-ticket structure (background, goal, acceptance criteria, constraints, test commands) is the right template for that output regardless of whether the task is executed by a human or an agent. The `skills/setup-workflow/` bootstrap skill in this repo generates a CLAUDE.md and task-framing template that captures the minimum-viable structure as part of new-project setup. The Stack Overflow shared coding guidelines post (linked in Sources) is the most accessible published treatment of what the shared conventions should contain and how to build team adoption around them. For teams starting: the minimum viable task framing upgrade is a single sentence appended to every agent prompt before delegation — "This is done when: [specific, verifiable condition]" — which converts the exit condition from heuristic to mechanical without requiring full ticket structure. That single addition, consistently applied, captures most of the benefit of full issue-quality framing for simple tasks and builds the habit of precision that full adoption requires.

**Code Health as a Delegation Gate: Using Maintainability Metrics Before Handing Off to Agents**

**What it is:** The CodeScene agentic coding best practices analysis introduces a structural reframing of code health metrics that is gaining adoption in mid-2026: rather than treating health scores solely as signals of technical debt accumulation (the traditional use), teams are deploying them as delegation gates — signals that determine which parts of a codebase are safe to hand off to AI agents autonomously and which parts require human-supervised implementation. The logic is structural: a function with high coupling (many callers, many callees), low cohesion (multiple distinct responsibilities in one unit), and high cyclomatic complexity presents a high-risk surface for agentic modification. An agent changing a highly-coupled function is more likely to inadvertently break callers that are not visible in the immediate context window, produce behavioral divergence in edge cases, or make a change that is locally correct but globally inconsistent with the function's contract. A function with low coupling, high cohesion, and low complexity presents a bounded risk surface: callers are few, the behavioral contract is straightforward to verify, and the blast radius of an error is contained. Code health metrics operationalize this distinction: a function with a strong health score is structurally safe to delegate autonomously; a function with a weak score requires human-supervised implementation or a refactoring pass before autonomous delegation. The infrastructure pattern 2026 teams are deploying is a three-layer stack: code health tools providing objective risk signals, MCP servers exposing those signals as actionable tools within the agent's context, and AGENTS.md encoding the delegation policy — which score thresholds trigger autonomous versus supervised execution, which paths are always human-reviewed, and which test coverage floors must hold for automated acceptance.

**Pros:** Using code health metrics as delegation gates converts a reactive practice (reviewing agent output for quality issues after generation) into a proactive one (preventing high-risk delegation before output is produced). The cost differential strongly favors the proactive approach: a health-score check that prevents an agent from modifying a highly-coupled function costs one tool call; reviewing and correcting the output of an agent that modified a highly-coupled function incorrectly costs review time, rework cycles, and potentially a production incident. The MCP integration is architecturally significant: when code health signals are available as tools in the agent's context, the agent can self-triage its task scope — checking a function's health score before deciding whether to proceed autonomously or escalate to a human checkpoint — which eliminates the human overhead of manually triaging which code paths are agent-safe before each task.

**Cons:** Code health metrics as delegation gates introduce a rigidity risk that runs counter to their purpose: low-health code is precisely the code that most needs improvement, and treating health scores as hard gates on delegation means the code that would benefit most from agentic refactoring is the code agents are least permitted to touch. A pure gate implementation produces a self-reinforcing dynamic where high-health code is continuously improved by agents while low-health code accumulates further debt because agents are excluded from it. The mitigation — permitting agents to refactor low-health code under human supervision before implementing features in it — adds oversight overhead that may exceed the benefit for teams without a dedicated refactoring budget. Health metrics also measure structural complexity, not behavioral correctness: a function can have low coupling and high cohesion but contain a logical bug, and a strong health score does not guarantee that an agent's modification will be correct — only that the blast radius of an error will be bounded.

**Suggestions:** CATALOG.md's Code Review & Quality category inventories the code health tools that supply the delegation-gate signal: CodeScene, SonarQube, and equivalent static analysis tools are catalogued there with their evaluation status. WORKFLOW.md's Plan stage is the natural integration point for the delegation-gate check: before delegating an implementation task, a Plan-stage code health query surfaces the risk profile of the target code and determines whether the task warrants full human supervision, supervised agent execution, or autonomous execution. For teams starting: the minimum viable delegation gate is a single AGENTS.md policy rule — "any file with a complexity score above [threshold] requires human review before you modify it; check the code health tool before proceeding" — which can be implemented without per-task triage overhead and calibrated over one sprint by comparing agent task success rates across code health tiers for the specific codebase.

**Loop Engineering: Designing Agentic Loops as Software Artifacts**

**What it is:** The Data Science Dojo 2026 guide and the Webfuse agentic coding analysis converge on a framing that is becoming standard practitioner vocabulary: an agentic loop is not an emergent property of a capable model but a software artifact that must be designed, specified, and maintained with the same rigor as any other system component. The distinction matters operationally. Teams that treat agentic loops as emergent behavior configure a capable model and observe what it does; teams that treat agentic loops as designed artifacts specify four elements before the loop runs. First, the verifiable goal: not "improve test coverage" (a direction) but "all files in src/ have ≥80% branch coverage as measured by coverage.py" (a condition that evaluates mechanically to true or false). Second, the trigger type: event-driven (the loop starts when a CI failure occurs), polling (the loop checks a condition on a schedule), or human-initiated (the loop starts on explicit delegation). Third, the loop architecture: ReAct (reason-act-observe cycles with no explicit pre-planning, suited to exploratory tasks with unknown action sequences), plan-execute (the loop generates a complete action plan before taking any action, suited to well-specified tasks where human plan review is valuable), or critic-refine (an executor produces output and a critic agent evaluates it against the verifiable goal, feeding corrections back for refinement, suited to quality-sensitive tasks where the output surface is defined but the path to it is not). Fourth, the guardrails: maximum iteration count, cost ceiling, and escalation condition — the specific state that triggers a human-oversight checkpoint rather than another loop iteration. Loop engineering is the discipline of making these four design decisions before delegation rather than discovering the loop's behavior from its outputs after the fact.

The failure mode loop engineering is designed to prevent is the runaway loop: an agent looping indefinitely because its exit condition is never reached, accumulating cost without producing output. Runaway loops are the most expensive agentic failure mode — not because they produce wrong results but because they produce no results while consuming substantial compute budget. A loop with a verifiable goal, a maximum iteration count, and a cost ceiling cannot run away: when the goal is not reached within the specified bounds, the loop terminates and escalates gracefully rather than continuing to iterate. The verifiable goal is the most important design element because it converts the loop's exit condition from a model-internal heuristic ("I think I'm done") to an external mechanical check ("this specific condition is now true") — the same transition that issue-quality task framing makes at the prompt level, applied to loop architecture.

**Pros:** Designed loops shift the primary agentic failure mode from silent, expensive runaway to explicit, bounded escalation. A loop that reaches its iteration or cost ceiling and escalates surfaces itself: the human receives the partial state and can decide whether to adjust the goal, extend the bounds, or handle the task manually. A runaway loop surfaces itself only when the cost bill arrives, by which point the intervention window has closed. The designed-loop discipline also improves reproducibility: a loop whose trigger, architecture, and guardrails are specified in AGENTS.md or a workflow config produces the same behavior across sessions, agents, and tool versions rather than depending on a specific model version's emergent cycle behavior. The plan-execute architecture variant has a further benefit: because the loop generates a complete plan before any filesystem changes occur, the human can inspect and modify the plan at a low-cost intervention point — the same oversight window that spec-driven development provides at the task level, applied to loop execution.

**Cons:** Loop engineering introduces upfront design work that casual agentic loops avoid: specifying a verifiable goal requires the same precision investment as writing acceptance criteria, and specifying guardrails requires knowing in advance how many iterations a goal should require and what a reasonable cost ceiling looks like — estimates that are hard to calibrate for novel task types without prior run data. The four-element specification is also not always decomposable for complex tasks: a goal that requires exploring the codebase before it can be precisely formulated requires the loop to run some iterations before the verifiable goal can be stated, which undermines the pre-specification discipline. ReAct architectures are particularly resistant to over-specification: the actions at each step depend on the observations of the prior step by design, and imposing a rigid pre-specified structure on a ReAct loop produces a plan-execute architecture that is a poorer fit for the exploratory task class ReAct is suited to. Teams that apply loop engineering uniformly across architectures — specifying tight iteration budgets for ReAct loops that require open-ended exploration — produce loops that hit their guardrails prematurely and escalate before the goal is reachable.

**Suggestions:** CATALOG.md's Agent Harnesses category inventories the frameworks that expose loop architecture as a configurable artifact: the Claude Code SDK, opencode, and workflow orchestration tools catalogued there implement the loop primitives (trigger registration, iteration limits, cost guardrails, escalation hooks) that loop engineering specifies. WORKFLOW.md's inner loop (Implement → Verify → Review) maps to the loop architecture choices: Implement maps to the executor, Verify maps to the critic or verifiable-goal check, and Review maps to the escalation condition. The Data Science Dojo 2026 guide (linked in Sources) is the most thorough published treatment of loop architecture variants and the decision criteria for choosing between ReAct, plan-execute, and critic-refine. For teams starting: the minimum viable loop engineering investment is two added fields on every agentic task before delegation — "Exit condition: [specific, mechanically verifiable statement]" and "Iteration limit: N" — which prevents runaway loops without requiring full loop architecture specification upfront. Those two fields, consistently applied, are sufficient to convert the most common agentic failure mode from silent runaway to explicit, diagnosable escalation.
