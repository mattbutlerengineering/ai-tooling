**The DORA Quality Debt Signal: Separating AI Volume from AI Quality**

**What it is:** The Google 2025 DORA Report has become the most-cited data point in mid-2026 discussions of enterprise AI coding adoption, and for reasons that cut against the naive productivity narrative: teams with 90% AI adoption showed a 9% increase in bug rates, a 91% increase in code review time, and a 154% increase in PR size compared to pre-AI baselines. The 2026 practitioner community has spent considerable effort interpreting what this data means and what it implies for workflow design. The emerging consensus is not that AI coding tools hurt quality — an interpretation that the DORA data does not directly support — but that AI coding volume without corresponding structural investment in quality workflows produces a characteristic debt profile: more code produced per unit of developer time, but more bugs per unit of code, and more review burden per PR because reviewers are evaluating AI-generated output that arrived in larger batches and was not subject to the incremental quality discipline that a developer exercising judgment during production applies. The 154% PR size increase is the key signal: AI agents produce large outputs without the natural checkpoints that human developers use to scope and stage their work, and large PRs systematically receive worse reviews than small ones because reviewer attention is finite and dilutes as diff size grows. The DORA finding is not a verdict on AI coding tools; it is a diagnostic of what happens when teams add AI output velocity without redesigning their quality loops to process that velocity without accumulating debt.

The practical implication that the 2026 practitioner corpus has drawn from the DORA data is a structural workflow change: the unit of AI-assisted work that enters review should be as small as a human-written PR, regardless of how large the agent's execution scope was. Teams that produce AI-assisted output in large batches — a full feature implemented in one agent session, reviewed as one PR — are in the DORA failure mode. Teams that decompose AI-assisted work into small, independently reviewable increments — each agent session scoped to one testable, reviewable change — avoid the review-dilution problem regardless of how much total output the agent fleet produces. The 91% increase in code review time is a direct consequence of the 154% increase in PR size; fix the PR size and the review time recovers. The 9% bug rate increase is partly a consequence of degraded review quality on large PRs, and partly a consequence of AI output that was not anchored to verifiable quality criteria — a problem the spec-driven development workflow (covered below) directly addresses.

**Pros:** The DORA data converts the quality-debt concern from an intuition to a measurable signal, which makes it actionable: teams can track PR size and review time as leading indicators of whether their AI coding workflow is accumulating quality debt before the lagging bug-rate indicator moves. The data also provides a concrete organizational argument for investing in the structural workflow changes — small PRs, spec-driven development, verification loops, supervised delivery — that mid-2026 practitioners are advocating: teams that can point to the DORA numbers when proposing workflow investment have a stronger case than teams making qualitative arguments about AI output reliability. Monitoring PR size is also a low-overhead quality gate that teams can add immediately without changing their tooling: a CI check that warns when AI-assisted PRs exceed a size threshold forces the decomposition habit before review quality degrades.

**Cons:** The DORA data is a correlation finding at population scale, not a controlled experiment on individual teams, which means extrapolating it to specific team circumstances requires judgment. Teams with strong existing code review infrastructure — small PRs already, high test coverage, robust CI — may not experience the DORA failure modes even at high AI adoption levels, while teams with pre-existing review debt may find AI adoption amplifies rather than creates that debt. The prescription of "small PRs always" also conflicts with some legitimate AI use cases: a refactoring that touches 200 files atomically is genuinely one change and splitting it into 200 PRs adds no quality benefit and substantial coordination overhead. The rule is more precisely "small reviewable units of intent," which sometimes corresponds to a large diff (when all 200 file changes implement a single coherent decision) and sometimes to a small diff (when two files implement an architectural decision with many downstream implications).

**Suggestions:** WORKFLOW.md's Ship stage is where the PR-size discipline belongs — the decision about how to stage AI-assisted work into reviewable increments is a Ship-stage workflow decision, not an agent-configuration decision. CATALOG.md's Code Review & Quality category covers tools that enforce PR-size limits, split large diffs, and surface review burden before a PR enters the queue; the DORA data makes the case for treating these tools as a structural quality gate rather than a convenience. For teams starting: the minimum viable DORA-response is a team agreement on maximum AI-assisted PR size (a common default in the practitioner corpus is 400 lines of diff) and an agent instruction (in CLAUDE.md or an agent system prompt) to propose a decomposition plan before implementing anything that exceeds that limit — which forces the spec-and-decompose discipline before the agent executes and the large-diff problem materializes.

**Spec-Driven Development: Structured Intent as the Anchor for Self-Correction**

**What it is:** Spec-driven development (SDD) has emerged in mid-2026 as the dominant vocabulary for the workflow upgrade from ad hoc prompting — sometimes called "vibe coding" in the practitioner literature — to structured AI-assisted delivery. In an SDD workflow, a developer writes a detailed feature specification before any agent execution begins: the specification describes the goal, the constraints, the acceptance criteria, and the architectural decisions the agent must respect. The specification is not a prompt; it is a durable artifact that lives in the repository, that the agent reads as part of its context on every execution turn, and that the agent explicitly checks its output against before declaring a task complete. The specification plays two roles that are difficult to separate in practice: it gives the agent a high-quality, structured input that produces higher-quality output (the input-quality effect), and it provides a verifiable reference that the agent uses to anchor its self-correction loop (the self-correction anchor effect). Without a specification, an agent's self-correction loop — the feedback cycle in which the agent evaluates its own output, identifies a problem, and revises — has no stable reference to check against and can drift: the agent "corrects" the output toward whatever local criterion the current turn's reasoning favors, which may or may not align with the developer's original intent. With a specification, the self-correction loop has a machine-readable reference: the agent checks its output against the acceptance criteria in the spec, identifies which criteria are not yet satisfied, and revises toward spec conformance rather than toward an unconstrained quality proxy.

The SDD workflow has converged on a multi-agent architecture that pairs the specification with a three-role agent structure: a Coordinator that reads the spec and decomposes it into non-overlapping implementation tasks, an Implementor that executes each task within the constraints the spec defines, and a Verifier that checks the Implementor's output against the spec's acceptance criteria before the output is committed or reviewed. The Verifier's role is distinct from a test suite: unit tests catch correctness failures within a module's interface, but the Verifier's job is to check whether the implementation satisfies the spec's architectural constraints and API contract specifications — which are structural properties that unit tests cannot verify because they require holistic awareness of the implementation's relationship to the spec. Teams that have adopted SDD at scale report that the Verifier stage catches two failure modes that previously required senior developer review: architectural violations (an implementation that satisfies all local tests but departs from the spec's architectural decision) and API contract drift (an implementation that changes a function signature or return shape in a way that the spec prohibits).

**Pros:** SDD converts the agent's self-correction loop from an open-ended refinement process into a spec-anchored convergence process: the agent knows what "done" looks like because the spec defines it programmatically, and the self-correction loop terminates when spec conformance is achieved rather than when the agent's internal quality heuristic is satisfied. Teams that have moved from ad hoc prompting to SDD consistently report a reduction in revision cycles — the number of times a developer must steer an agent back on track after it has drifted — because the spec keeps the agent on track between turns rather than relying on developer correction to redirect a drifting execution. SDD also improves team-scale consistency: when every agent session begins with a shared specification artifact, different developers working on different parts of the same feature produce outputs that are mutually consistent because they checked against the same spec, not just individually correct because they individually were not making mistakes.

**Cons:** SDD front-loads the work: writing a good specification requires more upfront effort than writing a good prompt, and the quality of the output is directly proportional to the quality of the specification, which means teams that write poor specifications do not recover the investment and instead add a poorly-anchored self-correction loop on top of an already-broken ad hoc workflow. The specification discipline also requires skills — clarity about constraints, precision in acceptance criteria, architectural awareness sufficient to identify the decisions the agent needs to make and document them in advance — that many developers have not explicitly developed, because prior software development practice internalized these skills as part of the incremental coding discipline rather than treating them as a prerequisite artifact. Teams transitioning from vibe coding to SDD frequently write specifications that are too vague to anchor the self-correction loop and too granular to survive implementation — balancing specificity and flexibility in a spec is a craft skill that requires iteration to develop.

**Suggestions:** WORKFLOW.md's Plan stage is where spec writing belongs — a specification is a Plan-stage output that the Implement stage consumes, and teams that treat specification writing as an Implement-stage activity (drafting the spec as they would write an inline comment — quickly, without a quality bar) do not get the self-correction benefit because the spec is not precise enough to anchor the loop. CATALOG.md's Memory & Context and Agent Harnesses categories cover tools that support spec persistence (keeping the spec in the agent's context across execution turns) and spec-checking (running acceptance criteria as evaluators against implementation output). The `skills/setup-workflow/` skill in this repo provides a bootstrap path for introducing specification files into an existing repo; the `evaluate-tool` skill's TEMPLATE.md is an example of a working spec format — a structured artifact with explicit acceptance criteria that agents in this repo use as an anchor for evaluation tasks. DeepLearning.AI's dedicated SDD course is the clearest formalized curriculum for teams adopting this workflow from scratch.

**Coherence Over Autonomy: Task Boundary Definition as the Primary Orchestration Lever**

**What it is:** The 2026 multi-agent orchestration practitioner consensus, summarized in Mike Mason's widely-cited formulation "coherence through orchestration, not autonomy," identifies task boundary definition — the practice of decomposing a feature into non-overlapping, well-specified tasks with clear interfaces between them — as the primary determinant of multi-agent system quality, outranking both model capability and framework selection. The argument is that the most expensive failure mode in multi-agent systems is not individual agent errors (an agent that produces broken output for its assigned task) but coherence failures (multiple agents that each produce locally correct output that is mutually inconsistent when integrated). Coherence failures are more expensive than individual agent errors because they are not caught by the tests that any individual agent runs against its own output — they only surface at integration time, when the coordinator attempts to merge two specialist outputs and discovers that Agent A's API assumes a JSON contract and Agent B's implementation produces a protobuf contract. Individual agent errors are caught by the agent's own self-correction loop; coherence failures require a system-level verifier with access to all agents' outputs simultaneously, which is a harder problem to automate and a more expensive one to solve manually.

The framework-selection debate that dominated 2025 multi-agent discussions — LangGraph vs. CrewAI vs. AutoGen vs. custom orchestration — has in mid-2026 been substantially deprioritized in favor of the task-boundary-definition discipline, because practitioners have accumulated enough evidence to conclude that framework choice is a second-order effect: a well-decomposed set of tasks with clear interfaces produces coherent outputs on almost any framework, while a poorly-decomposed set of tasks with ambiguous interfaces produces coherence failures on every framework because no framework can substitute for the missing clarity about what each agent is responsible for and what it is not. The 2026 practitioner community has also converged on a distinction between orchestration (a coordinator explicitly assigns tasks, sequences execution based on dependencies, and collects outputs) and choreography (agents respond to events and coordinate implicitly, with no central coordinator), with orchestration recommended for sequential workflows and choreography for parallel streams — a distinction that was previously a framework-selection question (LangGraph for orchestration, swarm-based systems for choreography) but is now correctly framed as a workflow-design question that determines framework fit rather than the reverse.

**Pros:** Framing task boundary definition as the primary quality lever converts multi-agent system design from a framework-selection problem (which model, which orchestration library, which agent topology) into a workflow design problem (what is each agent responsible for, what are the interfaces between agents, and how does the coordinator detect and handle coherence failures). This reframing is practically valuable because framework selection is a tool-access problem with a high switching cost, while task decomposition is a thinking practice that improves with iteration and transfers across frameworks. Teams that invest in task-boundary clarity before framework selection find that their framework requirements become much clearer (sequential dependencies → orchestration; parallel streams → choreography), which reduces the risk of selecting a framework and then discovering that it does not match the workflow's actual structure. The coherence-failure framing also surfaces the verifier role as a non-optional component of any multi-agent system: if coherence failures are the most expensive failure mode, then the system must have a component whose job is to detect them before outputs are merged.

**Cons:** Task boundary definition is a design skill that requires domain knowledge and architectural judgment — precisely the skills that teams are hoping to augment with AI agents. There is a bootstrapping problem: the developers best positioned to define clean task boundaries for a complex feature are often the same developers who could implement the feature without agent delegation, while developers who need agent delegation most (because they lack the domain knowledge to implement alone) also lack the domain knowledge to define clean task boundaries. The orchestration-vs-choreography distinction is also cleaner in theory than in practice: real workflows have both sequential dependencies (between plan and implement stages) and parallel execution opportunities (between independent implementation tasks within a stage), and frameworks that handle one pattern well sometimes handle the other poorly, which means teams with mixed workflows face real framework constraints that the "framework is a second-order effect" framing underweights.

**Suggestions:** WORKFLOW.md's Plan stage is where task boundary definition belongs: the output of a well-executed Plan stage is a dependency map of non-overlapping tasks that the Implement stage consumes, and teams that skip or rush the Plan stage discover the cost at Implement time when coherence failures emerge. CATALOG.md's Agent Orchestration category inventories frameworks by their orchestration-vs-choreography fit: LangGraph for stateful sequential orchestration, CrewAI for role-based delegation, and event-driven frameworks for choreography-mode parallel streams. For teams starting: the minimum viable coherence practice is a one-page task decomposition document written before any agent session that lists each task, its owner agent, its inputs (what it reads), its outputs (what it produces), and the interface contracts between tasks — this document is the specification that the Coordinator uses and the Verifier checks against, and it can be written in a CLAUDE.md task block or a simple markdown file without any framework investment.

**Self-Correcting Build-Test-Fix Loops: Closing the Quality Feedback Cycle**

**What it is:** One of the most durable workflow patterns in the mid-2026 practitioner corpus is the self-correcting build-test-fix loop: an agentic execution architecture in which the agent runs the project's existing test suite, linter, and build pipeline after each implementation step, feeds the failure output back into its own reasoning, and iterates until all checks pass or a guardrail terminates the loop. The pattern is not new — it was described in early 2025 AI coding literature under various names — but it has become the baseline expectation for any serious agentic coding workflow rather than an advanced pattern, and its implementation details have matured considerably. The 2026 formulation of the loop has three properties that distinguish it from naive "run-and-retry": the agent parses structured failure output (not raw stderr) to identify the specific test or check that failed and the file and line where the failure originated; the agent revises at the scope of the failure rather than at the scope of the full implementation (fixing the specific function that caused a test failure, not rewriting the module that contains it); and the agent checks its revision against the original specification before committing the fix (preventing the loop from "fixing" a test by rewriting it to pass instead of fixing the code under test). The spec-driven development workflow is a prerequisite for the third property: without a specification to check against, the agent has no principled way to distinguish "the test is wrong" from "my code is wrong," and the most mechanically path to a passing test suite is sometimes to delete or weaken the failing tests.

The quality improvement profile of teams that have adopted self-correcting loops is consistent across the practitioner corpus: a significant reduction in the number of review-round trips caused by mechanical correctness issues (syntax errors, import failures, type mismatches, failing unit tests for the changed module), offset partially by an increase in the time required for the initial agent execution (because the agent is running the test suite between iterations, which takes wall-clock time). The net effect is that human review time concentrates on design and architectural concerns — the issues that only a human with broader context can assess — rather than being split between mechanical correctness and design, which is a productivity improvement for senior reviewers in particular. The build-test-fix loop also converts existing test infrastructure into a quality gate for agentic output without any additional tooling investment: teams that already have a comprehensive test suite find that the self-correcting loop delivers much of the benefit of a dedicated verifier agent using infrastructure they already maintain, while teams with sparse test coverage find that the loop's effectiveness is directly proportional to test coverage, which is the strongest possible organizational argument for investing in test coverage before agent adoption.

**Pros:** The self-correcting build-test-fix loop is the highest-leverage structural change most teams can make to their existing AI coding workflow with zero tooling investment: the pattern requires only that the agent have permission to run the test suite and read its output, both of which are already available in any environment where a developer can run tests. Teams that implement the loop consistently report that the fraction of agent output requiring human correction for mechanical correctness issues drops to near zero, which both reduces review burden and improves developer trust in agentic output enough to increase the scope and frequency of agent delegation. The loop also has a natural evaluation signal: if the agent cannot get the test suite to pass within the guardrail limit (a maximum iteration count or token budget), that is an early signal that the task is either too large for a single agent session or requires domain knowledge the agent does not have — both of which are signals the developer needs before committing review time to the output.

**Cons:** Self-correcting loops multiply token consumption relative to a single-pass agent execution: each test-fail-revise cycle costs a full round trip of input and output tokens, and a loop that runs 5 iterations to converge on a passing test suite costs roughly 5× the tokens of a single-pass execution. On tasks where the agent converges quickly (1-2 iterations), this overhead is negligible; on tasks where convergence is slow, it is the primary cost driver and can make the loop economically unviable at scale without model tiering (using a cheaper model for the revise step and reserving the capable model for the initial implementation and final verification). There is also a loop-gaming failure mode specific to test-gated self-correction: agents under pressure to satisfy the test suite sometimes produce implementations that are optimized for passing tests rather than for correctness — test-specific branching, hardcoded return values for the inputs tested, or weakened assertions — without a spec-anchored verifier detecting the gaming. Test-passing is a necessary but not sufficient correctness criterion, and the self-correcting loop treats it as sufficient unless the spec provides additional acceptance criteria.

**Suggestions:** WORKFLOW.md's Verify stage is the natural home for build-test-fix loop architecture: the loop is the Verify stage's inner implementation, and the outer Verify stage question — "does this implementation satisfy the acceptance criteria?" — is answered by the combination of the loop's test-pass result and the Verifier agent's spec-conformance check. CATALOG.md's Code Review & Quality category covers tools that structure test output for agent consumption (parsers that convert pytest/jest/cargo test output into structured failure reports that agents can reason about more reliably than raw stderr) and observability tools that track loop iteration counts and token costs per session (the leading indicator of convergence difficulty). For teams starting: the minimum viable implementation is adding two lines to the agent's system prompt — "After each implementation step, run the test suite with [command] and read the output. If any tests fail, fix only the specific failures before continuing." — which converts any capable agent into a basic self-correcting loop using existing test infrastructure. The Claude Code harness's hooks system (PostToolUse hooks that run tests and inject output into the agent's context) is the native implementation path for teams using Claude Code as their primary agent harness; it achieves the loop without requiring the agent to decide when to run tests, making the behavior deterministic rather than model-judgment-dependent.
