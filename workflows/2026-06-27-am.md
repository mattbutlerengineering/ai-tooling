**The Reflection Pattern: Self-Critique as Standard Quality Gate**

**What it is:** The reflection pattern — generate an output, have an agent (or a second instance of the same model) critique that output against explicit criteria, then regenerate incorporating the critique — is in mid-2026 transitioning from an experimental technique used by researchers into a standard quality-assurance step in production agentic development pipelines. The pattern's adoption is being driven by a pair of benchmark results that practitioners have found consistently reproducible on their own tasks: the reflection pattern alone pushes coding benchmark accuracy from 80% to 91%, and combining it with external verification tools (test runners, linters, type checkers) can push accuracy gains past 30 percentage points above the single-pass baseline. These numbers have proved sufficiently robust across task types that "did you try reflection?" has become the first diagnostic question practitioners ask when an agentic workflow produces inconsistent output quality. The mechanism behind the gain is well-understood: single-pass generation is bounded by the distribution of plausible completions the model can produce from the initial prompt context; reflection breaks that bound by introducing a second reasoning pass that specifically targets the failure modes of the first — incorrect assumptions, missed edge cases, style violations, missing test coverage — that the model would not have caught in the original generation pass because it was not prompted to look for them.

The implementation pattern has two principal variants. The self-reflection variant uses the same model instance to critique its own output, prompted explicitly to find problems rather than validate the work. The external-reflection variant uses a second model instance — sometimes a smaller, faster model — to evaluate the generating model's output against a rubric, which provides partial immunity to the systematic biases of the generating model (if the generator consistently overestimates its own confidence, the external evaluator is not subject to that bias). In production coding workflows, the distinction between the two variants often comes down to cost: self-reflection doubles the model call count but avoids routing overhead; external reflection allows the evaluator to be a cheaper model, which can make the two-model pattern cost-neutral relative to a single expensive model call. Practitioners report that the external-reflection variant also enables a natural escalation gate: an evaluator that repeatedly marks the generator's output as failing a quality criterion is a diagnostic signal that the task exceeds the generator's reliable capability, surfacing information the single-pass workflow cannot produce without human intervention.

**Pros:** The reflection pattern's strongest argument is the benchmark evidence: an 11-percentage-point lift in coding accuracy from a single architectural change justifies the implementation overhead on its own. Practitioners report that the lift is most pronounced on tasks where the agent is working at the edge of its competence — complex algorithmic problems, multi-file refactors, test generation for code with implicit assumptions — precisely the tasks where single-pass generation is most unreliable and where human review is most expensive to substitute. The pattern also introduces a forcing function for specifying quality criteria explicitly: the reflection prompt requires the developer to write down what a good output looks like — does it pass the type checker, does it handle null inputs, does it follow the project's naming conventions? — a discipline that pays dividends beyond the reflection loop itself as a lightweight spec and onboarding artifact. When the external-reflection variant is deployed, the two-model architecture also provides a built-in cost optimization path: routing first drafts through a smaller, cheaper evaluator and only escalating to the full generator for revision passes can produce net model-call costs lower than the baseline single expensive model per task.

**Cons:** The reflection pattern's primary cost is latency: a generate-then-reflect workflow has at minimum twice the model call count and roughly twice the wall-clock time of the equivalent single-pass workflow. On tasks where single-pass generation is sufficient — autocomplete, simple refactors, boilerplate generation — the overhead is unjustified by the quality improvement; applying reflection universally rather than selectively is one of the most common over-engineering errors practitioners identify in agentic workflow design. The pattern also introduces a failure mode that is distinct from single-pass generation errors: "sycophantic reflection," where the model in the critic role praises the output of the model in the generator role rather than critiquing it, particularly when both roles are played by the same model instance. A critic that consistently agrees with the generator provides no quality lift and adds latency; reflection prompts require deliberate engineering to elicit genuine critique rather than validation, typically by explicitly instructing the critic to assume the output contains at least one mistake and to find it before approving. The benchmark accuracy numbers are also measured on coding tasks with verifiable correct answers and may not generalize to ambiguous design-level decisions; teams applying the pattern to architecture choices rather than implementation code report smaller and less consistent gains.

**Suggestions:** WORKFLOW.md's Verify stage is the structural home for the reflection pattern — it is a verification technique applied to implementation-stage output, and the tooling that makes it work (external evaluator models, quality rubrics, reflection prompt templates) belongs in the Verify-stage toolset. CATALOG.md's Agent Orchestration category covers the frameworks that implement the evaluator-optimizer architecture as a built-in pattern: LangGraph's conditional edges route output to a reflection step based on evaluator scoring, and the major agentic harnesses expose configurable reflection passes as session-level parameters. For teams starting: the minimum viable reflection deployment is a single self-reflection pass on high-stakes agent outputs with a reflection prompt that names three specific quality criteria — correctness (does this do what was asked?), robustness (does this handle edge cases?), and style (does this match the codebase conventions?) — before delivering the output. That three-criterion prompt produces measurable quality improvement with roughly twenty minutes of implementation work, and the evidence from that deployment builds the case for external evaluators and automated rubrics where self-reflection lift is insufficient.

**Graduated Human-in-the-Loop Oversight: Risk-Tiered Governance for Agentic Code**

**What it is:** The August 2, 2026 compliance deadline for the EU AI Act — which in Article 14 mandates that high-risk AI systems provide human oversight tools enabling "effective oversight by natural persons" that is trained, measurable, and provable — is accelerating an organizational practice that leading enterprise engineering teams had already begun independently: replacing flat human oversight (review everything, or review nothing) with a graduated, risk-tiered model that routes agent actions through different review tracks based on the action's regulatory impact, customer proximity, reversibility, and data sensitivity. The Oversight Classification Model (OCM) that has emerged from regulated-domain practitioners classifies code generation tasks and agent actions into three tiers: human-in-the-loop (HITL, for strategic and regulatory functions requiring synchronous human approval before the action executes), human-over-the-loop (HOTL, for customer-impacting code requiring asynchronous human review of agent output before it ships, with the agent proceeding in parallel to the review), and automated-with-monitoring (AWM, for internal low-risk code where the agent operates autonomously with post-hoc audit logging). The practical benefit of the graduated model over flat review is that it directs human attention to the decisions that require human judgment — the irreversible, high-impact, regulatory-boundary-crossing actions — while allowing agents to operate at full autonomy for decisions that do not. A flat review model that routes all agent output through human review, in teams operating at agentic scale, creates a human review bottleneck that eliminates the throughput advantage of agentic development entirely; the graduated model resolves the throughput-oversight tension by matching oversight intensity to decision risk rather than applying it uniformly.

The compliance driver also creates an organizational artifact requirement that did not exist in informal oversight practices: the EU AI Act's "demonstrable, trainable, and provable" language requires that oversight decisions be logged, that the classification criteria be documented, and that the humans performing oversight be trained against those criteria — which means the classification model itself becomes a compliance artifact that must be maintained and audited alongside the code it governs. Teams operating in regulated industries (financial services, healthcare, infrastructure) that have implemented a documented OCM with audit logs are finding that the same artifacts satisfy audit requirements from internal risk and security functions previously handled by ad-hoc code review processes, so the governance artifact does double or triple duty. The confidence-based escalation pattern — where agents signal their own uncertainty as a trigger for HITL rather than waiting for a human to audit the output — is increasingly deployed as a complement to the OCM: agents that rate their confidence in an output below a configurable threshold automatically route the action to the HITL tier, shifting the overhead of uncertainty detection from human reviewers to the agent itself.

**Pros:** The economic case for graduated oversight is straightforward: the bottleneck of flat human review is eliminated for the 70-80% of agent actions that fall into the AWM tier on a typical codebase, while the most consequential 20-30% receive oversight proportional to their risk. Practitioners report that the primary productivity gain from agentic development — the ability to delegate full tasks to agents operating in parallel — is only realizable when human oversight is not required for each individual agent action; the graduated model is the enabling structure that makes delegation-at-scale tractable without eliminating oversight entirely. The graduated model also produces a useful organizational forcing function: the process of classifying agent actions by OCM criteria surfaces implicit risk assumptions that were previously undocumented, and the documentation of those assumptions is itself a risk reduction measure independent of the oversight it triggers. Teams report that the classification exercise catches high-risk agent actions they had been running without oversight because the risk had never been made explicit — the OCM's value is partly in the assessment process, not only in the routing it produces.

**Cons:** The OCM classification is only as reliable as the criteria used to classify actions, and practitioners report that the criteria clearest in principle — reversibility, data sensitivity — are frequently ambiguous in practice. A code change that modifies a shared library is reversible in the sense that it can be reverted, but irreversible in the sense that if it ships to production before the revert it may have affected customer data; the HITL vs HOTL vs AWM classification depends on how the team defines "reversibility," and different engineers in the same team often classify the same action differently until the criteria are operationalized with specific examples. The compliance driver also creates a risk that governance artifacts become compliance theater — documentation that satisfies auditors but does not reflect actual oversight practice — if the classification criteria are defined at a level of abstraction that does not map to the specific agent actions the team's workflows produce. Teams that have implemented graduated oversight successfully report that the criteria had to be refined through several rounds of reviewing actual agent actions before they were specific enough to be consistently applied; the documentation satisfying the EU AI Act's "trainable" requirement is produced after that operationalization work, not before it.

**Suggestions:** WORKFLOW.md's Review and Ship stages are the natural homes for graduated oversight governance — the checkpoint design decisions that determine what triggers human review before ship are Ship-stage decisions, and the audit logging infrastructure that makes oversight "provable" for compliance belongs in the Observability toolchain that CATALOG.md's Observability category covers. CATALOG.md's Security & Safety category covers the identity and permission-management tools (agentic identity frameworks, least-privilege permission models) that are a prerequisite for the AWM tier: an agent operating without synchronous human oversight must be constrained by permissions and audit logging that make its autonomous actions recoverable and traceable, satisfying the "demonstrable oversight" standard even for actions that do not require synchronous human approval. For teams starting: the minimum viable OCM implementation is a two-tier model rather than three — HITL for irreversible or regulatory-boundary-crossing actions, AWM for everything else — with the classification criteria documented in a single page that every team member reviews and applies consistently. The three-tier model (adding HOTL) becomes worth implementing after the two-tier model is operational and the team has enough classification examples to operationalize the middle tier without ambiguity.

**ReAct: Reasoning and Acting in Alternating Steps as the Default Execution Pattern**

**What it is:** The ReAct (Reason + Act) execution pattern — in which the agent alternates between reasoning about what to do next (producing a chain-of-thought about current state, what information is missing, and which tool call would most reduce that uncertainty) and acting (making the tool call, observing the result, and incorporating it into the next reasoning step) — has emerged in mid-2026 as the dominant execution primitive for complex coding tasks, displacing both the single-prompt pattern (one prompt, one output) and the static plan-execute pattern (produce a complete plan, then execute it step-by-step without updating the plan as execution proceeds). The displacement is driven by a specific failure mode of the static plan-execute pattern that practitioners have identified and named: "plan drift," where the agent's plan is coherent at the moment it is generated but becomes increasingly misaligned with actual task state as execution proceeds, because the plan was generated without access to the tool outputs that execution produces. A plan to refactor a function that assumes a specific signature becomes invalid the moment execution reveals the actual signature is different; a ReAct agent encountering the same discrepancy updates its reasoning about approach in response to observed state, producing a revised action rather than executing an invalidated plan.

The ReAct pattern also has a distinct advantage for tasks where the agent's information about the problem is not monotonically improving — where each tool call reveals information that changes not just the current step but the overall strategy. Multi-file refactors, debugging sessions, and dependency resolution are canonical examples: the full plan cannot be written before execution because the information needed to write the plan is discovered during execution. The pattern has also been adopted by teams with observability and compliance requirements because the alternating thought-action structure produces a built-in trace of the agent's reasoning: each "reason" step is a logged thought, each "act" step is a logged tool call, and the resulting trace is readable as a narrative of how the agent arrived at its output. This observability property is absent from both single-prompt and static plan-execute patterns, making ReAct the natural pattern for regulated environments that require explainable AI outputs under Article 14 or equivalent requirements.

**Pros:** The observability advantage is the most practically valuable property of ReAct for production engineering workflows: the reasoning trace is a debugging artifact that makes agent failures diagnosable without replaying the run. When a ReAct agent produces an incorrect output, the trace reveals which reasoning step made the wrong assumption and which tool call produced the misleading information — a diagnostic resolution measured in minutes rather than hours of log archaeology. For tasks where the agent must navigate dynamic state — a codebase where files change while the task is in flight, a dependency graph where installing one package changes what's available for others — ReAct's continuous state observation is the only pattern that handles state changes without producing a corrupted output. Practitioners also report that ReAct agents are better at recognizing task infeasibility than plan-execute agents: a static plan executed against an impossible task runs to completion and produces a plausible-looking failure; a ReAct agent that observes infeasibility in an early tool call can terminate early and report the specific obstacle, saving the compute cost of executing the remainder of a doomed plan and giving the developer actionable diagnostic information rather than a confusing final output.

**Cons:** ReAct's primary limitation is that its sequential structure — reason, then act, observe, then reason again — does not parallelize: each reasoning step must complete before the next acting step begins, and each acting step must produce an observable result before the next reasoning step can use it. For tasks with independent sub-steps that could be executed in parallel, ReAct's sequential execution is substantially slower than a parallel execution pattern, and the performance gap grows with the number of independent sub-steps. Multi-agent orchestration frameworks address this by running multiple ReAct agents in parallel on separate subtasks, but that introduces inter-agent coordination complexity that the single-agent pattern does not have. The pattern's explainability property — the reasoning trace — also creates a false confidence risk: a reasoning trace that reads as coherent is not evidence that the agent's conclusion is correct, only that its reasoning was internally consistent; practitioners report that reviewers with access to the trace are more likely to approve incorrect outputs than reviewers who only see the output, because the trace narrows attention to verifying the reasoning steps rather than independently evaluating the conclusion. The trace is a debugging tool, not a correctness certificate, and treating it as the latter is a systematic review error.

**Suggestions:** WORKFLOW.md's Implement stage is the stage where the ReAct pattern decision is made — the choice between single-prompt, static plan-execute, and ReAct execution is a task-complexity routing decision that happens before implementation begins. CATALOG.md's Agent Orchestration category covers the frameworks that provide ReAct as a first-class execution primitive: LangGraph's conditional-edge graph structure models the reason-act-observe cycle as a built-in pattern with native loop and retry support, and the major agentic coding tools (Claude Code, Cursor, Aider) expose ReAct-style execution for tasks that require multi-step tool use. For teams starting: the simplest selection heuristic is whether the task requires tool calls whose output changes the approach. If the answer is yes — discovering the result of the first tool call would cause the agent to change its plan for the second — ReAct is the correct pattern; if not, single-prompt is sufficient and the latency overhead is unjustified. The three task types that reliably benefit from ReAct in coding workflows are debugging (each observation narrows the hypothesis space), dependency resolution (each install changes the available package set), and cross-file refactoring (each file read changes what changes are needed downstream).

**Multi-Agent Orchestration: The Build vs Buy Decision**

**What it is:** The proliferation of multi-agent orchestration frameworks in mid-2026 — LangGraph, CrewAI, OpenAI Agents SDK, AutoGen, and a growing set of cloud-managed platforms — has surfaced a decision that teams escalating from single-agent to multi-agent workflows must make before writing their first orchestration layer: build a custom orchestration layer or adopt a framework? The decision is structurally different from the typical build-vs-buy question in software because the "buy" options have meaningfully different expressive power, not just different API surface areas, and the wrong framework choice can require a full rewrite when task complexity exceeds the framework's design envelope. The framework differences practitioners identify as architecturally significant are: LangGraph models agents as nodes in directed cyclic graphs, with cyclic capability enabling loops, retries, and iterative reasoning that linear pipelines cannot express — the pattern that makes it the dominant choice for complex coding tasks with self-correction requirements. CrewAI implements role-based team organization, where each agent is assigned a named role (architect, developer, reviewer, security auditor) with a specific goal and backstory, optimized for tasks where different agents need genuinely different specializations and where the team metaphor maps cleanly to the task structure. The OpenAI Agents SDK is designed for tightly-scoped assistant workflows with clean multi-agent delegation — a high-abstraction interface that covers the delegation-and-handoff pattern well but does not expose the underlying loop and state primitives that complex self-correction workflows require. The "build" path — implementing a custom orchestration layer — remains common at teams operating at large scale or with strict latency and cost constraints, where framework abstraction overhead is measurable and the team has enough orchestration expertise that the framework's scaffolding provides less value than its cost.

The 2026 practitioner consensus identifies the selection criteria as: task complexity (does the task require loops, retries, and conditional routing, favoring LangGraph?), team structure (does the task map to a team of specialists, favoring CrewAI?), delegation simplicity (is the orchestration pattern primarily delegation and handoff, favoring the OpenAI Agents SDK?), and scale (is the latency and cost overhead of a framework measurable and unacceptable, favoring build?). Practitioners also report a second-order consideration that is frequently underweighted: framework community and tooling maturity. LangGraph's graph architecture has produced the largest library of published patterns (evaluator-optimizer, reflection, parallel-fan-out, human-in-the-loop checkpoint placement) because its structure makes those patterns expressible as configurations rather than custom implementations; a team that adopts LangGraph inherits community-validated solutions to the most common multi-agent design problems, which compounds the framework's expressive advantage with an adoption-time advantage.

**Pros:** Framework adoption eliminates a class of engineering work that is not differentiated: the infrastructure of agent communication, state handoff, error propagation, and retry logic is the same across most multi-agent tasks, and reimplementing it from scratch is engineering time spent on table stakes rather than the actual task. LangGraph's graph primitive in particular provides loop and conditional-routing capabilities that are substantially harder to implement correctly from scratch than they appear — the edge cases around loop termination, state snapshot-and-restore for retries, and conditional routing logic form a non-trivial engineering surface that framework adoption buys down. Framework adoption also provides the community's accumulated patterns: teams using LangGraph benefit from the large library of published evaluator-optimizer, reflection, and parallel-fan-out workflows rather than designing those patterns independently. For regulated environments with explainability requirements, frameworks with built-in trace logging and state inspection are directly valuable for compliance: the same trace infrastructure that makes debugging tractable satisfies the "demonstrable human oversight" requirements that the EU AI Act imposes on high-risk AI systems, producing compliance value from infrastructure that serves debugging value.

**Cons:** Framework adoption creates a dependency on the framework's design decisions, which can become hard constraints when task requirements exceed the framework's design envelope. Practitioners report that the most common cause of framework replacement in multi-agent workflows is not framework bugs or performance problems but expressive limitation: a team that adopted a high-abstraction framework for its initial use case finds that a subsequent use case requires loop and state primitives the framework does not expose, and must either rewrite in a lower-abstraction framework or implement the required primitives as workarounds within extension points not designed for them. The warning signal is consistent across frameworks: if the team is spending more time working around the framework than writing application code, the design envelope does not match task requirements. The proliferation of frameworks also creates a fragmentation problem for teams using multiple AI coding tools — a team using Claude Code, Cursor, and a separate CI orchestration system each with its own preferred orchestration framework must maintain three different orchestration layers or accept that some tools receive lower-quality orchestration support than others, reintroducing the coordination tax that orchestration frameworks are supposed to eliminate.

**Suggestions:** CATALOG.md's Agent Orchestration category is the direct reference for the framework comparison — LangGraph, CrewAI, AutoGen, and the cloud-managed platforms are all catalogued with their architectural patterns, overlap relationships, and adoption signals. WORKFLOW.md's Implement and Verify stages cover the decision points where orchestration architecture is most consequential: the choice of orchestration pattern determines both the implementation architecture and the verification strategy, since LangGraph's loop and conditional-routing structure creates different checkpoint opportunities than CrewAI's team-handoff structure. STACK.md's recommended installation includes LangGraph as the default for complex multi-step agentic tasks requiring self-correction or conditional routing. For teams starting: the lowest-risk entry point for multi-agent orchestration is the simplest framework that covers the first use case, not the most capable framework that might cover future use cases. A team whose initial use case is a two-agent delegation (planning agent hands off to coding agent) does not need LangGraph's graph expressiveness and should start with the OpenAI Agents SDK or CrewAI's simpler delegation pattern; the evidence from that deployment — specifically, whether the team is constrained by the framework's expressiveness — is the input to the next framework decision, not a theoretical assessment of future task complexity that invariably overstates the need for flexibility the team does not yet have.
