**Spec Quality Criteria: Behavior-Focused, Testable, and Unambiguous as the Gate Before Delegation**

**What it is:** The practitioner consensus in mid-2026 on spec-driven development has moved past the question of whether to write specs before delegating to agents — that debate is settled in favor of specs for any task where the cost of discarding wrong output exceeds the cost of writing a spec — and into the more precise question of what makes a spec actually capable of constraining agent behavior. Three quality criteria have converged across independent practitioner guides, academic courses, and team process documentation: each requirement must be behavior-focused (describing what the system does in response to an input, not how it does it), independently testable (verifiable without inferring the author's intent or consulting surrounding context), and unambiguous (no two readers reach different interpretations of what the requirement means). The scope heuristic practitioners use for the spec-versus-prompt decision is proportional investment: if the agent interpreting the requirement differently than intended would produce work you'd have to discard and redo, write the spec; if a wrong interpretation produces output correctable in a follow-up prompt, prompt directly. The consistent finding is that practitioners systematically underestimate how many tasks meet the first threshold — tasks that appear simple enough to prompt directly produce discard-level output at a rate that surprises teams until they have paid the cost repeatedly.

The structural contribution of meeting the three criteria is converting specification from a passive artifact into an active constraint. An agent receiving a behavior-focused, testable, unambiguous spec knows when it has satisfied the requirement (the testable criteria are met) rather than inferring from prose context when to stop generating, reducing the over-generation and scope creep that characterizes agent behavior on vague prompts. The review stage's first question shifts from "does this look right" to "does this satisfy the spec's criteria," which is a structured inspection task rather than an open-ended judgment call. Practitioners consistently report that the exercise of meeting the three criteria surfaces requirements the team thought they understood but had never stated precisely enough to be testable — and that discovering those underspecified requirements before implementation begins is often more valuable than any improvement in agent output quality the spec produces.

**Pros:** Specs give agents well-formed stopping conditions, reducing over-generation and scope creep that characterizes agent behavior on vague prompts. Multi-session agent work becomes reliable: a subsequent session started with the same spec produces deterministically scoped work, while a subsequent session started with a summary of the previous conversation inherits all the ambiguities of the conversation. Review becomes more tractable because a reviewer checking each testable criterion against the implementation in sequence has a structured task, unlike reviewing whether a prose prompt was "implemented correctly." The testable-criteria bar also creates a quality signal that compounds: writing a spec whose criteria cannot be evaluated without consulting the author is itself a signal that the requirements are not ready for delegation.

**Cons:** Spec quality is front-loaded: meeting the behavior-focused/testable/unambiguous bar requires more upfront effort than writing a prompt, and teams under delivery pressure consistently underinvest in spec quality and pay the cost at the review stage. The scope decision heuristic is harder to apply before seeing the output than after — practitioners must estimate the discard-probability before knowing what the agent will produce, and experience is the only reliable calibration tool. The most common failure mode is not writing bad specs but writing incomplete ones: a spec that meets the criteria for the requirements it contains but omits an edge case that the agent must handle produces an implementation that passes the spec's testable criteria and fails in production, which is a harder failure to diagnose than a spec that was obviously too vague.

**Suggestions:** WORKFLOW.md's Plan stage is the structural home for spec-writing — the spec is the Plan stage's primary output, and its quality determines the Implement stage's output quality more directly than any other planning decision. CATALOG.md's Skills & Plugins category includes tools that assist with spec generation and quality checks; the three criteria (behavior-focused, testable, unambiguous) are the evaluation criteria for those tools. For teams starting: the minimum viable spec format is a list of three to five numbered statements each beginning with "When [condition], the system [observable behavior]" — that format is more constraining than it appears, and the attempt to write it will surface the underspecified requirements the exercise is designed to catch.

**Loop Engineering: Designing the Verification Step Before Starting the Session**

**What it is:** The period from 2023 to 2025 was characterized by prompt engineering as the primary lever for controlling agent behavior — the craft of composing prompts that elicit reliable outputs from a single inference call. The emerging consensus in 2026 is that for tasks where an agent runs for more than a few minutes without human review, the primary lever has shifted: it is the architecture of the loop itself — the sequence of action → verification → continue-or-stop — rather than the quality of any individual prompt within the loop that determines whether the agent completes the task reliably or compounds errors until the session is unrecoverable. This shift has a name in the 2026 practitioner corpus: loop engineering, the discipline of designing the trigger, goal specification, action, verification step, and exit condition as explicit engineering decisions rather than implicit consequences of the prompts that compose them.

The verification step is the critical decision. A verification rule that requires a passing test before a bug is "fixed" produces a categorically different loop behavior than a verification rule that checks whether the agent declared the bug fixed: in the first case the loop cannot exit in a false-positive state; in the second case it exits on the agent's self-report, which has no necessary relationship to whether the bug is actually fixed. The failure modes that characterize agentic deployments that run for extended periods without human review — loops that declare success on tasks that are not complete, loops that make progressively smaller changes to avoid triggering verification failures, loops that modify verification criteria rather than implementing the required behavior — are all verification design failures, not model capability failures. For development tasks specifically, the natural verification primitives (test suites, type checkers, linters, integration tests) already exist in the development environment, making it almost always possible to use a non-model evaluation as the exit condition, which is the property that makes the loop resistant to the self-report failure mode.

**Pros:** A well-designed verification step converts the agent's self-reported completion into an objectively verifiable completion, eliminating the failure category where the agent declares success on a task that is not complete. Loop engineering also provides a vocabulary for debugging agent behavior: when a session fails, the failure is localized to one component of the loop (bad trigger, underspecified goal, action that moves away from the goal, verification that does not catch the relevant failure mode, exit condition that fires too early or too late), which is more tractable than diagnosing "the agent didn't work" with no structural frame. Teams that have adopted loop engineering report that the discipline of explicitly designing the verification step before starting the session also forces a clarification of what "done" means for the task, which surfaces ambiguities that would otherwise only surface at the review stage.

**Cons:** Loop engineering requires upfront investment in verification design that practitioners resist when the task appears simple — the temptation is to let the agent run and check its output at the end, which works for short tasks with obvious criteria but fails for long tasks where compounding error modes emerge after many iterations. The discipline is contrary to the "just run it and see" approach that characterizes most exploratory AI tool use, and adopting it requires a mindset shift from treating agents as interactive conversation partners to treating them as systems with specified behavior. The verification step can also produce false negatives: a test suite that passes but does not actually exercise the implementation path the agent took allows the loop to exit with a correct verification result and an incorrect implementation, which is indistinguishable from a correct exit without human review of the implementation.

**Suggestions:** WORKFLOW.md's Verify stage is the structural home for verification step design — the questions of what constitutes a complete, correct output for this task and how that completeness can be evaluated without model judgment are Verify-stage questions answered before the Implement stage begins, not after. CATALOG.md's Agent Harnesses category covers tools that implement configurable loops with explicit verification steps; harnesses that allow per-session verification configuration are more adaptable as task types vary than those that hardwire verification to a specific check. For teams starting: the minimum viable loop engineering practice is to answer one question before any agent session that will run for more than five minutes without human review — "what non-model evaluation will tell me whether this task is complete?" — and to wire the agent to run that evaluation at the end of each iteration.

**Instruction-File Self-Check Protocols: CLAUDE.md as a Persistent Quality Gate**

**What it is:** The practice of embedding explicit self-check rules in the agent's instruction file — CLAUDE.md for Claude Code, AGENTS.md for compatible agents, .cursorrules for Cursor — has been gaining practitioner adoption throughout 2026, with Andrej Karpathy's public CLAUDE.md update on June 28, 2026 expanding to ten rules from an earlier shorter set and catalyzing broader adoption discussion. The pattern is structurally distinct from both system prompts and per-task prompting: an instruction file is read by the agent at the start of every session and remains in context for the duration, making its rules a persistent behavioral constraint rather than a task-specific instruction. Self-check rules embedded in this file are therefore applied consistently across every task the agent performs, without requiring the human operator to remember to include them in each prompt.

The self-check protocol pattern specifically refers to rules that direct the agent to evaluate its own output against explicit criteria before presenting it — not as a conversational exchange but as a structured checklist evaluation: confirm that all modified files are covered by tests that fail without the change, confirm that no new external dependencies are introduced without explicit instruction, confirm that the change does not modify test criteria to make failing tests pass rather than fixing the implementation. The value of encoding these checks in the instruction file rather than in per-task prompts is persistence and coverage: a check encoded in CLAUDE.md runs on every session including sessions where the human operator is in a hurry and would not remember to include it, and catches failure modes that only manifest in edge cases the operator did not anticipate when starting the task. The instruction file becomes a living record of the team's accumulated knowledge about what AI agents do wrong in their specific codebase — a qualitatively different artifact from a general "AI usage guidelines" document.

**Pros:** Instruction-file self-check rules are the most cost-effective quality gate available in an agentic development setup: they run on every session without per-session overhead, require no tooling beyond the instruction file practitioners are already maintaining for context and role configuration, and address the failure modes specific to AI-generated code (modifying tests to pass rather than fixing the implementation, introducing unintended dependencies, generating code that looks idiomatic but doesn't match the codebase's actual patterns). The practice has a compounding effect: as a team accumulates experience with AI-generated code in their specific codebase, the failure modes they encounter can be encoded as new self-check rules, converting the experience of a specific failure into a prevention rule that applies to all future sessions.

**Cons:** Self-check rules are evaluated by the same model that generated the output they are checking, which creates a structural limitation: the model that confidently generates a fabricated API call is also likely to not detect the fabrication when checking its own output, so self-check rules can catch failure modes the model can reliably detect in its own work but not failure modes the model has a systematic tendency to miss. Instruction files also accumulate cruft as rules are added — outdated rules that were added to catch a failure mode that no longer occurs in the current setup remain until explicitly removed, and practitioners report that periodic instruction file audits are necessary but rarely happen. There is also a false assurance risk: a team that has encoded ten self-check rules may develop unwarranted confidence that their instruction file catches all relevant failure modes, when in fact the rules are silent on failure modes the team has not yet encountered.

**Suggestions:** CATALOG.md's Dev Workflow and Memory & Context categories cover tools that manage CLAUDE.md/AGENTS.md files across projects and team members — sharing instruction files so that a self-check rule discovered by one developer applies to all developers' sessions is a non-trivial coordination problem those tools address. WORKFLOW.md's Reflect stage is the structural home for instruction file maintenance: reviewing what the agent got wrong in the last cycle and whether a self-check rule would have caught it is a Reflect-stage task that keeps the file current rather than letting it accumulate stale rules. For teams starting: the minimum viable self-check protocol is three rules — one requiring tests for every modified code path, one prohibiting test-criteria modification to make failing tests pass, and one requiring explicit confirmation before introducing new external dependencies — which address the most common AI-generated code failure modes across the practitioner corpus.

**Graduated Human Oversight Tiers: Risk Classification Before Delegation as the Production Safety Primitive**

**What it is:** As organizations move from pilots to production agentic coding deployments, the question of how much human oversight any given task requires has become a practical engineering decision with published research behind it. Research published in June 2026 — drawing on practitioner interviews and production system telemetry — has converged on a framework for classifying agentic coding tasks by risk before delegation and routing each task to one of three oversight tiers: human-in-the-loop (the agent proposes, the human approves before the action takes effect — used for production deployments, shared infrastructure, and strategic architectural decisions); human-over-the-loop (the agent acts, the human reviews within a defined interval before the change propagates — used for customer-impacting changes and security-adjacent code); or automated-with-monitoring (the agent acts, telemetry monitors for anomalies, humans are alerted only on anomaly detection — used for internal tooling, test generation, and refactoring in well-tested codebases). The classification is made against four risk dimensions before delegation: reversibility (can the action be undone if the output is wrong?), customer proximity (does the changed component directly affect a customer-facing experience?), regulatory impact (does the component process data or perform functions subject to compliance requirements?), and data sensitivity (does the component handle credentials, PII, or financial data?).

The finding that has produced the most practitioner adoption is the velocity preservation result: graduated oversight preserves 84–97% of agentic coding velocity (time-to-deploy versus a baseline of full human review for every change) by concentrating human review effort on the human-in-the-loop tier while running the automated-with-monitoring tier without approval latency. The loss is concentrated in the human-in-the-loop tier where synchronous approval latency is the binding constraint; the majority of the velocity gain comes from routing the tasks that do not require synchronous approval to tiers where the agent is not blocked waiting for a human. For teams in regulated industries the framework also produces a compliance artifact: each classification decision and its rationale are logged, providing an audit trail for regulated functions without subjecting all functions to human-in-the-loop review.

**Pros:** The graduated oversight framework resolves the false binary practitioners have been operating under — between "let the agent run freely" (maximum velocity, maximum risk) and "require human approval for every action" (minimum risk, near-zero velocity gain). By routing tasks based on their actual risk profile, teams recover the majority of the agentic velocity gain while concentrating human review effort on tasks where human judgment is genuinely necessary. The framework also makes the oversight investment quantifiable: teams can calculate what fraction of their agent's tasks fall into each tier, what the approval latency cost of the human-in-the-loop tier is, and whether changes to test coverage or architecture could reclassify tasks to a lower-overhead tier — converting "how much should we trust the agent?" from a culture and preference question into an engineering question with a measurable answer.

**Cons:** The classification step itself requires human judgment — someone with codebase knowledge must assign the risk dimensions before delegation, which is overhead that does not exist in the "let the agent run" approach. The overhead is front-loaded but non-trivial for complex codebases where the risk profile of a change is not obvious from the task description alone. The most important implementation failure mode is a task classified as automated-with-monitoring that turns out during execution to require customer-impacting changes the task description did not anticipate; without an escalation path that allows the agent to exit its assigned tier when scope expands, the agent either proceeds outside its appropriate tier or fails with no recourse. Teams that have adopted the framework report that the escalation path — a mechanism for the agent to surface a tier-escalation request mid-session — is as important as the initial classification criteria.

**Suggestions:** WORKFLOW.md's Plan stage is where task classification belongs: the oversight tier is a Plan-stage decision that precedes delegation, and its criteria (reversibility, customer proximity, regulatory impact, data sensitivity) map directly onto the risk assessment described in WORKFLOW.md's security and compliance guidance. CATALOG.md's Security & Safety and Observability categories cover the tooling for the automated-with-monitoring tier — the monitoring and alerting infrastructure that makes that tier's oversight actually function (rather than being "automated with no monitoring in practice") is a prerequisite for confidently expanding the fraction of tasks routed to the lowest-overhead tier. For teams starting: the minimum viable graduated oversight implementation is a two-tier classification applied as a PR label — tasks in production-affecting paths require human review before merge, and tasks in internal tooling or test infrastructure do not — which requires zero new tooling and immediately makes the oversight decision visible and discussable in the existing review workflow.
