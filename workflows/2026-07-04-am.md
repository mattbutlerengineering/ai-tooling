**Loop Engineering: Designing Agentic Loops with Verifiable Exit Conditions**

**What it is:** An agentic loop is a trigger plus a verifiable goal — the agent executes until the goal is met, rather than responding to a single prompt. Loop engineering is the practice of deliberately designing those loops by specifying the goal in testable terms, defining when to stop, and building guardrails that prevent runaway execution. The canonical loop architecture in 2026 builds on the ReAct pattern (Reason + Act), where the agent alternates between reasoning about what to do next and actually doing it — step by step, tool call by tool call, with each cycle producing a Thought, an Action, and an Observation that inform the next iteration. The Reflection pattern, in which the agent critiques its own output at each iteration and decides whether to revise or submit, extends ReAct with a self-correction layer that has been measured to push coding benchmark accuracy from roughly 80% to 91% — a 10-point gain from loop structure alone, not from model improvement. A well-engineered loop has three defined components: an entry trigger (what starts it — a failing test, a lint error, an unfulfilled acceptance criterion), a verifiable goal (a condition the agent can mechanically evaluate — all tests pass, no linter errors, a specific file exists with specific content), and an exit guardrail (a maximum iteration count or wall-clock timeout that prevents the loop from continuing past the point of diminishing returns). Loop engineering is distinct from prompt engineering in that it operates at the level of the execution architecture — how the agent sequences and checks its own work — rather than the level of a single instruction. A well-engineered loop can recover from errors that would stall a single-shot prompt; a poorly-engineered loop can run indefinitely, consuming tokens, modifying files in unexpected ways, and producing output that drifts farther from the goal with each unchecked iteration.

**Pros:** Verifiable exit conditions are the single most important property of a reliable agentic loop: when the goal is defined in terms the agent can check mechanically (test suite passes, linter exits zero, file matches a schema), the agent knows without human judgment whether it has succeeded. This eliminates the failure mode where the agent declares completion based on heuristic confidence rather than objective verification — the most common source of agent-generated code that appears correct but contains subtle defects. The Reflection pattern specifically addresses the quality gap between first-draft and publishable output: an agent that critiques its own work and revises based on the critique converges on higher-quality output than an agent that submits its first draft, even when the model and the instruction are identical. The build-test-fix loop — where the agent's exit condition is a clean CI run — is the most widely adopted instantiation of loop engineering in 2026 because it requires no novel infrastructure: the test suite and linter that the project already has become the agent's verifiable goal, and the loop runs until they pass or the guardrail fires. This pattern also produces a feedback signal that benefits the development team independently of agentic coding: a test suite precise enough to serve as an agent's exit oracle is a more useful test suite for human developers too, since it catches the same class of regressions with the same mechanical reliability.

**Cons:** Verifiable exit conditions are only as reliable as the tests and linters that implement them. An agent that loops until its test suite passes will ship broken code if the test suite does not cover the failure mode the agent introduced — and agents have a consistent failure mode profile (hallucinated API calls, subtle type errors, missing edge-case handling) that is often orthogonal to the failure modes existing tests were written to catch. Loop guardrails that fire on iteration count rather than on a meaningful convergence signal produce a failure mode of their own: an agent that hits its iteration cap mid-task produces a partially-complete output that may be more dangerous than no output at all, because it appears complete from the outside. The Reflection pattern adds latency and cost proportional to the number of revision iterations: a loop that runs 4 reflection cycles consumes roughly 4× the tokens of a single-shot invocation for the same task, and the quality improvement is non-linear — the first reflection cycle yields the highest gain, with diminishing returns on subsequent cycles. The ReAct pattern also accumulates context with each Observation: a loop running 10+ iterations with detailed tool outputs in each Observation can exhaust the context window before reaching the exit condition, producing truncation errors that are difficult to diagnose because they manifest as apparently random failures rather than explicit out-of-memory signals.

For teams starting: the minimum viable loop engineering investment is to define explicit iteration caps for every agent task before the session begins — a cap of 10 iterations with a mandatory human checkpoint if the cap is reached is enough to prevent runaway execution while preserving the agent's self-correction capacity. **Code Health as AI Readiness: Technical Debt as an Agentic Blocker**

**What it is:** CodeScene's 2026 research establishes a quantified relationship between code health metrics and AI agent effectiveness: problem code lacking in quality is not AI-ready, and low code health increases the likelihood that agents fail on their task or burn excess tokens navigating complex, tightly-coupled codebases. The mechanism is structural — agents decompose tasks by reading and reasoning over the code that already exists, and code that is difficult for a human engineer to understand (high cyclomatic complexity, deep inheritance chains, large files with many responsibilities, inconsistent naming, missing tests) is correspondingly difficult for an agent to reason over correctly. The second data point from the 2026 corpus: Anthropic's agentic coding trends report finds that AI-generated pull requests wait 4.6× longer in review without PR governance — a signal that the bottleneck in agentic development has moved from code generation to review bandwidth, and that teams without explicit review governance accumulate a growing backlog of unreviewed agent output. These two findings converge on a single structural investment: code health and review governance are AI readiness infrastructure, not technical hygiene concerns separate from the agentic workflow. A team that adopts agentic coding without investing in code health and review governance discovers the bottleneck empirically — agents producing low-quality output from a low-quality codebase, with that output accumulating in a review queue that the team cannot clear at agentic velocity.

**Pros:** Framing code health as AI readiness converts a perpetually-deferred maintenance investment into a visible prerequisite for agentic productivity. Teams that have historically struggled to justify code health investment on maintenance grounds find the AI-readiness framing more tractable: the ROI calculation changes from "this reduces future maintenance effort" (diffuse, long-horizon, difficult to measure) to "this reduces agent failure rates and token costs on the tasks we're running this week" (direct, near-term, measurable against current agent session logs). The code health framing also provides a prioritization signal for technical debt reduction: rather than uniformly improving all code health metrics, teams can focus on the specific modules where agents are most frequently tasked — the files agents touch most often are the highest-leverage targets for code health investment because agent failure rates correlate most strongly with the health of the files the agent actually reads. The PR governance framing similarly provides a measurable optimization target: tracking PR-to-merge latency for agent-authored PRs, compared to human-authored PRs, reveals the review bandwidth gap directly and makes the case for the specific governance intervention (automated first-pass review, PR size discipline, CI hardening) that the data supports.

**Cons:** Code health improvement is expensive relative to the short-term throughput gains it produces — a team that pauses agentic coding to reduce technical debt in its highest-traffic modules will see reduced output volume during the investment period, and the productivity gains from the investment materialize over weeks rather than immediately. The AI-readiness framing also risks narrowing the code health investment to only the modules where agents currently operate, leaving the rest of the codebase — which agents may be tasked against in future cycles — in a state that will produce the same failure modes when agents encounter it. PR governance interventions (automated review, size discipline, CI hardening) all require configuration and calibration investment before they deliver the bandwidth gains they promise: an automated review agent that is miscalibrated for the codebase produces noise rather than signal, and a CI gate that is too coarse catches fewer agent failure modes than expected. The 4.6× PR review latency finding is a population average — teams with strong existing review cultures may see a smaller gap, while teams with weak review cultures may see a larger one, and the governance interventions appropriate for each differ substantially.

**Suggestions:** CATALOG.md's Code Review & Quality category is directly relevant: code health tooling (complexity analyzers, coupling metrics, test coverage analysis) is the measurement infrastructure that makes AI readiness quantifiable, and automated review agents are the governance infrastructure that addresses the review bandwidth gap. WORKFLOW.md's Reflect stage is where the code health feedback loop belongs — using agent session outcomes (failure rates, token costs, revision cycles per task) as a signal for which modules to prioritize for code health investment converts the Reflect stage into a continuous AI-readiness improvement cycle. The CodeScene blog (linked in Sources) provides the most rigorous published treatment of the code health / agent effectiveness relationship, including the specific metrics most predictive of agent failure. For teams starting: the minimum viable code health investment for AI readiness is to run a complexity analysis against the modules where agents are most frequently tasked, identify the top-five highest-complexity files in that set, and refactor those files before expanding agentic task scope — the investment is bounded, the benefit is direct, and the before/after comparison provides a concrete data point on the code health / agent effectiveness relationship in the team's specific codebase.

**Context Engineering as Primary Craft: Shifting from Prompt Design to Knowledge Architecture**

**What it is:** The 2026 practitioner consensus, synthesized from the State of AI Coding Agents survey and the spec-driven development literature, is that the primary skill in effective agentic development has shifted from prompt engineering (how to phrase a single instruction to a model) to context engineering (how to architect what the agent knows throughout a session and across sessions). The distinction is structural: a well-crafted prompt improves a single interaction; well-engineered context improves every interaction in a session and persists quality gains across sessions. Context engineering encompasses three layers. The first is session context — the information provided at session start via instruction files (CLAUDE.md, AGENTS.md, .cursorrules, or their equivalents), which establish the agent's behavioral baseline, codebase conventions, tool access, and quality expectations before the first user message. The second is task context — the information attached to individual task delegations, including the issue-style task description (background, goal, acceptance criteria, constraints, test commands), links to relevant files, and the spec or plan produced in a prior stage. The third is memory context — persistent state that survives across sessions, including architectural decision records, past agent outputs and their outcomes, and the evolving spec that accumulates behavioral contracts from previous implementation cycles. The practitioner observation that crystallizes the shift: for a non-trivial feature, effective agentic development takes 2–6 hours of human time — most of it review and clarification, not typing. The clarification time is a direct measure of context gap: every clarifying question the agent asks represents context that should have been in the session or task context at the start. Reducing clarification cycles is the measurable payoff of context engineering investment.

**Pros:** Context engineering is reusable in a way that prompt engineering is not. A well-crafted prompt solves one task; a well-crafted CLAUDE.md or AGENTS.md instruction file encodes the behavioral contract that governs every task the agent performs in the repository, without requiring the developer to re-specify conventions for each session. The compound return is significant: a team that invests one hour in writing a high-quality instruction file recovers that hour across every subsequent agent session in the repository, and the instruction file improves in precision as the team discovers gaps (from clarifying questions and agent failures) and patches them. Memory context — the persistent architectural state that accumulates across sessions — is the structural feature that converts agentic development from a session-by-session productivity tool into a learning system: an agent that begins each session with a record of past decisions, past outputs, and their outcomes can reason over the project's history rather than inferring it from the current state of the code, which substantially reduces the failure modes that arise from the agent not knowing why the code is structured the way it is. The Anthropic agentic coding trends data supports the investment: teams that invest in context infrastructure (instruction files, persistent memory, structured task framing) consistently outperform teams that rely on session-by-session prompting for equivalent tasks, and the gap widens as task complexity increases.

**Cons:** Context engineering investment has a high upfront cost relative to its visible short-term payoff. A developer who writes a 500-line CLAUDE.md for the first time is investing an hour or more in infrastructure that produces no immediate output — and the return materializes across future sessions, which means the developer must trust the investment before they can measure it. Instruction files also drift: a CLAUDE.md that accurately reflects the project's conventions at the time it was written becomes a source of conflicting instructions as the project evolves and the file is not updated, and an agent working from a stale instruction file can produce output that is inconsistent with the current codebase in ways that are difficult to detect because the inconsistency is with an implicit convention rather than an explicit test. Memory context systems add their own failure mode: persistent state that accumulates across sessions can accumulate errors — a wrong architectural decision recorded in the memory, a past agent output marked as successful that was actually wrong — and those errors propagate into future sessions with the same authority as accurate records. The context engineering discipline also does not eliminate the need for good prompts: a session with perfect context and a vague task description still produces vague output, because context establishes the environment for execution but does not substitute for clear task framing.

**Suggestions:** CATALOG.md's Memory & Context category is the direct tooling support for context engineering — memory systems, persistent context frameworks, and instruction-file tooling are all inventoried there. WORKFLOW.md's Plan stage is where the task context is authored: the issue-style task description (background, goal, acceptance criteria, constraints, test commands) is a Plan-stage artifact, and the investment in writing it precisely is the context engineering practice that most directly reduces clarifying questions during implementation. The `skills/setup-workflow/` bootstrap skill in this repo produces the minimum viable context engineering infrastructure for a new project: a CLAUDE.md that establishes conventions, a test hook, and a review discipline that collectively give the agent the baseline context it needs for the most common task types. For teams starting: the highest-leverage single context engineering investment is to write an explicit architectural intent section in the project's instruction file — two to four paragraphs describing what the project is, what it is not, and the key design decisions that should not be revisited without deliberate review — because this is the context that agents are least likely to infer correctly from code alone and most likely to violate if it is absent.

**Role-Specialized Agent Teams: Planner / Worker / Judge with Git Worktree Isolation**

**What it is:** By mid-2026, the practitioner consensus on multi-agent coding architecture has converged on a three-role taxonomy — Planner, Worker, Judge — as the structural template that scales to team-level task complexity while keeping each agent's context bounded and its failure modes localized. The Planner agent operates continuously at the project level: it reads the codebase, decomposes the task backlog into executable sub-tasks, and assigns those sub-tasks to available Worker agents. The Planner does not write code; its output is a sequenced, scoped task list with explicit acceptance criteria per sub-task. Worker agents execute individual sub-tasks in isolation: each Worker receives a bounded task with a defined exit condition, produces output against that task, and reports completion or failure to the Planner. Workers do not coordinate with each other; the Planner manages sequencing and dependency resolution. The Judge agent evaluates Worker outputs at the end of each execution cycle: it applies acceptance criteria from the spec or Planner's task description, determines whether each output passes or fails, and — crucially — decides whether to continue the current cycle (send the task back to a Worker for revision), accept the output and advance to the next task, or escalate to a human checkpoint. Git worktrees provide the isolation infrastructure that makes parallel Worker execution safe: each Worker operates in its own worktree (a separate working directory checked out from the same repository), so concurrent Workers cannot produce merge conflicts during execution, and failed Worker outputs can be discarded without affecting other Workers' progress. This three-role pattern is being operationalized by multi-agent tools including claude-squad, workmux, and Zenflow's orchestration layer, which expose Planner/Worker/Judge as first-class configuration primitives rather than requiring teams to implement the role separation manually.

**Pros:** Role specialization resolves the context accumulation problem that collapses naive multi-agent architectures at scale. A single orchestrator agent that decomposes, implements, and evaluates all sub-tasks accumulates context from every stage of every sub-task; at four or more sub-tasks with realistic output volumes, the orchestrator's context overflows and it begins hallucinating or ignoring earlier outputs. The Planner/Worker/Judge separation keeps each agent's context bounded: the Planner holds only the task graph, not implementation details; each Worker holds only its specific sub-task, not the global state; the Judge holds only the acceptance criteria and the output to evaluate, not the implementation history. Git worktree isolation makes the isolation boundary physical rather than logical: a Worker cannot accidentally modify files outside its task scope because it is operating in a separate working directory, and the Planner cannot observe partial Worker state because each Worker's output is only visible to the Planner at completion time. The Judge role specifically addresses the failure mode of agents declaring premature success: the Judge's explicit mandate is to evaluate against the spec, not to confirm that the Worker tried hard, which produces a structural separation between "the agent was confident" and "the output meets the acceptance criteria" — a distinction that naive self-assessment collapses.

**Cons:** The three-role architecture adds coordination overhead that is only justified for tasks complex enough to benefit from parallelism and role separation. A simple, sequential coding task — add a function, write a test, submit a PR — runs faster with a single agent than with a Planner that decomposes it, multiple Workers that execute in parallel, and a Judge that evaluates the result: the coordination overhead exceeds the parallelism benefit for small tasks, and the role boundaries add latency (task assignment, result collection, evaluation) at each stage. Git worktree setup and teardown is non-trivial: each worktree requires a checkout, dependency installation, and environment setup before the Worker can begin, and failed worktrees that are not cleanly torn down can accumulate disk usage and git state that requires manual cleanup. The Judge's evaluation is only as reliable as the acceptance criteria it is evaluating against: a Judge working from vague or incomplete acceptance criteria produces inconsistent verdicts — the same output passes one cycle and fails the next — which defeats the structural purpose of the role separation. The architecture also introduces a new class of coordination failure: a Planner that decomposes tasks incorrectly (assigns overlapping scope to two Workers, or sequences tasks in the wrong order) produces Worker outputs that conflict or build on incorrect prior state, and the Judge may not detect the root cause of the conflict because it evaluates each Worker's output in isolation.

WORKFLOW.md's Plan stage is where the Planner role belongs — the task decomposition and acceptance criteria authoring that the Planner agent performs is a Plan-stage activity, and teams that invest in the Plan stage produce the precise task graphs that the Worker/Judge pattern requires to function reliably. The Scopir and htdocs.dev multi-agent orchestration guides (linked in Sources) provide concrete implementation examples for the Planner/Worker/Judge pattern with git worktree isolation, including the specific coordination primitives and configuration patterns that production deployments have settled on. For teams starting: the minimum viable three-role adoption is to add a Judge step to an existing single-agent workflow — before human review, run a second agent pass with explicit instructions to evaluate the first agent's output against the task's acceptance criteria and flag failures — rather than attempting to implement the full Planner/Worker/Judge architecture from the start. The Judge-only intervention captures the most important structural benefit (separating confident assertion from objective verification) at a fraction of the coordination overhead of the full three-role pattern.
