**TDAD: Graph-Based Test Impact Analysis as Agentic Regression Defense**

**What it is:** Published in March 2026 on arXiv (paper 2603.17973), the Test-Driven Agentic Development (TDAD) methodology and companion open-source tooling address a specific failure mode of AI coding agents that conventional TDD does not: silent regressions. An AI coding agent that resolves a task correctly — passes the target tests — while breaking tests that previously passed produces a net-negative outcome; the patch looks done but is not safe to merge. TDAD is designed to expose this class of failure before the agent submits. The mechanism is a static AST-based code-test graph: TDAD indexes the project source tree by parsing abstract syntax trees, constructs a weighted dependency graph mapping each source file to the tests that exercise code paths through it, and produces a test impact map (`test_map.txt`) that identifies, for any proposed change, which tests are most likely to regress. When the agent is about to edit a file, TDAD's GraphRAG workflow surfaces the highest-risk tests from the impact map and adds them to the agent's context and execution plan before it writes the change. The agent runs that targeted subset of tests at each implementation step rather than the entire suite, and uses regression signals as an additional Observe channel in its ReAct loop. The empirical results on SWE-bench Verified are specific: TDAD reduced test-level regressions from 6.08% to 1.82% (a 70% reduction) while simultaneously improving task resolution from 24% to 32%, demonstrating that regression defense and resolution quality are complementary rather than in tension.

**Pros:** TDAD's 70% regression reduction addresses the most damaging failure mode at the intersection of AI coding and production safety: an agent introducing regressions at 6% of the test level produces output that looks correct to the agent but breaks the codebase for the next human reviewer, which erodes team trust in agentic outputs far faster than individual resolution failures do. The graph-based impact analysis is computationally tractable: the impact map pre-computes which tests are relevant to each file, so the agent runs a targeted high-risk subset rather than the full suite at each implementation step, keeping the regression defense affordable. A TypeScript port (fmguerreiro/tdad-ts on GitHub) makes the methodology accessible without the full research tooling setup, with the impact map exportable as a static file that can be version-controlled alongside the codebase. The resolution improvement (24% → 32%) is an equally important finding: including high-risk tests in the agent's context before it makes changes is not merely defensive — it gives the agent information about what the change must not break, improving implementation quality by making the problem better-specified (implement this change without breaking these specific tests rather than implement this change).

**Cons:** TDAD's impact map quality is constrained by the existing test suite: the graph maps source files to tests that exercise them, so a file with no coverage produces no entries and receives no regression defense — precisely the legacy areas where agents are most likely to introduce regressions against implicit behavioral contracts. The AST-based graph also does not model dynamic dispatch, reflection, or runtime-evaluated code paths: a change to a function called only through an interface or a dependency-injection container may not appear in any test's static call path to the changed file, causing the impact map to miss the relevant tests. The TypeScript port partially addresses this via TypeScript's static type information, but the limitation persists. Building the initial impact map adds a setup step for large codebases with complex dependency graphs, and the map must be refreshed after structural refactors that significantly change the call graph topology.

**Suggestions:** CATALOG.md's Code Review & Quality category includes tools in the test generation and coverage improvement cluster that are the preconditions for TDAD's full effectiveness: a project must have meaningful test coverage before the impact graph provides meaningful regression defense, and tools that generate tests for uncovered paths directly expand the graph's protective scope. WORKFLOW.md's Verify stage is the natural home for TDAD integration: the impact map fires at the Implement-to-Verify boundary, making targeted test execution part of the agent's implementation loop rather than a post-implementation gate. For teams starting: the TypeScript port (fmguerreiro/tdad-ts) provides the lowest-friction adoption path — generate the initial `test_map.txt`, commit it to the repository, and configure the agent to include the relevant test-map entries in its context before each file-level change. Including the relevant test identifiers in the agent's implementation prompt is sufficient to capture most of the resolution quality improvement the paper documents without the full GraphRAG tooling integration.

**Conductor-to-Orchestrator Transition: Three-Tier Fleet Architecture**

**What it is:** The most-cited structural shift in the July 2026 multi-agent coding discourse is described identically in multiple independent analyses: the practitioner role is changing from conductor — directly guiding a single agent in real time, like a musician playing an instrument — to orchestrator — directing an ensemble of agents asynchronously, like a conductor shaping an orchestra where each section plays independently between direction signals. The htdocs.dev guide and the amux.io orchestration reference both document the same three-tier architecture that has emerged as the dominant fleet management pattern for multi-agent coding in mid-2026. **Tier 1 (Interactive)** is the traditional single-agent loop: the developer directs one agent in a tight feedback cycle, reviewing output before the next step — appropriate for high-uncertainty tasks in poorly-understood codebases. **Tier 2 (Parallel Sprint)** runs 3–10 agents simultaneously on a known codebase with clear task boundaries, using orchestration tools like amux, Claude Squad, Parallel Code, or Conductor (Melty Labs) to manage concurrent execution and diff merging; the developer stays in the loop via dashboards and diff review. **Tier 3 (Overnight Fleet)** dispatches a fleet of agents against a backlog of well-specified issues with no interactive oversight, running while the developer is away; the developer reviews merged diffs at morning handoff. The shift is enabled by dedicated orchestration tooling that shipped in the first half of 2026: amux, workmux, Claude Squad for isolation, and native multi-agent features in the major coding tools — Cursor v3 with 8-agent parallelism, Google Antigravity 2.0 with Agent Teams, GitHub Copilot's multi-agent workspace. Each agent runs in a git worktree (isolated working copy), and the orchestrator manages merge sequencing.

**Pros:** The three-tier architecture provides explicit tradeoffs between speed and oversight at each tier, matching the developer's risk tolerance for a given task to the tier where the cost-benefit ratio is best. A well-understood bug fix in a well-tested module is a Tier 3 candidate — maximum throughput, minimum supervision, risk capped by test coverage; a new feature in a poorly-understood legacy module is a Tier 1 task regardless of agent trust, because the cost of oversight is low and the cost of an unchecked divergence is high. The Tier 2/3 throughput numbers are the primary driver of the sustained productivity claims in the 2026 literature: a developer running 8 parallel agents on clearly-scoped tasks is not 8× more productive due to coordination overhead and review bottleneck, but they are operating qualitatively differently — amortizing review time across 8 simultaneous implementations rather than 1, which changes the economics of agentic development at team scale. Worktree isolation provides the safety property that makes Tier 2/3 viable: a miscalculated agent cannot corrupt another agent's context or produce merge conflicts that destroy the working tree.

**Cons:** The three-tier fleet architecture requires a coordination infrastructure investment that does not pay for itself at small scale: orchestration tools add setup complexity, and the worktree isolation model has a disk consumption profile that is prohibitive for large repositories — 20 worktrees on a 10GB repository consumes 200GB of disk. The Tier 3 overnight fleet model has a failure mode the Tier 1 interactive model does not: an agent that diverges from its specification 30% through an 8-hour overnight run produces a patch that is neither correct nor trivially reversible, and the developer discovers this at morning review rather than 5 minutes into an interactive session. This is the stale-at-merge problem: the longer the fleet runs unattended, the larger the gap between the task specification (written hours earlier) and the codebase state (changed by parallel agents resolving other tasks simultaneously). Well-specified tasks with complete acceptance criteria at dispatch time are the only structural mitigation; teams that dispatch ambiguously-specified issues to overnight fleets accumulate rework debt that is invisible until morning review.

**Suggestions:** CATALOG.md's Agent Orchestration and Agent Harnesses categories inventory the tools at each tier — the evaluations distinguish between orchestration tools (amux, Claude Squad) that manage multi-agent coordination and individual agent harnesses (Claude Code, Cursor, Windsurf) that implement single-agent execution. WORKFLOW.md's Implement stage is the home for the tier-selection decision: choosing which tier matches a given task is the orchestration discipline that determines whether the fleet architecture produces throughput or rework. For teams starting with multi-agent fleets: begin with Tier 2 (3 agents, well-specified tasks, worktrees, same-session review) before advancing to Tier 3 overnight runs. The Tier 2 discipline of writing task specifications precise enough to dispatch to a parallel agent is identical to the specification quality required for Tier 3 — learning it at Tier 2 scale, where divergence is caught in hours rather than at morning review, is the correct progression. The AGENTS.md file in each repository is the persistent coordination artifact that carries task-boundary conventions, codebase architecture context, and merge-sequencing rules across all agents in the fleet — the document that makes the same specification precise enough for multiple parallel agents to interpret consistently.

**SDD Tooling Convergence: Every Major Platform Now Has a Spec-Driven Mode**

**What it is:** A structural market transition has occurred in the AI coding tool landscape between late 2025 and mid-2026: spec-driven development (SDD) — previously a practitioner methodology requiring custom workflows — has been absorbed by every major AI coding platform as a first-class product feature. The BCMS SDD guide documents the full landscape: GitHub Spec Kit, AWS Kiro, Claude Code's spec mode, Cursor, OpenSpec, BMAD, Tessl, and Google Antigravity have all shipped their own SDD implementation. SDD emerged in 2025 as a direct practitioner response to "vibe coding" — exploratory, context-free prompting that produced fast initial implementations and unreliable long-term codebases — and the 2026 platform convergence means the structural response to vibe coding is no longer a workflow practice developers must implement manually; it is a mode or a button in tools they are already using. The convergence also reveals what SDD means at the platform implementation level: each tool ships the same core workflow (write a specification documenting behavior, architecture, edge cases, and constraints before any code is generated; let the agent produce implementation and tests from the specification; validate the implementation against the specification) but with different emphasis on which stage the platform specializes in. AWS Kiro emphasizes specification authoring — the tool generates a draft spec from a task description for the developer to refine. Claude Code emphasizes specification-as-context — the spec file is loaded as the primary context for each implementation session. BMAD emphasizes specification-as-executable-contract — the spec is validated against the implementation as a formal test. This divergence creates a platform selection decision that did not exist 18 months ago: choosing an AI coding platform in mid-2026 is also choosing a SDD implementation style.

**Pros:** Platform-native SDD eliminates the primary adoption barrier that kept spec-driven workflows a practitioner minority practice in 2025: the requirement to design and maintain a custom workflow. A developer using AWS Kiro does not need to design their own specification format, prompt the agent to read the spec at each step, or build a validation harness — the platform handles all three. The market convergence also means SDD tooling is now actively developed and supported by each platform's engineering team rather than maintained by individual practitioners, improving reliability and capability over time. The survey data behind the convergence is concrete: teams with intentional workflow integration — curated AGENTS.md, specs written before code, sub-agents for wide reads, tight review loop — report time savings near 60%, versus 30% for teams using AI tools without integration discipline. The double-digit productivity differential between integrated and non-integrated AI coding is large enough that platforms not shipping SDD face a structural product disadvantage, which is why the convergence has happened so quickly.

**Cons:** Platform convergence has fragmented the SDD specification format across tool ecosystems: a spec written for AWS Kiro's workflow is not directly executable in Claude Code's spec mode, which is not directly validatable by BMAD's contract-checking approach. Teams using more than one AI coding platform — a common enterprise pattern where different teams or task types use different tools — must maintain multiple specification formats or accept that specifications authored in one platform's format are degraded when the team switches tools. The convergence has also produced a product marketing dynamic where "spec-driven" describes a wide range of actual implementation quality: a platform that generates a draft specification from a one-sentence task description and calls the result SDD is not providing the same quality of specification anchor as a platform requiring the developer to fully author behavioral contracts, edge cases, and architectural constraints before code generation begins. Teams adopting platform-native SDD on the strength of the 60% productivity claim without understanding which part of the SDD workflow their chosen platform actually implements may not realize the claimed gains.

**Suggestions:** CATALOG.md's Dev Workflow and Agent Harnesses categories include evaluations of the major SDD platforms — the evaluations document which platforms implement specification authoring assistance, which implement specification-as-context, and which implement specification-as-validation-gate, which is the analytical framework for mapping platform choice to actual workflow contribution. WORKFLOW.md's Plan stage is the SDD insertion point: the specification is the Plan stage's primary output artifact, and adopting platform-native SDD is effectively a formalization of Plan stage tooling. For teams evaluating SDD platforms: the three-question diagnostic is (1) does the platform help write a better specification, or does it only consume a specification I write myself? (2) does the specification travel with the implementation across the full session, or only at session start? (3) does the platform validate the implementation against the specification, or only use the specification as prompting context? These three questions locate each platform in the SDD implementation space. Teams that need specification authoring help — because they lack a specification writing practice — should prioritize platforms with spec generation assistance such as AWS Kiro; teams that need specification-as-validation — because they have a quality gate problem rather than an authoring problem — should prioritize BMAD or equivalent contract-checking platforms.

**CI as the Organizational Trust Contract for Agent-Generated Code**

**What it is:** The 2026 AI coding practitioner discourse has converged on a structural distinction between two levels of verification for agent-generated code: the agent's self-correction loop (the first layer, internal to the agent) and the CI pipeline (the second layer, the organizational trust contract that makes agent-generated code indistinguishable from human-authored code in terms of what the team will merge). The Kilo AI agentic workflow guide documents this two-layer model explicitly: the TDD AI agent loop handles local self-correction — the agent writes tests, runs them, fixes the code, repeats — but organizational trust requires a second layer in which the same CI pipeline that validates human-authored code (linters, unit tests, integration tests, security scans, style checks) validates agent-generated code. The critical operational insight is that the CI pipeline is not an additional quality gate layered on top of agentic output — it is the termination criterion for the agent's self-correction loop. An agent's output is not done when the agent believes it is done; it is done when the CI pipeline passes. This reframing has a structural implication for workflow configuration: CI must be accessible to the agent's feedback loop, not only to the post-submission review process. Agents that can read CI output and re-enter their self-correction loop in response to CI failures produce qualitatively better code than agents that receive CI feedback only through a human who reads the CI report and re-prompts. The complementary research finding — that TDD AI agents only work reliably in codebases that already have functioning test infrastructure — closes the loop: an agent with no CI signal to optimize toward has no convergence criterion for its self-correction loop and cannot distinguish a correct implementation from an incorrect one by observable means alone.

**Pros:** The CI-as-trust-contract model provides a clear organizational decision rule for when agent-generated code is ready for human review: not when the agent says it is done, not when a reviewer glances at the diff, but when the CI pipeline passes the same checks it would pass for human-authored code. This rule eliminates the implicit trust-calibration problem that plagues early agentic adoptions — "how much do I need to review this?" — by anchoring the review decision on an objective, verifiable signal rather than a subjective assessment of the agent's reliability on this class of task. For teams with mature CI pipelines, the CI-as-trust-contract model also provides a productivity return on existing CI investment: infrastructure built to enforce code quality standards for human-authored code now also functions as the quality enforcement layer for agentic contributions with no additional configuration. The model additionally creates a specific incentive for improving CI quality: an agent that can run CI locally and optimize toward passing its checks will expose gaps in CI coverage — tests that should exist but don't, linting rules that should catch a class of error but don't — in a way a human developer operating on individual files typically would not.

**Cons:** The CI-as-trust-contract model requires CI to be fast enough to function as a tight feedback loop for the agent's self-correction cycle. A CI pipeline that takes 30 minutes to run is not usable as a per-step agent feedback signal; it can only function as a post-submission gate. Teams with slow CI pipelines face a choice: invest in CI speed (parallel test execution, test selection, faster test infrastructure) before deploying agents in a CI-gated loop, or accept that the CI gate is a post-submission check rather than an in-loop self-correction signal. The model also assumes the CI pipeline is a comprehensive validator of code quality — but a pipeline that passes all checks for code embedding a subtle architectural violation, an implicit security assumption, or a behavioral regression in untested paths provides false confidence. Organizational trust contracts are only as strong as what the CI checks, and gaps that a human reviewer would catch through contextual understanding are not caught by the CI gate regardless of whether the author is a human or an agent. The CI-as-trust-contract model can inadvertently reduce the depth of human review if teams treat CI passage as a sufficient condition for merge rather than a necessary one.

**Suggestions:** CATALOG.md's Code Review & Quality category includes evaluations of CI integration tools and test generation tools that expand the scope of what the CI gate can verify — improving CI completeness is the highest-leverage investment a team can make to improve the quality of the trust contract model. WORKFLOW.md's Verify and Ship stages are the CI gate's canonical locations: the Verify stage runs CI checks as a pass/fail acceptance criterion before the implementation is submitted for review, and the Ship stage gates merge on CI passage. For teams starting: the minimum viable CI-as-trust-contract configuration is three checks — a linter (catches stylistic violations and common anti-patterns), a unit test suite (catches behavioral regressions in covered paths), and a dependency security scan (catches introduced vulnerabilities at the package level). These three checks can be run by the agent locally in under 5 minutes for most codebases, provide a termination criterion for the self-correction loop, and produce a CI passage record that is meaningful evidence for a human reviewer deciding how deeply to review the diff. Teams that invest in this three-check baseline before deploying agents see the quality improvement the CI-as-trust-contract model promises; teams that deploy agents against a codebase with no CI infrastructure get the agent's self-reported completion with no independent organizational verification.
