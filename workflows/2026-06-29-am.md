**Prompt Chaining as the Disciplined Default: Sequential Fixation Before Orchestration Escalation**

**What it is:** The dominant practitioner pattern for over-engineering agentic systems in 2026 is reaching for full multi-agent orchestration frameworks — LangGraph, CrewAI, the Microsoft Agent Framework — before establishing whether a simpler, deterministic prompt chain would produce the same result with less coordination overhead. Prompt chaining, the simplest agentic design pattern, breaks a complex task into a fixed sequence of steps where each step's output becomes the next step's input, with the agent's context bounded to one step at a time and the full sequence controlled by the orchestrating caller rather than by inter-agent negotiation. The practitioner consensus that has stabilized in 2026 is that this pattern should be the default starting point for any agentic workflow design — not because it scales to every use case, but because it fails in ways that are diagnosable and recoverable: when a chain step produces bad output, the failure is localized to that step and the fix is targeted at that step's prompt rather than distributed across a complex inter-agent communication topology. Orchestration frameworks become appropriate when tasks genuinely require dynamic routing (the next step cannot be determined until a prior step's output is evaluated), parallel execution across truly independent work streams, or agent-to-agent state that persists across a conversation boundary — conditions that apply to a smaller fraction of typical development tasks than teams tend to assume before encountering the debugging cost of unnecessary orchestration complexity.

The escalation decision is the critical judgment call in workflow design. Practitioners who have moved through the cycle of under-engineering (single prompts for complex tasks), over-engineering (orchestration frameworks for tasks a prompt chain would handle), and calibration (matching the pattern to the task's actual coordination requirements) consistently report that the over-engineering phase is more costly than the under-engineering phase: over-engineered systems accumulate coordination failures that are harder to diagnose than the straightforward output-quality problems of under-engineered single prompts. The heuristic that has stabilized in the 2026 practitioner corpus is to start with prompt chaining, identify the specific step where the chain produces output that requires runtime judgment to route, and escalate to a router or orchestrator only at that step — rather than designing the full system as an orchestration problem from the start.

**Pros:** Prompt chains fail locally — when a step produces bad output the failure is visible at that step and the fix is bounded to that step's prompt, unlike orchestration failures where bad state propagates across agent boundaries before surfacing as a downstream error with no clear origin. Deterministic chains are reproducible by definition: the same input to the same chain produces the same sequence of steps, which makes debugging, testing, and explaining the system's behavior tractable. The coordination overhead is zero: no inter-agent message passing, no shared memory to synchronize, no routing decisions to instrument. For the large fraction of development tasks that are sequential by nature — describe, plan, implement, test, review — a prompt chain is architecturally correct as well as simple.

**Cons:** Prompt chaining scales only to tasks where the sequence of steps is known in advance — tasks where the appropriate next action depends on the content of a previous step's output cannot be handled by a fixed chain without the orchestrator anticipating every branching condition at design time. Chains provide no parallelism: each step waits for the prior step to complete, so a chain processing ten independent items in sequence takes ten times as long as an orchestrator processing them in parallel. The absence of inter-step memory means each step sees only the output of the immediately preceding step, not the full history of the chain's execution, which limits the pattern's applicability to tasks that require accumulated context across steps.

**Suggestions:** CATALOG.md's Agent Orchestration and Agent Harnesses categories distinguish frameworks that implement prompt chains natively from those that add orchestration routing, parallel execution, and inter-agent memory on top; the simpler harnesses in those categories are often the right starting point for teams still calibrating their workflow complexity before selecting a framework that adds more coordination surface area than they currently need. WORKFLOW.md's Plan stage is where the escalation decision belongs: whether a task's structure requires dynamic routing, parallelism, or cross-step state — and therefore whether a prompt chain or an orchestration framework is the correct architecture — is a pre-delegation question, not a post-hoc adjustment made after hitting the simpler approach's limits. For teams starting: the diagnostic that determines whether to escalate from a chain to a router is one question — "at any step in this sequence, does the next step depend on a decision that cannot be encoded as a condition in the prior step's output?" — if no, a chain is sufficient; if yes, escalate only at that specific step.

**Test-Integrated Delegation: Weaving Verification Into the Delegation Loop Itself**

**What it is:** The pattern of delegating implementation to an agent and then running tests to check the result has a structural limitation that becomes apparent at longer session durations: the agent does not receive a correctness signal until the tests run at the end, which means errors accumulate across the full implementation before any verification feedback arrives. The test-integrated delegation pattern restructures this by making tests a first-class participant in the delegation loop rather than a final-stage check: the planning stage generates a list of test cases alongside the implementation plan — so the agent knows what it must pass before writing the first line — and each implementation step runs the relevant tests immediately after completion, so failures surface as local errors rather than accumulating into a large undifferentiated failure set against a completed implementation. Research on agentic test-driven development (TDFlow and related 2025–2026 work) finds that this pattern also reduces the most consequential failure mode of extended agentic sessions: agents cannot accumulate a large number of test failures and then attempt to satisfy them all at once, which is the condition under which agents are most likely to modify test criteria rather than fix the implementation.

Step-level verification also constrains scope creep — an agent receiving a test-failure signal at step N has a clear, bounded remediation task (fix step N to pass its tests) rather than an open-ended remediation task that may affect components beyond the step's scope. The planning stage's requirement to produce testable criteria for each step enforces the same behavior-focused, independently-testable spec quality standard that has converged across the 2026 practitioner corpus at the session level, making test-integrated delegation and spec quality mutually reinforcing rather than alternatives: good specs produce testable per-step criteria, and the discipline of writing per-step testable criteria surfaces underspecified requirements the planning stage would otherwise leave implicit.

**Pros:** Test integration at the step level converts the agent's self-reported completion into an empirically verified completion at each step rather than only at the session's end. Failures are localized: a test failure at step three surfaces a problem in step three's output, not in the accumulated output of all preceding steps, making diagnosis and remediation targeted rather than open-ended. The planning stage's requirement to generate test cases alongside the implementation plan also forces a testability check on each planned step before implementation begins — and the exercise of specifying testable criteria is often where teams discover that a task they thought was well-defined contains ambiguities they had not resolved.

**Cons:** Test-integrated delegation requires a test suite fast enough to run after every implementation step — a suite that takes twenty minutes to complete is incompatible with step-level verification, which means teams with slow test infrastructure face a prerequisite investment before the pattern is viable. Writing tests for each planning step increases the planning stage's duration relative to planning without test case generation; under delivery pressure, teams consistently trade away front-loaded investments, and test generation at the planning stage is reliably one of the first things cut. The pattern also assumes the planning stage can anticipate what tests are needed: for tasks where implementation reveals edge cases not foreseeable at planning time, a fixed test list may give false assurance that coverage is complete when it is actually bounded by what was foreseeable before implementation began.

**Suggestions:** CATALOG.md's Code Review & Quality and Agent Harnesses categories cover tools that implement test-woven execution loops; harnesses that support per-step hooks or callbacks — allowing the test suite to be invoked after each agent action rather than only at session end — are the key selection criterion for this pattern. WORKFLOW.md's Verify stage is the structural home for step-level test integration: defining what "done" means for a step (its test cases pass) maps directly onto the session-level Verify-stage definition and makes the two levels consistent rather than requiring separate quality criteria at each scope. For teams starting: the minimum viable implementation requires three configuration changes — adding a test-case generation step to the planning stage, configuring the agent to run the test suite after every implementation step rather than at session end, and configuring the agent to halt and report on test failure rather than continuing implementation — none of which require new tooling, only behavioral configuration of the harness already in use.

**Three-Tier Agent Tooling: Matching Agent Mode to Task Duration and Reversibility**

**What it is:** The proliferation of agent tools in 2026 has surfaced a practical operational problem: teams with access to interactive coding assistants, parallel worktree orchestrators, and overnight batch runners are frequently using the wrong tool for the task type — applying high-overhead orchestration to tasks that interactive use would complete faster, or running sessions interactively that would be better executed in parallel overnight. The three-tier architecture that has stabilized in the 2026 practitioner corpus matches the agent deployment mode to the task's characteristics: Tier 1 (interactive mode, same-session feedback) for tasks requiring exploration, ambiguity resolution, or real-time human direction; Tier 2 (parallel sprint mode, multiple isolated concurrent sessions) for well-specified tasks that can be parallelized across independent work items; and Tier 3 (overnight batch mode, agents queued to run without human attendance) for high-volume, fully-specified, low-risk tasks such as test generation, documentation updates, and type annotation backfills. At Stripe, AI agents now autonomously generate and merge over 1,000 pull requests per week — a throughput that requires Tier 3 deployment with monitoring infrastructure, not interactive use.

The operational insight behind the three-tier framework is that each tier has a different primary bottleneck. Tier 1 is bottlenecked by human attention: the human must be present to review and redirect, so Tier 1 throughput is bounded by human availability. Tier 2 is bottlenecked by task specification quality: parallel sessions cannot resolve ambiguity by asking the human, so underspecified tasks produce divergent output across sessions that requires costly reconciliation. Tier 3 is bottlenecked by monitoring infrastructure: overnight runs surface failures only through alerts, so the monitoring system must catch failures that would otherwise wait until morning review. Mismatching the tier to the task compounds each bottleneck — a Tier 3 task run in Tier 1 mode consumes human attention that could be used for higher-value judgment work; a Tier 1 task run in Tier 3 mode produces output with no opportunity for human course-correction at the ambiguity points where the task's requirements were underspecified.

**Pros:** The three-tier framework provides a vocabulary for the "what should I run interactively versus overnight?" decision that practitioners had been making ad hoc based on intuition rather than explicit criteria. By making the tier decision explicit — against criteria of task specification completeness, output reversibility, and whether the task requires human judgment to proceed past ambiguity points — teams can improve their agent-hour-to-value ratio: fewer interactive sessions on tasks that would run correctly in batch, fewer overnight runs on tasks where the output will require significant human revision regardless. The framework also makes monitoring requirements explicit: Tier 3 tasks require monitoring infrastructure before deployment, and a team that has not built the monitoring is not yet positioned to run Tier 3 tasks safely — the dependency that is most often assumed away in practice.

**Cons:** Tier assignment requires judgment about specification completeness and output reversibility made before running the task — practitioners systematically overestimate specification completeness (the task seems clear until the agent hits the first ambiguity point) and underestimate the cost of a failed Tier 3 run (an overnight batch that diverges from the intended behavior at scale before monitoring catches it). The monitoring infrastructure required for Tier 3 is a non-trivial investment most teams have not built before they start running overnight batch tasks; the gap between "we have an agent that can run overnight" and "we have monitoring that catches overnight failures before they propagate" is the most common Tier 3 adoption failure mode. Teams that adopt the three-tier vocabulary without the corresponding infrastructure are effectively running Tier 1 tasks at Tier 3 throughput — agents that can fail at scale with no human present.

**Suggestions:** CATALOG.md's Agent Orchestration and Observability categories cover the tools that respectively implement parallel and batch execution (Tier 2 and Tier 3 infrastructure) and monitor running agents (the prerequisite for Tier 3 safety). The Observability tools in CATALOG.md are load-bearing for Tier 3 adoption — Tier 3 task volume scales faster than human review capacity, so monitoring is not optional overhead but the safety condition for the tier. WORKFLOW.md's Plan stage is where the tier assignment belongs: which tier is appropriate for this task, and whether the team currently has the infrastructure that tier requires, is a pre-delegation decision, not a runtime one. For teams starting: the minimum viable three-tier implementation is a binary classification applied consistently before each session — tasks where human judgment is required at any decision point (Tier 1) versus tasks where the specification is complete enough to run without human presence (Tiers 2/3) — rather than defaulting to interactive mode for everything and optimizing only after the throughput bottleneck becomes acute.

**The PR Throughput Crisis: Governance Before Agent Scale**

**What it is:** The productivity measurement problem that has become most visible in mid-2026 agentic coding deployments is the PR throughput mismatch: as agent-assisted development increases the rate at which code is produced and submitted for review, the review stage becomes the binding constraint on delivery velocity, and the backlog of AI-generated pull requests grows faster than human reviewer capacity. The empirical finding that has driven the most governance investment in 2026 is that AI-generated pull requests wait 4.6 times longer for review than human-generated pull requests in the absence of governance — not because reviewers are unwilling to review AI-generated code, but because the existing PR review process was not designed for AI-scale throughput and breaks under the volume. Teams running agents at scale without PR governance are not experiencing 4.6× slower delivery on AI-generated work; they are experiencing a compounding backlog that accumulates merge conflicts, stale branches, and eventual reversion costs that exceed the implementation savings the agents produced.

The governance patterns that have emerged in response fall into three categories. Triage: automated first-pass review that categorizes PRs by risk and complexity before human reviewers see them, routing low-risk, low-complexity PRs to expedited review and high-risk or high-complexity PRs to senior reviewer queues. Throughput management: PR size limits for AI-generated work, batching small changes into reviewable units, and dedicated review blocks in the team's schedule for agent-generated output rather than treating it as part of the same queue as human-generated work. Quality gates: CI checks and automated review tools that catch the most common categories of AI-generated code issues — test coverage gaps, unintended dependency introductions, style inconsistencies — before the PR reaches a human reviewer, reducing the per-PR human review burden to judgment decisions the automated tools cannot make. At the throughput levels Stripe is operating at (1,000+ AI-generated merges per week), all three governance categories are required; at smaller scales, teams typically start with quality gates because they reduce per-PR review burden without requiring process restructuring, and add triage and throughput management as volume increases.

**Pros:** PR governance resolves the mismatch between AI-scale production throughput and human-scale review throughput — the primary failure mode of agentic coding deployments that operate at scale without governance. Automated quality gates reduce the human reviewer's per-PR workload to judgment calls the automated tools cannot make (architectural decisions, product alignment, edge cases the test suite does not cover), increasing the fraction of reviewer time spent on high-value decisions rather than issues a tool could catch. The governance investment also produces a traceability artifact: a PR that has passed automated quality gates before human review arrives at the reviewer with a record of what was checked, reducing the cognitive overhead of establishing the PR's baseline quality.

**Cons:** Governance is overhead that teams under delivery pressure resist investing in before the bottleneck is acute — the PR review backlog must grow painful before the governance investment appears justified, and by then the merge conflicts and stale branches it has produced are additional remediation costs the governance investment must pay before it begins returning value. The quality gate category carries a specific risk: a gate that catches a high fraction of common issues creates the impression of thorough pre-screening, which may reduce human reviewers' scrutiny of PRs that have passed the gates — when in fact the gates cover only the categories they were designed to check and are silent on categories they were not. Governance also imposes a classification overhead on each PR at triage time that scales with throughput; at very high agent output volumes, the triage step itself becomes a bottleneck requiring its own automation investment.

**Suggestions:** CATALOG.md's Code Review & Quality and Dev Workflow categories cover the automated quality gates, triage, and PR management tooling that implement each governance category; the Code Review & Quality category specifically contains tools designed for the automated first-pass review role that is the most immediately deployable governance pattern. WORKFLOW.md's Review and Ship stages are the structural home for PR governance: the review stage's throughput constraint is a Review-stage design problem, and the governance patterns (triage, throughput management, quality gates) are Review-stage infrastructure rather than bolt-on process. For teams starting: the minimum viable PR governance investment for AI-generated output is configuring CI to run automated code review before a human reviewer is assigned — requiring only CI configuration changes and no new tooling purchases — and directly reduces per-PR human review burden for the most common categories of AI-generated code issues before any throughput crisis forces the investment.
