**Hub-and-Spoke Orchestration: Orchestrator + Isolated Subagents**

**What it is:** Five major AI vendors — Anthropic, OpenAI, AutoGen, Cognition, and LangChain — converged by mid-2026 on a single default multi-agent architecture: a single orchestrator agent that retains full task context and spawns ephemeral, isolated subagents for discrete units of work. Each subagent receives a narrowly scoped task, executes it in its own context window, and returns a compressed summary to the orchestrator rather than its full output. The orchestrator maintains the running plan and synthesizes summaries into a coherent result. This is distinct from the flat "many agents sharing a context" topology that dominated earlier multi-agent experiments — in hub-and-spoke, only the orchestrator ever sees the full picture; subagents are intentionally kept narrow and context-light. The pattern emerged from a practical observation: giving every agent in a pipeline access to the full task context bloats token usage without improving output quality, while isolated subagents returning compressed summaries preserve the information the orchestrator needs at a fraction of the cost.

**Pros:** Context isolation prevents the "telephone game" failure mode where a long chain of agents each adds noise to accumulated context — compressed summaries force each subagent to distill what actually matters before returning results. The orchestrator's monopoly on full context makes task state auditable from a single vantage point rather than distributed across multiple agents' context windows. Ephemeral subagents are cheap to spawn and cheap to retry: if a subagent produces an incorrect result, the orchestrator can reassign the task without invalidating other concurrent work. The architecture also separates concerns cleanly — the orchestrator is responsible for planning, routing, and synthesis; subagents are responsible only for execution within their assigned scope — making each layer independently improvable.

**Cons:** The compression step between subagent and orchestrator introduces a lossy transformation: what a subagent chooses to include in its summary reflects its model of what the orchestrator needs, and a poorly calibrated subagent can omit context that turns out to be critical. Orchestrator failure is catastrophic in a way subagent failure is not — a failed subagent is retried, but an orchestrator that loses its context or produces a flawed plan corrupts the whole pipeline. Hub-and-spoke works well for parallelizable tasks but poorly for deeply sequential tasks where each step depends on full output from the previous one rather than a summary; forcing summary compression into a sequential dependency chain adds latency without the cost savings that make the architecture attractive in parallel scenarios. Teams also discover that writing good compression prompts — instructing subagents on what to include and exclude in their summaries — requires as much craft as the task prompts themselves.

**Suggestions:** The Agent Orchestration category in CATALOG.md (fleet, multi-claude) provides the scaffolding for orchestrator + subagent pipelines in Claude Code-based workflows. WORKFLOW.md's outer-loop Reflect stage is where orchestrator prompt design and compression calibration should be reviewed: a pipeline that consistently loses critical context in the compression step has a reflection artifact, not a task-execution artifact. The two-agent minimum principle applies here — validate the orchestrator + one subagent pattern before adding a second subagent, because each subagent addition multiplies compression surface area. The bounded collaboration pattern (three agents sharing a controlled peer mesh) from the NiteAgent 2026 report is the alternative to hub-and-spoke for tasks where agents genuinely need to negotiate with each other rather than report upward to a coordinator.

**The AI-First SDLC Redesign vs. Tool Retrofit**

**What it is:** Organizations adopting AI coding agents fall into two distinct structural categories: those that redesign their development workflow from the ground up around AI capabilities (AI-first SDLC), and those that insert AI tools into existing workflows as optional accelerators (tool retrofit). The distinction is not about how many AI tools are in use — a team can use five AI tools in a retrofit posture and one in an AI-first posture — it is about whether the workflow's structure assumes AI participation. An AI-first SDLC encodes AI participation at the process level: a guidelines file (like CLAUDE.md or AGENTS.md) is maintained as a first-class artifact alongside the codebase, specs are written before code generation rather than concurrently, test suites are designed for AI execution rather than human execution (runnable, deterministic, with machine-readable output), and review stages are structured around spec provenance rather than diff scrutiny. Teams operating in tool-retrofit mode report AI improving individual-developer throughput inconsistently; teams operating in AI-first mode report consistent gains that compound with each improvement to the process, because the structure itself creates a feedback loop.

**Pros:** AI-first SDLC makes AI participation legible to the whole team rather than invisible and developer-dependent: when the guidelines file, spec format, and test conventions are shared infrastructure, every developer's AI sessions operate on the same foundation, and improvements to that foundation benefit every session. The feedback loop is explicit: when an agent fails because the spec was ambiguous or the test suite was non-deterministic, the failure points to a process artifact that can be improved, rather than to an opaque session-level interaction that will recur unpredictably. AI-first teams also discover that structural investments compound — a better CLAUDE.md means better sessions, which surfaces more opportunities to improve the spec format, which means better generations, which means less review friction — whereas tool-retrofit teams find that each AI tool adds isolated value without reinforcing the others.

**Cons:** AI-first SDLC requires upfront structural investment that has no immediate payoff: writing a comprehensive guidelines file, redesigning tests for agent execution, and establishing a spec discipline add time before the first line of AI-generated code ships, making it difficult to justify to stakeholders measuring short-term throughput. The investment also creates an organizational dependency: AI-first teams that reduce human participation in the workflow find themselves more exposed when model capability regressions, API outages, or capability gaps force a step back to fully manual operation. Spec discipline in particular is difficult to sustain under deadline pressure — when a release is late, the first thing teams cut is pre-generation spec review, which is also the highest-leverage step; retrofit posture lets developers apply AI selectively under pressure, while AI-first posture tends to be all-or-nothing.

**Suggestions:** WORKFLOW.md's Plan stage is the entry point for SDLC redesign: AI-first adoption starts by encoding process conventions at the Plan→Implement handoff, not at the ship stage. The setup-workflow skill (skills/setup-workflow/) provides the bootstrap sequence for establishing AI-first infrastructure in a new repository — guidelines file, AGENTS.md conventions, and spec template — without requiring a complete process redesign upfront. A minimum viable AI-first SDLC has three artifacts: a guidelines file encoding conventions, a spec template for pre-generation planning, and a test runner the agent can invoke and interpret without human assistance; adding all three before adding more AI tools produces more consistent gains than adding tools one at a time in a retrofit posture.

**ReAct Execution Loops: Agents Running Multi-Step in Real Time**

**What it is:** The dominant execution model for production coding agents in 2026 is the ReAct loop — an alternating cycle of Reason (the agent produces a plan or analysis), Act (the agent executes a tool call or code operation), and Observe (the agent reads the result and updates its plan) — running continuously until the task is complete or a termination condition is reached. This replaces the single-prompt model in which a developer sends a request and waits for a response: in a ReAct loop, the agent is continuously executing, and the developer's role shifts from prompt-to-result evaluation to loop monitoring and intervention. The impact on session structure is measurable: average coding agent session duration grew from 4 minutes in 2024 to 23 minutes in 2026, with 78% of sessions now involving multi-file edits, as agents iterate through implementation, test execution, error diagnosis, and correction within a single loop rather than requiring developer re-prompting at each step. Specs serve as the loop's persistent memory — a detailed spec written before the loop starts gives the agent a stable reference to check its progress against at each Observe step without relying on accumulated context.

**Pros:** ReAct loops eliminate the round-trip latency that dominated earlier AI coding workflows: instead of a developer reviewing a draft, sending corrections, and waiting for a revised draft, the agent observes its own errors (test failures, lint violations, type errors) and corrects them within the loop. Multi-step debugging — reproduce, isolate, fix, verify — is natural in a ReAct loop where each step is an Act followed by an Observe; the agent accumulates state across steps in a way that single-prompt interactions cannot. Long-running loops also surface a class of errors that single-prompt interactions never reach: the agent that makes a correct first move but discovers three files later that the first move was incompatible with an existing interface — a ReAct loop catches and corrects this mid-run rather than delivering a half-correct result.

**Cons:** Long-running ReAct loops are high-variance: a loop that starts from a flawed reasoning step can execute many Actions before the error is observable, accumulating technical debt that is harder to reverse than a bad first draft in a single-prompt interaction. Monitoring a 23-minute loop requires sustained developer attention that many teams don't budget — developers who walk away during a long loop return to either a completed result or a deeply wrong one, with no intermediate state to course-correct. ReAct loops also consume tokens at a rate proportional to loop length — a loop that runs 15 Reason+Act+Observe cycles before completing a task uses substantially more tokens than a single-prompt interaction that produces the same output, and this cost is not always visible to developers who are tracking task completion rather than token consumption.

**Suggestions:** WORKFLOW.md's Implement stage is where ReAct loops live — the Plan stage produces the spec that anchors the loop, and the Verify stage begins when the loop terminates. The harness tools in CATALOG.md's Agent Harnesses category provide the infrastructure for instrumenting and terminating long-running loops: hooks on tool calls, step limits, and error-rate thresholds turn an unbounded loop into a bounded one. The two-fold initializer+coding-agent architecture from Anthropic's effective-harnesses guidance is the production-ready version of a long-running ReAct loop: the initializer sets up the environment and validates preconditions, and the coding agent runs the ReAct loop with a stable foundation beneath it rather than discovering setup failures mid-loop. Teams starting with ReAct agents should set an explicit step budget (e.g., 20 Act steps maximum) and require the agent to produce a progress summary at each Observe step — both limit runaway loops and create an intervention point for monitoring developers.

**Token-Aware Model Tiering: Cheap Workers, Smart Orchestrators**

**What it is:** Multi-agent coding systems consume tokens at 15× the rate of chat interactions, and token usage explains 80% of performance variance across organizations — meaning the teams with the lowest effective cost per useful output are not necessarily using the best models, but the best model-to-task assignment. Token-aware model tiering assigns the capable frontier model to orchestration tasks that require planning, synthesis, and judgment, while routing execution tasks — file reads, test runs, boilerplate generation, lint fixes — to smaller, cheaper, task-specific models. The practical implementation is a two-tier assignment: the orchestrator runs on a capable model (Claude Opus or equivalent) and spawns worker subagents on a cheaper model (Claude Haiku or equivalent) for well-defined, machine-verifiable subtasks. Organizations that have adopted this pattern report 40–60% cost reductions on multi-agent workflows without measurable quality regression on the tasks routed to cheaper models, because those tasks are fully specified (the orchestrator defines the scope precisely before handoff) and verifiable (the orchestrator checks the worker's output with a known-correct criterion rather than by judgment).

**Pros:** The cost savings are not speculative — 40–60% reductions on multi-agent workflows are documented across the organizations that have implemented model tiering in production, and the savings compound as agent usage scales: a team running 100 agent sessions per day at full frontier-model cost sees a proportionally larger saving than a team running 10. Model tiering also forces cleaner task decomposition: the discipline of deciding which tasks go to cheap workers and which stay with the smart orchestrator requires the pipeline designer to be explicit about where judgment is actually needed, surfacing tasks that were unnecessarily routed to expensive models in non-tiered systems. Cheaper worker models are also faster for simple tasks — a Haiku-equivalent model generating a test stub or reformatting a file is meaningfully faster than a frontier model doing the same work, reducing loop latency at the steps where latency is cheapest to cut.

**Cons:** Model tiering adds complexity to the orchestration layer: the router logic that decides which tasks go to which model tier is itself a source of bugs, and a misconfigured router that sends judgment-requiring tasks to cheap workers can produce results that look superficially correct but fail in production. Quality regression from cheap-worker tasks is subtle and often delayed: a Haiku-generated test stub that passes initial review but has wrong coverage boundaries won't surface as a problem until a bug slips through the gap — by which point the model tiering decision that caused the gap is invisible in the debug trace. Teams that implement tiering based on task type rather than verifiability (routing "all code generation" to cheap workers) frequently discover that code generation for ambiguous requirements is a judgment task, not an execution task, and the cost savings come at a quality cost that only appears in the review queue weeks later.

**Suggestions:** WORKFLOW.md's cost efficiency quality signal is the tracking category for model tiering impact — measuring cost per shipped feature before and after tiering implementation is the most direct way to validate the 40–60% reduction claim against your specific workload. The CATALOG.md Agent Harnesses category includes the tooling to implement model tiering at the harness level rather than the prompt level: routing logic encoded in harness configuration is more auditable and consistent than routing logic encoded in an orchestrator prompt. The minimum viable tiering decision is binary: tasks with machine-verifiable correct answers (test pass/fail, lint clean/dirty, type-check pass/fail) go to cheap workers; tasks requiring human-readable judgment (spec review, architecture decisions, cross-file consistency evaluation) stay with the capable orchestrator. This binary is easier to implement correctly than a fine-grained task taxonomy and captures most of the available savings.

**Complexity-Adjusted Velocity: The Honest Productivity Metric**

**What it is:** The most common mistake in measuring agentic development productivity in 2026 is tracking lines of code or PR count — metrics that AI agents can inflate arbitrarily by generating verbose implementations, splitting tasks into micro-PRs, or producing boilerplate at scale. Complexity-adjusted velocity measures how much meaningful work gets shipped per unit of time, where "meaningful" is defined by the cognitive complexity of the changes rather than their volume: a 50-line algorithmic fix to a performance bottleneck counts more than 500 lines of auto-generated CRUD boilerplate. Organizations that track complexity-adjusted velocity report 2.5–3.5× ROI on agentic development on average, with top-quartile performers reaching 4–6×, and the gap between median and top quartile is explained primarily by three compounding multipliers: agent capabilities (which models and tools are in use), orchestration quality (how well the workflow sequences those tools), and human experience (how effectively developers guide, review, and correct the agent). Teams measuring only lines of code systematically overestimate agentic ROI at launch (agents generate more lines than developers) and underestimate the quality cost that surfaces in maintenance (AI-generated code tends toward verbosity, which increases future complexity and slows future agents working in the same codebase).

**Pros:** Complexity-adjusted velocity exposes the actual productivity distribution: a developer using agents on well-understood, well-scoped tasks in a healthy codebase will show high complexity-adjusted velocity; a developer using agents on ambiguous tasks in a degraded codebase will show low velocity regardless of lines generated. This makes the metric actionable — low complexity-adjusted velocity points to either task clarity problems (fixable at the spec layer) or codebase health problems (fixable at the code health layer), whereas low lines-of-code metrics are unintelligible. The three-multiplier model (capabilities × orchestration × experience) also gives teams a framework for targeted investment: if velocity is below benchmark, the question is which multiplier is the binding constraint, not whether to add more agents.

**Cons:** Complexity is difficult to measure objectively — most complexity proxies (cyclomatic complexity, cognitive complexity scores from tools like CodeScene) are correlated with meaningful work but imperfect; a team that games cyclomatic complexity by splitting complex functions can inflate complexity-adjusted velocity just as easily as lines-of-code velocity. Establishing a pre-agentic velocity baseline that is comparable to a post-agentic baseline requires methodological care: the tasks are not identical, the developers' skills change, and the codebase changes in ways that independently affect complexity — controlling for all three is difficult outside a controlled study. Teams also find that the three-multiplier model is hard to use operationally because the multipliers are not independent: orchestration quality depends on human experience (the orchestration improves as developers learn what works), and agent capabilities depend on orchestration (the same model produces higher-quality output in a better-structured pipeline than a worse one).

**Suggestions:** WORKFLOW.md's Reflect stage is where velocity measurement belongs — complexity-adjusted velocity is a retrospective signal, not a real-time one, and the outer loop's Reflect cycle is the natural home for tracking it across sprints or release cycles. The Code Review & Quality category in CATALOG.md (CodeScene, SonarQube equivalents) provides the complexity measurement tooling that makes the "adjusted" in complexity-adjusted velocity computable rather than estimated. The evaluate-tool skill (skills/evaluate-tool/) follows a similar logic to complexity-adjusted velocity: rather than measuring whether an evaluation *exists*, it evaluates whether the evidence supports the verdict — a structurally identical distinction between volume (number of evals) and quality (evidence-backed verdicts). For teams starting the measurement exercise, the minimum viable complexity proxy is a code health score from an existing static analysis tool applied to changed files per PR, divided by story points — imperfect, but immediately actionable without a measurement infrastructure investment.
