**TDD-First Agentic Loop**

**What it is:** The 2026 practitioner community has converged on test-driven development as the primary mechanism for controlling agent output quality — not because TDD is new, but because the failure mode it eliminates is the specific failure mode that agentic code generation amplifies. The Fundesk TDD guide and the TDAD arXiv paper (2603.17973) both document the same structural insight: an AI coding agent that writes both the implementation and the tests for that implementation has no external oracle for its own correctness — it can pass all the tests it wrote while violating behavioral intent, because the tests reflect the same misunderstanding as the implementation. TDD inverts this by requiring the developer to write the failing test first, which forces the test to encode the developer's intent before the agent generates an implementation. The agent then writes code to pass the externally-authored test, which it cannot rewrite to match a wrong implementation. The TDAD paper operationalizes this into a concrete agentic workflow using graph-based impact analysis (GraphRAG): when an agent makes a change, the graph identifies which tests are transitively affected by that change and runs only those tests in the agent's self-correction loop, rather than the full suite. The targeted approach reduces the agent's self-correction overhead while improving regression coverage: TDAD measured a 70% reduction in test-level regressions (from 6.08% to 1.82%) and improved task resolution from 24% to 32% when deployed as an agent skill. Boris Cherny (Claude Code lead) has characterized the feedback loop dimension of this more broadly: "Probably the most important thing to get great results out of Claude Code — give Claude a way to verify its work. If Claude has that feedback loop, it will 2-3x the quality of the final result." Kent Beck, whose TDD book established the practice, has described the TDD + AI agent combination as a "superpower" for exactly the reason the TDAD paper demonstrates empirically: the developer owns the specification (the failing test), the agent owns the implementation, and the separation prevents the agent from validating its own bugs. The Forbes Tech Council piece documents the practitioner pattern: write the failing tests, hand off to the agent, run the tests, feed failures back to the agent, iterate until green — a loop that is faster than manual implementation and safer than prompt-and-generate because the exit criterion is mechanical rather than subjective.

**Pros:** TDD-first agentic development provides a mechanical exit criterion that eliminates the hardest part of agentic code review: determining whether the output is correct. A test suite written before the implementation exists defines what "correct" means in a form that neither the developer nor the agent can retroactively adjust — if the tests pass, the behavioral contract is met. The 70% regression reduction from the TDAD paper quantifies the quality improvement from adding graph-aware impact analysis to the loop, but the base quality improvement from TDD alone (developer-authored tests as the acceptance criterion rather than agent-authored tests or visual inspection) is the structural change. TDD-first also produces a more auditable development record: the git history shows the failing test first, then the implementation that makes it pass — a sequence that makes the intent-to-implementation relationship explicit in a way that prompt-and-generate cannot reproduce. Platforms that support TDD-first natively — where the agent can run tests, read failures, and fix code in a single execution loop (Claude Code, Codex, KiloCode, Cursor) — remove the developer from the iteration loop entirely for the test-pass phase, which is where the 2-3x quality improvement claim originates: the agent iterates on correctness autonomously against a fixed oracle, rather than requiring developer judgment to stop.

**Cons:** TDD-first agentic development requires the developer to write the failing tests before the agent writes any implementation, which front-loads the specification burden onto the developer at exactly the phase where the implementation details are most uncertain. For exploratory or prototyping work — where the correct interface or behavior is not known until the implementation is partially complete — TDD-first adds friction without adding proportionate quality benefit, because the tests written before any implementation exists may encode the wrong interface and require rewriting once the implementation reveals the correct abstraction. The TDAD paper's 70% regression reduction is measured on a specific task set with existing test infrastructure; teams whose codebases have low test coverage cannot apply TDD-first agentic development without first investing in the test infrastructure that the approach requires, which is a prerequisite investment that the Forbes Tech Council piece identifies as the most commonly skipped step. The self-correction loop also has a failure mode the approach does not address: an agent that cannot find an implementation that passes the tests will loop indefinitely (or until a token or time budget is exhausted) rather than signaling that the tests may be under-specified or that the task is outside the agent's capability envelope — a loop that burns token budget without converging.

**Suggestions:** CATALOG.md's Code Review & Quality category includes tools that integrate test execution into the agent loop: the evaluations distinguish tools that support mechanical test-gate self-correction from tools that require manual test invocation. WORKFLOW.md's Verify stage is the correct structural home for the TDD-first exit criterion — the failing test is a Plan-stage artifact, the implementation is an Implement-stage artifact, and the green test suite is the Verify-stage gate before code review. The `/verify` skill in this repo maps directly to this: its "drive the affected flow end-to-end" mandate corresponds to running the developer-authored test suite against the agent's implementation before treating the task as complete. For teams starting with TDD-first: the minimum viable investment is one failing test per agent task, written in the task's issue or spec file before the agent is invoked, referenced explicitly in the agent prompt as the acceptance criterion ("your output must make this test pass"). One test per task is enough to eliminate the most common agentic failure mode — an implementation that does something other than what was intended but passes visual inspection — and scales to a full TDD harness incrementally.

**Conductor-to-Orchestrator Transition**

**What it is:** The htdocs.dev multi-agent coding guide and the Beam AI production orchestration analysis both document a developmental transition that practitioners are navigating in 2026: the shift from the conductor model (a human developer manually directing one agent at a time, reviewing each output before issuing the next instruction) to the orchestrator model (a coordinator agent that decomposes tasks, delegates to worker agents, collects results, and manages the iteration loop, with the human developer providing oversight at the task level rather than the step level). The conductor model is the default starting point: a developer who uses a single AI agent interactively is acting as a conductor — they hold the context of the overall task, decide what to ask the agent next, review the output, and determine whether to accept or iterate. The conductor model scales linearly with developer attention: throughput is bounded by how fast the developer can review agent output and issue the next instruction, which is faster than writing the code manually but still bottlenecked on human review bandwidth. The orchestrator model replaces the human conductor with a coordinator agent that can hold the task decomposition in its own context and issue sub-task delegations to worker agents without waiting for human review at each step. The human developer's role shifts to task-level oversight: reviewing the coordinator's decomposition plan before execution begins, monitoring the orchestration in progress, and reviewing the final output rather than each intermediate step. The Beam AI analysis documents the production outcome: three focused agents consistently outperform one generalist agent working three times as long, with gains that are multiplicative rather than additive (parallelism, specialization, and isolation each contribute). The transition is accompanied by a measurable change in session structure: average coding agent session length has grown from four minutes to twenty-three minutes, and 78% of sessions now involve multi-file edits — a signal that the industry is moving from task fragments (individual files, isolated functions) toward end-to-end feature delegation (multi-file implementations with coordination across the affected surface).

**Pros:** The orchestrator model removes the developer from the inner iteration loop of each sub-task, which is the primary throughput bottleneck in the conductor model. A coordinator agent that can assign a sub-task to a worker agent, receive its output, and decide whether to accept or re-assign without human review at each step runs the inner loop at machine speed rather than human review speed. The specialization benefit is independent: a coordinator that assigns each sub-task to a worker agent with a focused context (one agent for implementation, one for test generation, one for documentation) produces higher quality per sub-task than a generalist agent context-switching across all three roles in the same session. The isolation benefit is also independent: worker agents operating on separate file scopes do not interfere with each other's context, eliminating the context pollution that occurs when a single agent holds the full multi-file implementation in one context window. The combination of these three gains (parallelism, specialization, isolation) is what produces the "multiplicative not additive" characterization — they compound rather than sum.

**Cons:** The conductor-to-orchestrator transition introduces coordination complexity that does not exist in the conductor model: the coordinator agent must decompose the task correctly (a decomposition error propagates to all worker agents), manage the context-passing between worker agents (an output format mismatch between coordinator and worker produces worker failures), and decide when worker output is sufficient to accept or requires re-assignment (a coordinator that accepts low-quality output passes it downstream). Forty percent of multi-agent pilots fail within six months of production deployment, and the Beam AI analysis identifies the primary failure mode as pattern mismatch — teams selecting the wrong orchestration topology for their task's work structure, or selecting the right topology without understanding how it fails under the actual task distribution. The orchestrator model also raises the bar for task specification: a coordinator agent that receives an under-specified task produces under-specified worker assignments, and the human developer's review gap (not reviewing each intermediate step) means the under-specification is not caught until the final output is delivered. The orchestrator model is not appropriate for tasks with genuine creative or architectural uncertainty: if the correct task decomposition is not knowable until the implementation is partially complete, a coordinator agent that commits to a decomposition upfront will block on the first worker assignment that cannot be completed without information the coordinator didn't have.

**Suggestions:** CATALOG.md's Agent Orchestration category inventories the frameworks that implement the coordinator/worker pattern — LangGraph, CrewAI, and AutoGen each support the coordinator pattern with different tradeoffs. WORKFLOW.md's outer loop Plan and Reflect stages are the correct checkpoints for the transition: the coordinator's decomposition plan is a Plan-stage artifact that should be reviewed before execution begins (the human oversight contribution that the orchestrator model preserves), and a Reflect-stage post-session audit that asks "did the coordinator's decomposition match the task's work structure?" produces the calibration that converts the transition from a one-time effort into an improving practice. For teams starting the transition: the minimum viable orchestrator model is a two-layer system — a human-authored decomposition plan (three to five numbered steps) plus one worker agent per step, run in sequence. This preserves the human conductor's decomposition judgment while delegating the implementation of each step to a worker agent, eliminating the per-step review bottleneck without fully automating the decomposition decision.

**Shared Coding Guidelines as AI Infrastructure**

**What it is:** The Stack Overflow blog post on building coding guidelines for AI agents documents a practice that the 2026 practitioner community is converging on as the team-level equivalent of the individual developer's instruction file: a shared, version-controlled document that specifies how code should be written in a given codebase, written to be read by both AI agents and human developers, and treated as a first-class engineering artifact rather than a wiki entry. The key insight the Stack Overflow analysis documents is that AI coding agents read coding guidelines differently from human developers: a human developer reads a guideline document once and internalizes its principles, applying them with judgment; an AI coding agent reads the guideline at the beginning of each session and applies it literally, without the judgment that allows a human to recognize when the guideline's spirit matters more than its letter. This difference has two implications: guidelines that are written for human developers (high-level principles, contextual examples, implicit constraints) are less effective for AI agents than guidelines written with explicit, literal instructions for common decisions, and guidelines that are kept in a machine-readable location (CLAUDE.md, AGENTS.md, or equivalent instruction files that agents load automatically) are more reliably applied than guidelines that the developer must manually reference. The emerging practice is to treat the shared coding guideline as a dual-audience artifact — specific enough that an AI agent applies it consistently, and structured enough that a human developer uses it as a reference — and to version it in the repository alongside the code it governs. The Kilo AI analysis documents the practitioner outcomes: teams with curated agent documentation and a habit of writing specs before code report consistently higher productivity gains from agentic tools than teams without them, with the delta attributed to the reduction in agent output that requires human correction because it violated an unstated convention.

**Pros:** Shared coding guidelines create a consistent agent behavior baseline across the entire development team: every developer's agent session starts from the same conventions, eliminating the per-developer prompt variation that causes the same codebase to receive inconsistent style, naming, and architectural treatment from different agents. The consistency benefit compounds as the team scales: a ten-person team without shared guidelines has ten agent configurations that may produce ten different conventions for the same decision; a ten-person team with shared guidelines has one agent configuration that produces consistent conventions across all sessions. Version-controlled guidelines also create an audit trail for convention evolution: when a guideline is changed, the git history records when the change was made and (if commit messages are informative) why — a record that helps future developers understand whether a convention they encounter in the code reflects an intentional architectural decision or a guideline that has since been superseded. The dual-audience nature of well-written guidelines also improves human onboarding: a new developer who reads the shared coding guideline understands not just what conventions the codebase uses but what decisions the team has made explicitly, which is more useful than reading the code itself and inferring conventions from examples.

**Cons:** Shared coding guidelines become stale at the rate of codebase evolution: a guideline that was correct when written may be incorrect for a new architectural layer, a new dependency, or a new file type introduced after the guideline was last updated. An AI agent that applies a stale guideline produces code that follows the old convention rather than the current one, which is a subtler failure mode than producing code that follows no convention — the output looks intentional but is wrong relative to the current codebase state. Guideline maintenance is also a team coordination problem: a guideline updated by one developer without team review may introduce a convention that other developers disagree with, but the other developers' agents will apply it consistently as soon as the file is updated, before the disagreement surfaces in code review. The dual-audience writing requirement — specific enough for agents, readable enough for humans — is also non-trivial: guidelines written for humans tend toward high-level principles that agents don't apply consistently; guidelines written for agents tend toward enumerated rules that are mechanical to apply but lose the judgment context that makes a principle useful in ambiguous cases. Teams that write agent-optimized guidelines often find that human developers stop reading them because they read like a configuration file rather than an architectural rationale document.

**Suggestions:** CATALOG.md's Memory & Context category inventories the tools that manage instruction file persistence — the evaluations there distinguish tools that load instruction files automatically at session start from tools that require manual reference, which is the distinction that determines whether a shared coding guideline is reliably applied. WORKFLOW.md's inner loop Plan stage is the correct checkpoint for guideline application: the agent should load the shared coding guideline as context at the start of each task, before any implementation begins, so that all subsequent generation is constrained by the guideline rather than having guideline compliance checked in post-generation review. The `skills/setup-workflow/` skill in this repo bootstraps the instruction file infrastructure (CLAUDE.md, AGENTS.md) that makes shared guidelines machine-readable for agents; for teams starting: draft the shared coding guideline as a CLAUDE.md section titled "Coding conventions" with three to five numbered rules covering the decisions that cause the most review friction today — those are the decisions that agents are currently getting wrong without explicit guidance.

**Vibe Coding as Exploration, Spec-Driven as Production**

**What it is:** The InterCode analysis and the Augment Code Spec+TDD guide both document an operational decision framework that the 2026 practitioner community is articulating as the successor to the earlier binary framing of "vibe coding vs. spec-driven development": the two approaches are not competing alternatives but sequential phases of the same development cycle, each appropriate to a different epistemic state of the developer's knowledge about what needs to be built. Vibe coding — prompt-and-generate iteration without a formal specification, where the developer refines the task description in real time based on the agent's output — is the correct mode when the correct implementation is not yet known: when the developer is prototyping a new feature, exploring a design space, or trying to understand what a solution would look like before committing to it. In this phase, the speed of vibe coding is a structural advantage: the developer can try three different implementations in the time it would take to write a formal specification for one, and the exploration produces the information (about the correct interface, the right abstraction, the feasible scope) that a specification requires. Spec-driven development is the correct mode when the correct implementation is known: when the developer has enough information from the exploration phase to write a six-element specification (outcomes, scope boundaries, constraints, prior decisions, task breakdown, verification criteria) that the agent can implement without resolving ambiguities by guessing. The operational failure mode the InterCode analysis documents is applying vibe coding to production work: an agent that generates code against an informal description can produce plausible, passing-tests code that violates behavioral intent, because the informal description did not specify the constraints the code must satisfy. The complementary failure mode is applying spec-driven development to exploratory work: a developer who insists on a complete specification before any implementation exists cannot use the implementation to discover what the specification should say. The 2026 maturity signal the analysis identifies is that teams that have built this two-phase discipline — vibe coding to explore, spec-driven development to produce — are the teams that report both the highest prototyping throughput and the lowest production rework rate, because they apply each mode to the epistemic state it is designed for.

**Pros:** The two-phase framework resolves the organizational debate between vibe coding advocates (who observe that spec-first slows exploration) and spec-driven advocates (who observe that vibe coding produces unreliable production code): both observations are correct within their respective epistemic states, and the framework assigns each mode to the state where it is correct rather than treating the modes as competing ideologies. The exploration phase also produces the most valuable spec-writing input: a developer who has seen three different agentic prototype implementations of a feature can write a specification that constrains the production implementation based on what the prototypes revealed, rather than speculating about constraints based on a feature description alone. The phased approach also produces a natural knowledge transfer artifact: the spec written at the transition from exploration to production captures the design decisions the prototyping phase resolved, which is information that the production implementation requires and that would otherwise be embedded in the developer's head rather than in a version-controlled document.

**Cons:** The two-phase framework requires the developer to recognize the transition point — the moment when the correct implementation is known enough to write a specification — which is a judgment call that is easy to get wrong in both directions. A developer who transitions too early (writes a spec before the exploration phase has produced enough information) writes a specification that encodes the wrong design decisions and constrains the production implementation to the wrong architecture. A developer who transitions too late (continues vibe coding into the production phase) accumulates agentic output without behavioral contracts, producing the AI-introduced bug debt that the framework is designed to prevent. The exploration phase's speed advantage also creates a scope creep pressure: a developer who can prototype in hours rather than days may explore more scope than the production phase can deliver, setting expectations based on prototype behavior that the spec-driven production implementation cannot match within the project's constraints. The two-phase framework also requires two distinct agent interaction patterns — the informal, iterative prompting of vibe coding and the structured, spec-anchored prompting of spec-driven development — which is a skill split that teams must train across all developers, not just the senior developers who typically author specifications in conventional development.

**Suggestions:** CATALOG.md's Code Review & Quality and Skills & Plugins categories both include tools at the vibe-to-spec transition point — the Code Review & Quality tools that provide behavioral intent verification (rather than only style or coverage checking) are the tools that operationalize the spec's verification criteria in the production phase. WORKFLOW.md's Plan stage is the natural transition point: the decision to move from exploration to specification is a Plan-stage gate, not an Implement-stage decision, and making it explicit in the team's inner loop prevents the common failure of drifting from vibe coding into production without the transition. The `/verify` skill in this repo operationalizes the production phase's exit criterion: its mandate to "drive the affected flow end-to-end" corresponds to executing the spec's verification criteria as the acceptance gate, which is the boundary between production implementation and deployment. For teams starting: establish an explicit transition trigger — a specific condition (the developer can write the verification criteria without referencing a prototype, the scope is defined enough to estimate in story points) — and use it as the gate between exploration and spec-driven phases, so the transition decision is mechanical rather than a judgment call that defaults to "continue vibe coding."

**Asynchronous Multi-File Fleet Coordination**

**What it is:** The Kilo AI analysis and the Beam AI orchestration patterns study both document a session-structure change that is visible in aggregate usage data for 2026: the average AI coding agent session has grown from approximately four minutes to twenty-three minutes, and 78% of sessions now involve multi-file edits. This is a structural signal that the dominant developer workflow has shifted from task-fragment delegation (isolated functions, individual files, scoped questions) to end-to-end feature delegation (multi-file implementations that span the feature's full surface area), and that the corresponding coordination practice has shifted from synchronous (developer reviews each output before the session continues) to asynchronous (developer delegates a multi-file task, the agent works through it autonomously, and the developer reviews the complete output at the end). The fleet coordination pattern extends this asynchrony to parallel agents: the most productive developers in 2026 are described as coordinating multiple agents running asynchronously and simultaneously, each with its own context window and file scope, each working on a separate aspect of the same feature or separate features in parallel. The Kilo analysis identifies the recommended production architecture for this pattern as a hybrid: orchestrate locally in the IDE (the developer initiates agent sessions, sets their scope, and monitors their progress from the IDE's agent panel), verify asynchronously through CI (each agent's output triggers a CI run that validates the implementation against the test suite and coding standards before the developer reviews it). This hybrid — local orchestration, async CI verification — provides the throughput of parallel delegation with the quality gate of automated verification, without requiring the developer to review each agent's output manually before the CI run confirms or refutes its correctness.

**Pros:** Asynchronous multi-file fleet coordination multiplies developer throughput by the number of agents running in parallel, bounded only by the number of independent work items that can be scoped without agent interference. A developer who can identify three non-overlapping feature scopes and delegate each to a parallel agent session processes three times the work in the time it would take to complete one session sequentially, with CI verification running on all three in parallel rather than sequentially. The multi-file session structure also produces more coherent implementations than task-fragment delegation: an agent that holds the full feature context across all affected files can make consistent naming, structuring, and dependency decisions, whereas an agent that handles individual files in isolation produces implementations that may be correct per-file but inconsistent at the feature level. The asynchronous review pattern also compresses the developer's active involvement to two points: task delegation (set the scope, hand off to the agent) and final review (after CI has confirmed or flagged the implementation) — which is a fundamentally more efficient use of developer time than the conductor model's per-step involvement.

**Cons:** Asynchronous multi-file delegation requires each task's scope to be defined completely before the agent starts, because the developer is not in the loop to clarify ambiguities during execution. An agent that encounters an ambiguity in a synchronous session can ask; an agent running asynchronously resolves the ambiguity by guessing, and the guess may not surface until the final review (or until CI catches the resulting error). Multi-file sessions also have a higher context management overhead per session: an agent holding twenty files in context is more prone to context rot (early-context information degrading as the context window fills) than an agent holding two files. The CI-gate hybrid also shifts the quality responsibility from developer review to test coverage: if the test suite is incomplete, a CI green result does not mean the implementation is correct — it means the implementation passes the tests that exist, which may not cover the behavioral intent the task specified. Teams that adopt async fleet coordination without investing in test coverage find that CI becomes a style-and-syntax gate rather than a behavioral correctness gate, which is a false confidence signal. The fleet coordination pattern also introduces a merge complexity that task-fragment delegation avoids: multiple agents editing the same repository in parallel can produce merge conflicts that the developer must resolve manually, adding a coordination cost that partially offsets the parallelism throughput gain.

**Suggestions:** CATALOG.md's Agent Harnesses category inventories the tools that support multi-session fleet coordination — Claude Code's parallel worktrees and the harness-level session management tools that allow the developer to monitor multiple concurrent agent sessions are the infrastructure primitives this pattern requires. WORKFLOW.md's outer loop Ship stage is the correct gate for fleet coordination output: the CI pipeline that runs on each agent's PR is a Ship-stage gate, and the developer review that follows CI confirmation is the final human accountability step that the asynchronous pattern preserves. For teams starting with fleet coordination: begin with two parallel sessions on non-overlapping feature scopes before scaling to a larger fleet. Two parallel sessions expose the scope definition discipline required (each scope must be complete and non-overlapping) and the CI quality dependency (CI must be a behavioral gate, not just a style gate) without the merge complexity of a large fleet, and the throughput gain is sufficient to validate the pattern before investing in the tooling required to scale it.
