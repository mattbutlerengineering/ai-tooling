**Test-Driven Agentic Development: Tests as Executable Specifications for Agent Delegation**

**What it is:** The most operationally significant workflow shift in the mid-2026 practitioner corpus is the migration of test authorship from the post-implementation phase — where tests verify what was built — to the pre-delegation phase, where tests define what the agent must build. Test-driven agentic development (TDAD) treats the test suite as the agent's specification: rather than describing desired behavior in natural language and hoping the agent infers edge cases correctly, the developer writes failing tests before delegating implementation, and the agent's task is to produce code that makes those tests pass. The critical observation driving this shift is that natural-language task descriptions are systematically underspecified relative to executable tests: a task description that says "add rate limiting to the API" leaves the agent to infer the rate (per request, per user, per IP?), the window (seconds, minutes, hours?), the behavior at the limit (429 with retry-after, silent drop, queue?), and the exception cases (health checks, internal callers, authenticated premium users?), whereas a test suite that exercises the rate-limiting behavior at the boundary — exactly at the limit, one over, after the window resets, for an exempted caller — makes each of those decisions explicit in executable form. The TDFlow research (arXiv 2510.23761) demonstrates that agentic workflows structured around test-first delegation produce measurably fewer implementation errors than prompt-only delegation because the test suite eliminates an entire class of specification ambiguity that causes models to make plausible but incorrect assumptions.

The workflow that has stabilized in the 2026 practitioner corpus has four steps: the developer writes tests for the desired behavior (or prompts the agent to write tests from a specification and reviews them before proceeding), those tests become the delegation artifact, the agent implements until the tests pass, and the developer reviews the implementation knowing that functional correctness is already verified. The review burden shifts from "does this do what I intended?" to "is this maintainable and does it handle cases the tests don't cover?" — a substantially narrower and more tractable question. The pattern pairs naturally with CI-gated agentic sessions: the agent cannot report completion without a green test run, making the handoff signal objective rather than the agent's own assessment of whether the task is done, which the 2026 practitioner corpus consistently rates as unreliable for edge cases and interface correctness.

**Pros:** Tests as delegation artifacts eliminate the most common class of agentic implementation defects: assumptions about behavior the task description left implicit. An agent implementing against a failing test suite cannot misunderstand the intended behavior at the tested boundaries — it must produce code that satisfies the assertion, making the test the authoritative statement of intent rather than the developer's memory of what the prompt said. The review process also becomes structurally cheaper: the developer reviewing an agent-generated implementation backed by a passing test suite can focus on implementation quality and untested edge cases rather than re-deriving whether the implementation is functionally correct. Pre-written tests also provide a regression net from the start: when later agent sessions make changes to the same code, the existing test suite catches regressions automatically, which closes the feedback loop that unverified agentic output leaves open.

**Cons:** Test-first delegation is most effective for units of behavior that are bounded and testable in isolation; it is substantially less effective for tasks where the correct behavior is genuinely uncertain at the outset and the development process is exploratory rather than specification-driven — discovery tasks where the developer is using the agent to explore possibility space rather than implement a known solution. Writing tests before implementation also requires more upfront planning investment per task than prompt-only delegation: the developer must think through the behavior precisely enough to write tests before writing any implementation, which is the right investment for production-path features but is overhead for exploratory, throw-away, or inherently ambiguous tasks. The pattern also creates a false assurance risk: a passing test suite verifies the behavior the tests cover, not the behavior the developer intended — gaps in the test suite can leave significant incorrect behavior undetected, and delegating confidence in test coverage to the agent introduces a failure mode where the agent writes tests that its own implementation passes rather than tests that exercise the intended behavior at its boundaries.

**Suggestions:** CATALOG.md's Code Review & Quality category and WORKFLOW.md's Verify stage are the structural homes for this pattern; tools in the quality category that support test-first workflows or CI-gated agentic sessions are architecturally matched to TDAD. WORKFLOW.md's Plan stage is where test authorship belongs in the dev-loop map: writing tests before implementation is a planning-stage activity, not a post-implementation one, and teams that treat test authorship as part of task decomposition rather than verification are applying the pattern correctly. For teams starting: the minimum viable implementation is a single constraint — the agent's completion condition is a passing test run, not its own assessment of completeness — which captures most of TDAD's value without requiring the full discipline of test-first authorship in every task. Claude Code's `/verify` skill and the harness-level completion-gating available in tools like Plandex and aider implement this constraint without custom scaffolding.

**Evaluator-Optimizer Loop: Systematic Quality Through Generate-Critique-Revise Cycles**

**What it is:** The evaluator-optimizer pattern — one agent generates a response, a second agent critiques it against explicit quality criteria, and the first agent revises based on that critique — has emerged as the primary quality mechanism for agentic coding workflows in mid-2026. The pattern addresses a structural limitation of single-agent generation: an agent optimizing for task completion is not simultaneously optimizing for correctness, maintainability, security, and edge-case coverage, because those objectives conflict with the least-resistance path to a working implementation. A separate Evaluator agent, prompted to find problems rather than complete tasks, applies a different optimization target to the same output — and the 2026 practitioner corpus consistently shows that an adversarially-framed Evaluator ("find reasons this implementation is wrong, insecure, or unmaintainable") surfaces defects the Implementor did not self-report, because the Evaluator's prompt is not constrained by the task completion objective that caused the Implementor to make the defect in the first place. vdf.ai's design patterns compendium identifies the evaluator-optimizer loop as one of the three foundational agentic patterns alongside prompt chaining and ReAct, characterizing it as the primary mechanism for self-correction in production agentic systems.

The loop is not limited to a single generate-critique-revise cycle. Teams deploying the pattern for high-stakes changes run multiple rounds — the Implementor revises based on the first round of critique, the Evaluator critiques again, and the cycle continues until the Evaluator finds no actionable findings or a maximum round count is reached. The key operational finding from 2026 deployments is that the Evaluator's prompt must be specific to be useful: an Evaluator prompted to review "general quality" produces generic observations with low actionability, whereas one prompted to check a specific enumerated list of criteria — boundary conditions at the interface, error path handling, security inputs (unsanitized user data, missing auth checks), and the maintainability heuristics the team's convention file specifies — produces findings the Implementor can act on. The pattern also decomposes naturally by dimension: a security-focused Evaluator, a maintainability-focused Evaluator, and a correctness-focused Evaluator running in parallel (each with its own specialized prompt) provide broader coverage than a single general-purpose reviewer at comparable cost.

**Pros:** The evaluator-optimizer loop is structurally additive to the Implementor's own self-checking: because the Evaluator's optimization target differs from the Implementor's, it surfaces a category of defects that self-review cannot find. The pattern also makes quality criteria explicit in executable form — the Evaluator's prompt is a machine-readable statement of what good output looks like, which documents the team's quality standard in a way that is immediately enforceable rather than requiring human reviewers to remember and consistently apply it. Multi-dimension parallel Evaluators extend coverage without increasing wall-clock time: a security Evaluator, correctness Evaluator, and maintainability Evaluator running in parallel produce the combined output of three specialized reviewers in one Implementor's cycle time.

**Cons:** The evaluator-optimizer loop adds infrastructure and cost that is not justified for every task: a trivial change (rename a variable, update a comment, add a logging statement) does not need a generate-critique-revise cycle, and applying the pattern uniformly produces overhead without proportionate quality gain. The pattern is also only as good as the Evaluator's prompt: a poorly-specified Evaluator finds generic issues that the Implementor's own self-review would have caught, making the loop ceremonial rather than quality-additive. The most common deployment failure is an Evaluator prompted to confirm correctness rather than find defects — "does this implementation look correct?" produces confirmation bias where the Evaluator agrees with the Implementor's choices, while "what is wrong with this implementation?" produces adversarial output that is more useful despite feeling less comfortable. Teams that run multiple revision rounds without a maximum-round backstop also risk infinite correction loops where the Evaluator and Implementor cycle indefinitely without converging.

**Suggestions:** CATALOG.md's Agent Orchestration category covers frameworks with native support for multi-agent generate-critique-revise loops; the selection criterion is whether the framework supports role-specialized agent definitions with distinct prompts rather than a single undifferentiated agent pool. WORKFLOW.md's Review stage maps directly to the Evaluator role — the evaluator-optimizer loop is a structural implementation of the Review stage using an agent as the reviewer rather than a human, and the criteria in the Evaluator's prompt should match the criteria the team's human reviewers apply. For teams starting: the minimum viable implementation is a two-step session — Implementor followed by a single Evaluator with an adversarial prompt and a specific enumerated checklist — run as a Claude Code workflow, with human review reserved for the Evaluator's findings rather than raw Implementor output.

**Least-Privilege Permission Architecture: Treating Agent Permissions Like Production Systems**

**What it is:** As agentic coding sessions have expanded from single-file completions to multi-file implementations that touch authentication paths, database schemas, and CI/CD configuration, the 2026 practitioner corpus has converged on treating agent permissions as a production security concern rather than a convenience setting. The principle that has emerged — permissions managed like production systems: least privilege, short-lived tokens, sandboxes, network restrictions, and audit logs — reflects the operational reality that an agent with broad write access and outbound network capability is a lateral-movement surface in the event of prompt injection, a misconfigured tool call, or a runaway loop that consumes resources and modifies state at a rate no human could replicate. The security case is straightforward: a coding agent that can write to any file, run arbitrary shell commands with full environment access, and make outbound network calls to arbitrary hosts has the same risk profile as a compromised developer workstation — and the mitigations are the same: scope the accessible surface to the minimum the task requires, revoke access when the session ends, and log every action the agent takes so post-hoc audit is possible.

The concrete practices that have stabilized in the 2026 corpus operate at three levels. At the session level: agents are configured with file-system scope limited to the repository tree, shell command allowlists that enumerate permitted commands rather than permitting everything, and outbound network restrictions that allow package registry and VCS access but block arbitrary HTTP. At the token level: credentials the agent needs (API keys, deployment tokens) are injected as short-lived session-scoped tokens rather than long-lived secrets, so a token exfiltrated during an agent session expires before it can be used in a subsequent attack. At the audit level: every tool call the agent makes — every file write, every shell command, every external request — is logged to an append-only audit trail that can be reviewed if the session produces unexpected output. Claude Code's permission tiers and `.claude/settings.json` allowlists implement the session-level controls; the token and audit levels require additional infrastructure but are increasingly provided by CI/CD platforms that expose agent session contexts as sandboxed environments.

**Pros:** Least-privilege permission architecture limits blast radius: an agent that can only write within the repository tree and run permitted commands cannot modify host configuration, exfiltrate credentials from the environment, or make outbound calls to attacker-controlled infrastructure even if the agent's reasoning is manipulated by prompt injection in a retrieved file. Audit logs make agent sessions post-hoc reviewable: when agent output is unexpectedly wrong, the tool-call log shows exactly what the agent read, what it wrote, and what commands it ran, which makes debugging substantially faster than reconstructing session state from file diffs alone. Permission scoping also forces explicit task decomposition: an agent session scoped to a single package in a monorepo cannot inadvertently fix an unrelated issue in a different package, which produces sessions with cleaner change surfaces and easier-to-review diffs.

**Cons:** Constructing precise permission scopes requires knowing in advance what the agent will need to access, which is non-trivial for exploratory or multi-stage tasks where the agent may need to read files outside the expected scope to understand a dependency or debug an unexpected failure. Over-restrictive permissions produce blocked sessions: an agent that cannot run the test command to verify its output, or cannot read a configuration file that governs the behavior it is implementing, will stall or produce incorrect output, and the developer must either widen the scope or intervene manually — which negates the productivity benefit of delegation. Audit log infrastructure also adds operational overhead that is significant for small teams: append-only logging, log retention, and log review are production operations practices that require tooling investment before they are useful.

**Suggestions:** CATALOG.md's Security & Safety category covers tools that implement session-level access controls, sandboxing, and audit logging for agentic sessions. WORKFLOW.md's outer-loop Ship and Reflect stages are where permission architecture belongs: a permission configuration that was appropriate for a development session may need to be narrowed for a CI/CD agentic session that runs automatically without a developer watching, and the Reflect stage is where those differences should be reviewed and standardized. For teams starting: the minimum viable implementation is the allowlist approach in Claude Code's `.claude/settings.json` — specifying permitted shell commands and restricting file access to the project directory — which captures most of the blast-radius reduction benefit without requiring custom sandbox infrastructure. The `fewer-permission-prompts` skill in this repo's `.claude/skills/` provides a starting point for calibrating allowlists based on actual session usage patterns.

**Sequential Prompt Chaining: Breaking Complex Tasks Into Ordered Stages with Explicit Handoffs**

**What it is:** Prompt chaining — breaking a complex task into smaller, ordered steps and passing the output of each step as the explicit input to the next — is the foundational compositional pattern for agentic development in 2026, and the one that most directly separates reliable delegation from unreliable single-prompt delegation. The operational insight is simple: a model producing a complete implementation in a single prompt must simultaneously plan, implement, verify, and document — competing objectives that are all in scope at once — and the result is output that satisfies the easiest objectives (syntax is valid, the happy path runs) while underperforming on the harder ones (edge cases, error handling, maintainability). Decomposing the same task into an explicit chain — Plan (produce a structured plan with subtasks and acceptance criteria), Implement (implement each subtask against the plan), Verify (check each subtask's output against acceptance criteria), Document (generate documentation from the verified implementation) — isolates each objective into a stage where it is the only optimization target, producing consistently higher output quality on each dimension than a single-stage prompt achieves on any of them. Virtido's 2026 agentic workflow patterns survey identifies prompt chaining as the most widely deployed workflow pattern across enterprise agentic deployments, preceding more complex orchestration patterns like routing and parallelization because it requires no multi-agent infrastructure and produces quality improvements immediately.

The handoff structure between stages is the operationally critical design decision: a chain where each stage receives the prior stage's raw output without a structured format constraint degrades as the chain lengthens — errors in early stages propagate unmodified, and by the time a downstream stage encounters them, the context required to understand the error source has scrolled off. The chains that produce the most reliable output in the 2026 corpus use structured intermediate formats — JSON schemas, typed markdown templates, or enumerated checklists — that force each stage to produce output in a form the next stage can parse without ambiguity. The Plan stage producing a JSON array of subtasks with explicit acceptance criteria and dependency ordering is the canonical structured intermediate format: the Implement stage can iterate over it mechanically, the Verify stage can check each subtask's output against its criteria, and the Document stage can generate structured documentation from the verified subtask tree.

**Pros:** Stage isolation means each stage's errors are bounded to that stage rather than propagating silently through the chain: a planning error surfaces at the Plan stage when it produces an incomplete or contradictory subtask list, not three stages later when the implementation based on that plan fails a test the plan didn't specify. Quality improvements are also additive across stages: output that has been explicitly planned, implemented, verified, and documented is more complete on each dimension than single-prompt output, because each dimension was the sole optimization target for a full stage rather than one of many competing objectives. Chaining also makes long-horizon tasks tractable for models with bounded context windows: by making each stage's scope small and bounded, the chain can tackle tasks that are too large to fit in a single prompt while maintaining coherence through structured intermediate formats.

**Cons:** Chain length is subject to error accumulation: a small misspecification in the Plan stage propagates to every downstream stage, and by the time it surfaces as a test failure in the Verify stage, the Implement stage has already produced a full implementation based on the incorrect plan. Correcting an early-stage error in a multi-stage chain requires re-running every downstream stage from the point of correction, which in a long chain can be more expensive than the original run. Structured intermediate formats also add design overhead: defining a JSON schema for the Plan stage's output, ensuring every downstream stage can parse it, and handling edge cases where the Plan stage produces output that doesn't conform to the schema requires upfront investment that simple chains skip and then pay for as debugging cost when the chain breaks at an unexpected format boundary. Human review at each stage checkpoint reduces error accumulation but adds latency that negates some of the throughput benefit of chaining.

**Suggestions:** CATALOG.md's Agent Harnesses category covers frameworks that implement prompt chaining natively with structured intermediate formats; the selection criterion is whether the framework supports typed stage outputs and explicit dependency ordering between stages rather than free-form output passing. WORKFLOW.md's inner-loop stages (Plan, Implement, Verify, Review, Ship) map directly to the stages of a well-structured chain — the dev-loop vocabulary is the natural stage decomposition for most development tasks, making the chain architecture and the workflow vocabulary consistent rather than requiring a separate abstraction. For teams starting: the minimum viable implementation is a two-stage chain — a Plan stage that produces a structured list of subtasks with acceptance criteria, followed by an Implement stage that works through that list sequentially — which captures most of chaining's quality benefit without the infrastructure of a full multi-stage orchestrated pipeline.
