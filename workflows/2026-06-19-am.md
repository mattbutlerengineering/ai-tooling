# AI Developer Workflows — 2026-06-19 (am)

Twice-daily web-research pass. Theme this run: **verification discipline is the new bottleneck.** Generation speed is largely solved; the workflows winning in 2026 are the ones that plan before generating, ship small diffs, and gate on tests/specs rather than vibes ("prompt and pray" is explicitly the anti-pattern being replaced).

## 1. Spec-Driven Development (SDD)

**What it is:** The spec is the durable artifact; code is the build output. Standard loop is **Specify → Plan → Tasks → Implement**, each with a human checkpoint. SDD specs increasingly *execute as validation gates* rather than just being read by humans (arXiv "From Code to Contract in the Age of AI", Feb 2026). Tooling: GitHub Spec Kit, AWS Kiro, OpenSpec, BMAD-METHOD, Cursor Plan Mode, Tessl.

**Pros:**
- Attacks the #1 failure mode (misalignment) before any code is written.
- Spec becomes reusable context — "refined context engineering," portable across agents and runs.
- ICSE 2026 peer-reviewed result: feeding architectural docs into generation measurably improves correctness, architectural conformance, and modularity.

**Cons:**
- Heavyweight for small changes; the four-phase ceremony can exceed the cost of the edit.
- Spec rot — if the spec isn't kept executable/checked, it drifts from the code like stale docs.
- Risks recreating waterfall if checkpoints become approval gates instead of fast iterations.

**Suggestions:** Adopt for net-new features and risky refactors, skip for one-liners. In this catalog the equivalent is mattpocock's `grill-with-docs` + `to-prd`/`to-issues` (align → spec → tickets), and `domain-modeling` for the shared-vocabulary layer SDD depends on. Keep specs in-repo and run them as checks, not as prose.

## 2. Plan → Execute → Verify → Adapt (bounded agentic loop)

**What it is:** The agent plans, executes in small diffs, verifies with tests, and adapts — bounded by explicit **stop rules** and documented rollback paths. Treats review bandwidth as a real, finite system constraint.

**Pros:**
- Small diffs keep human review tractable and rollbacks cheap.
- Stop rules prevent runaway agents from drifting off-task or burning tokens.
- "More per token" — better context management and stronger first passes beat raw retries.

**Cons:**
- Requires a rigorous test suite up front; weak tests make "verify" theater.
- Defining good stop rules is a skill; too tight stalls, too loose runs away.

**Suggestions:** Pair with TDD so the verify step has teeth (mattpocock `tdd`, superpowers `test-driven-development`). The catalog's `WORKFLOW.md` inner loop (Plan → Implement → Verify) already encodes this; the missing discipline is usually the *stop rule* — set a diff-size or test-pass ceiling per task.

## 3. PR-Review Bots as First-Pass Reviewer

**What it is:** Automated review on every PR (CodeRabbit, Qodo, Sourcery, PR-Agent) to catch patterns and obvious defects before a human looks — human review *ownership* stays required.

**Pros:**
- Cuts human reviewer load on mechanical issues; raises the floor on every PR.
- Consistent, never-tired first pass; good at surfacing style/security/pattern violations.

**Cons:**
- Noise and false positives erode trust if not tuned.
- Risk of reviewers rubber-stamping bot-approved PRs (automation complacency).

**Suggestions:** Use as a *filter*, not a *gate* — bot first pass, human owns the merge decision. Catalog support: `pr-agent`, plus the local `pr-review-toolkit` agents (silent-failure-hunter, type-design-analyzer) for deeper passes.

## 4. Multi-Agent Orchestration (Coordinator → Specialists)

**What it is:** A coordinator agent analyzes the codebase, drafts a spec, and delegates scoped tasks to specialist agents that run in parallel.

**Pros:**
- Parallelism on genuinely independent work; coordinator holds the plan so specialists stay scoped.
- Maps cleanly onto fan-out/verify patterns for large migrations and audits.

**Cons:**
- Coordination overhead and context-handoff loss; specialists can conflict on shared files.
- Hard to debug when an agent mid-chain goes wrong; cost scales with agent count.

**Suggestions:** Reserve for work one context can't hold (broad migrations, multi-module audits); use worktree isolation when specialists mutate files in parallel. Catalog support: `claude-squad`, `sandcastle`, `langgraph`, `humanlayer`.

### Cross-cutting takeaways

- **Model-agnostic + unified gateway + strong tests** is the repeated org-level prerequisite before unleashing autonomous agents.
- The through-line across all four workflows: **design the workflow, don't assemble it** — execution discipline (planning, small diffs, real verification, stop rules) is the differentiator, not the model.
- Net for this catalog: SDD and the bounded agentic loop are already represented in `WORKFLOW.md`; the gap worth tracking is **executable specs as validation gates** (Spec Kit / Kiro / OpenSpec) — none are yet evaluated here.
