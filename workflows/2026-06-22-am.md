**Shadow Testing: Co-Located Test Generation**

**What it is:** Shadow testing is a multi-agent pattern where a dedicated test agent runs alongside the implementation agent, generating tests concurrently as code is written rather than in a separate post-implementation pass. The test agent observes the implementation agent's output in real time — as each function or module is written, the test agent immediately generates corresponding tests, creating a tight feedback loop that surfaces failures before the implementation is complete. The pattern was named in the 2026 pattern catalog surveys as the emerging alternative to the sequential "implement → test" cycle that dominates single-agent workflows. The key insight is sequencing: in traditional agentic sessions, tests are an afterthought prompted after the implementation is done; in shadow testing, implementation and tests advance together, each constraining the other. Teams running this pattern report that the test agent often generates assertions that reveal spec gaps the implementation agent glossed over — not because the implementation was wrong per se, but because writing the test forces the question "under what conditions should this function behave differently?" that ad-hoc implementation skips.

**Pros:** Tests are written while implementation context is fresh, so edge cases and implicit assumptions are captured that post-hoc testing misses after the session has moved on. Failures surface before the implementation is finalized, preventing the expensive "write code, then write tests, then rewrite code" cycle that plagues sequential agentic workflows. Coverage emerges naturally rather than requiring a separate "add tests" pass that the agent often satisfies mechanically with shallow assertions. The pattern aligns with how experienced TDD practitioners work, making the agent's output legible to human reviewers who expect test-first discipline. When the test agent finds a mismatch between its understanding of the spec and the implementation agent's interpretation, that mismatch is valuable signal — it exposes ambiguity in the original task description rather than letting it silently propagate into shipped code.

**Cons:** Coordinating two concurrent agents adds orchestration complexity and cost that exceeds the benefit at small scale — for a single-function fix, shadow testing overhead outweighs the testing value. The test agent can only test what it observes the implementation agent producing: if the implementation has a subtle design flaw, the test agent generates tests that confirm the flawed behavior rather than exposing it, creating confident-looking coverage that hides the real problem. Running both agents simultaneously roughly doubles the per-session token cost, making budget management critical in parallel-agent deployments. Teams new to multi-agent orchestration frequently find that debugging failures in the two-agent coordination layer takes longer than the time saved by concurrent test generation.

**Suggestions:** The multi-agent orchestration tools in CATALOG.md (Agent Orchestration category) — fleet, multi-claude — provide the coordination infrastructure this pattern requires. The verifier agent pattern (2026-06-21-pm) is complementary and sequential to shadow testing: shadow testing generates tests concurrently during implementation, verification runs those tests after each iteration; together they form a continuous quality loop without redundancy. For teams not ready for two simultaneous agents, the minimal approximation is within a single session: instruct the agent to implement and test each function before moving to the next, alternating explicitly rather than batching all implementation then all tests — this recovers most of the freshness benefit without multi-agent coordination overhead. The loop engineering pattern (2026-06-21-am) provides the harness architecture that shadow testing runs inside.

**Bounded Execution Design: Avoiding Loopmaxxing**

**What it is:** Loopmaxxing is the failure mode where developers over-engineer agentic loops — layering reflection stages, verification rounds, retry logic, and sub-agent calls until the loop infrastructure is more complex than the tasks it executes, and any individual failure becomes hard to isolate. Bounded execution design is the counter-discipline: defining explicit stopping conditions before adding loop complexity, and holding the minimum viable loop architecture — perceive (gather context), reason (plan next action), act (execute), observe (evaluate result) — as the baseline to defend against accretion. Production guidance from 2026 practitioner accounts converges on a single rule: always set at least two independent stopping conditions before deploying an agent loop. Typical pairs are task-complete (all tests pass or the output satisfies a spec criterion) and iteration-cap (a hard ceiling on loop cycles, regardless of progress). The agentic loop's observation stage is where stopping conditions are evaluated; omitting explicit conditions there turns a loop into an open-ended agent that continues spending tokens on a task that already failed in the first iteration.

**Pros:** Explicit stopping conditions prevent runaway agents that continue burning tokens on an already-failed task without human intervention to break the cycle. Minimum viable loop architecture is faster to debug: the four-stage perceive-reason-act-observe structure has clear handoffs, so a failure localizes to one stage rather than propagating through a deeply nested pipeline. Bounded loops produce predictable cost profiles — iteration cap × per-iteration cost is a reliable upper bound that can be committed to a budget before the run starts. The discipline also forces teams to articulate what "done" means before the agent starts, which surfaces ambiguous acceptance criteria earlier than they would surface in an unbounded run. A rising iteration count on a given task type is a diagnostic signal: it indicates either prompt drift (the agent's understanding of the task has wandered) or a tool description that has grown ambiguous and is causing the agent to loop on misunderstood tool calls.

**Cons:** Over-constrained loops fail too early when tasks legitimately require more iterations than the cap allows — teams calibrating caps without data frequently set them too low for complex refactoring tasks and too high for simple fixes. Flat stopping conditions treat all iterations as equivalent: an agent making genuine progress toward completion and an agent spinning on the same failed tool call both count against the same iteration budget, making the cap a blunt instrument. The minimum-viable-loop principle is sometimes applied too aggressively: teams pruning reflection and verification stages that are genuinely necessary — not gold-plating — end up with faster but lower-quality output that fails review and requires a retry anyway. Defining "task complete" programmatically is harder than it sounds; "all tests pass" is clean but incomplete, since passing tests don't guarantee the implementation satisfies the spec.

**Suggestions:** The loop engineering pattern (2026-06-21-am) covers the broader harness design; bounded execution is the safety discipline layered on top. WORKFLOW.md's Verify stage maps directly onto the observation stage of the perceive-reason-act-observe loop: the same test run that gates CI can be the signal that satisfies the loop's "done" condition, avoiding a separate verification mechanism. For teams with existing CI, the simplest two-condition stopping criteria are already available without additional tooling: (1) CI green and (2) iteration limit, both of which can be encoded in a pre-existing GitHub Actions workflow without a custom agent harness. The Anthropic harnesses guidance (CATALOG.md → Agent Harnesses) provides the reference architecture for building loops with defined checkpoints — the bounded execution discipline is the policy that governs when those checkpoints trigger a halt rather than a retry.

**AGENTS.md as Universal Coordination Layer**

**What it is:** AGENTS.md is a context-file convention now consumed natively by nine major AI coding tools: Claude Code, OpenAI Codex CLI, Cursor, Aider, Devin, GitHub Copilot, Gemini CLI, Windsurf, and Amazon Q. Where CLAUDE.md is Claude-specific and Cursor's .cursor/rules/*.mdc are Cursor-specific, AGENTS.md is the cross-tool layer that gives any agent picking up the repository the conventions, architectural constraints, and testing policy it needs to produce aligned output regardless of which tool the developer is currently using. By mid-2026, engineering teams using more than one AI coding tool have converged on a split: AGENTS.md holds universal conventions (naming standards, testing requirements, architectural invariants, off-limits files), while tool-specific files handle per-tool behaviors on top. The practical driver is team heterogeneity: different developers use different tools, and without a shared context file, the same repository produces different behavior depending on whether the developer is in Cursor or Claude Code. AGENTS.md is the coordination mechanism that makes agent behavior consistent across the team even when tool choice varies.

**Pros:** A single authoritative file prevents convention drift between tools — when a developer switches from Cursor to Claude Code mid-task, AGENTS.md carries the accumulated knowledge forward without a context-file migration. Teams can adopt new AI coding tools without rebuilding context from scratch: the new tool reads AGENTS.md on first use and immediately operates within the team's conventions. The file is also readable by humans, serving as living documentation of conventions that would otherwise exist only in tribal knowledge or onboarding documents. Multi-agent pipelines where different agents use different tools — a Cursor agent for initial drafting, a Claude Code agent for review — share the same context file automatically, without an explicit handoff layer.

**Cons:** AGENTS.md has no schema enforcement — each tool interprets the file's contents differently, so a rule that works for Claude Code's parser may be silently ignored by Cursor's engine or truncated by a tool that reads only the first N tokens. Maintaining AGENTS.md alongside CLAUDE.md and .cursor/rules/ creates three context files to keep synchronized, and the maintenance overhead compounds as conventions evolve. The "universal" claim is limited by each tool's implementation fidelity: some tools read AGENTS.md superficially (using it as a prompt prefix) while others apply more sophisticated parsing, and developers cannot easily know which behavior a given tool is exhibiting. Teams face the same over-specification risk that afflicts CLAUDE.md: loading every conceivable convention into AGENTS.md without prioritizing the rules that actually move output quality produces a file so long that agents weight all rules equally regardless of importance.

**Suggestions:** Context engineering (2026-06-21-pm) established the framework for treating context files as first-class artifacts; AGENTS.md is the cross-tool extension that framework was missing. The practical split this repo uses — CLAUDE.md for Claude Code-specific behavior (tool permissions, MCP server config, hook configuration), AGENTS.md for cross-agent conventions — is the minimum viable adoption path and avoids the redundancy that comes from maintaining duplicate conventions in both files. The Memory & Context category in CATALOG.md (claude-mem, SimpleMem, mem0) addresses the dynamic complement: per-session state and accumulated knowledge that supplements the static conventions in AGENTS.md. For teams starting from an existing CLAUDE.md, the migration path is to extract the universal conventions into AGENTS.md and leave the Claude-specific configuration in CLAUDE.md — a one-time split that immediately benefits any second tool the team adopts.

**Code Health as AI Readiness**

**What it is:** Code health — structural quality measured by cyclomatic complexity, coupling density, duplicated logic, and test coverage — is now framed as a prerequisite for reliable agentic development, not just a legacy quality concern. The 2026 Agentic Coding Trends Report and CodeScene's analysis of agentic coding patterns converge on the same finding: agents working in low-code-health codebases fail more often, produce lower-quality patches, and burn significantly more tokens than agents working in structurally clean codebases. The mechanism is well-understood: high complexity and high coupling expand the context an agent must hold to make a safe change, pushing tasks beyond context limits and causing agents to make changes that satisfy the visible scope while breaking invisible dependencies. The CodeScene analysis is specific: problem code with poor quality metrics isn't "AI-ready" — low code health increases both the failure rate of agent tasks and the per-task token cost, making the productivity math worse precisely where developer pain is highest. The term "AI readiness" reframes code health investment from a quality concern (nice to have) into a platform concern (prerequisite for the tools the team depends on).

**Pros:** Investing in code health before deploying agents yields compounding returns: the same improvements that make agents more effective also make the codebase more maintainable by humans, so the investment is not siloed to AI productivity. Measuring code health before and after agentic development sessions creates an accountability metric for agent output — teams can track whether AI-generated code is improving or degrading the codebase's structural quality over time, rather than treating agent output as a black box. Teams with high-code-health codebases consistently report higher agent task success rates and fewer "regenerate from scratch" cycles, consistent with the token-efficiency framing. The AI readiness framing also gives developers a principled way to decide which parts of the codebase to target first for agentic automation: high-health modules are lower-risk, lower-cost targets for the first wave of agentic work.

**Cons:** Code health remediation is a significant investment that delays the productivity gains of agentic development — teams with large legacy codebases face a chicken-and-egg problem: they need agents to help improve the code, but the code's quality makes agent output unreliable enough to require heavy review. The AI readiness framing can become a rationalization for indefinite remediation before adopting agents at all, which avoids the harder organizational change while appearing principled. Agents assigned to improve code health (refactoring with tests) are themselves sensitive to the code health problem — a refactoring agent working in a high-coupling module can introduce regressions that the existing test suite doesn't cover. Code health thresholds are tool-dependent and lack a universal "good enough for agents" benchmark, making it difficult to know when remediation has achieved its goal.

**Suggestions:** CodeScene is the primary catalog tool for code health analysis (Code Review & Quality category in CATALOG.md) and is the source of the agentic AI readiness framing. WORKFLOW.md's Reflect stage is the natural measurement point: tracking health metrics before and after each sprint of agentic development identifies whether agentic output is paying down or accumulating structural debt. For teams baselining before agent deployment, the two highest-leverage metrics are cyclomatic complexity (directly drives the context load per task) and test coverage (defines the safety net for agent-generated changes) — both are available in standard analysis tools without CodeScene's full suite. The parallel worktree pattern (2026-06-19-pm, 2026-06-21-pm) interacts directly with code health: worktree isolation prevents agent tasks in high-coupling modules from contaminating each other's context, partially mitigating the code health risk without requiring remediation first.
