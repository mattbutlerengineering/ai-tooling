**Spec-Driven Development: Executable Specs as the Contract**

**What it is:** Spec-driven development (SDD) treats a structured, human-authored specification — not a prompt, and not the code itself — as the source of truth that governs everything the agent produces. The workflow: write a spec describing behavior, constraints, acceptance criteria, and edge cases; hand the spec to the agent as the implementation contract; evaluate agent output by running it against the spec rather than reading it for plausibility. SDD emerged as the structured response to "vibe coding" — the pattern where a developer prompts an agent with an informal description, iterates on visible output until it looks right, and ends up with code whose behavior is only as well-defined as the developer's eye. GitHub's Spec Kit announcement in mid-2025 and the explosion of SDD frameworks by early 2026 (Vishal Mysore maps 30+ in a single survey) mark the moment the pattern went from practitioner advice to ecosystem standard. The core claim, backed by GitHub's internal data: teams using structured specs ship with an order-of-magnitude fewer "regenerate from scratch" cycles than teams using ad-hoc prompting, because the agent has a verifiable target rather than an implicit one.

**Pros:** Specs make agent output auditable — the question "does this implementation satisfy the spec?" has a binary answer, unlike "does this code look right?" Specs persist across sessions, model upgrades, and team member changes, so the agent's target doesn't drift with whoever wrote the last prompt. When an agent's output fails, the spec localizes the failure: which acceptance criterion wasn't met? Teams can reuse specs across different agents and models, creating a library of verified behavioral contracts independent of the tool that executes them. The pattern also forces clarification of ambiguous requirements before implementation starts, surfacing decisions that would otherwise be made silently by the agent.

**Cons:** Writing a good spec takes more upfront time than writing a prompt — the discipline pays off at scale but adds friction for exploratory or one-off tasks where the requirements aren't yet known. Over-specified specs constrain the agent unnecessarily, producing brittle implementations that satisfy every stated criterion while missing emergent requirements. Maintaining spec files alongside code creates a second artifact that can drift: an out-of-date spec is actively misleading, not just absent. The tooling for spec-to-agent workflows is fragmented — over 30 frameworks exist but most teams are assembling their own pipeline from primitives rather than adopting a standard.

**Suggestions:** WORKFLOW.md's Plan stage is the natural home for spec creation — the spec is the machine-readable output of planning and the input to execution. The verification-aware planning pattern (2026-06-20-am) is the closest existing practice: acceptance criteria written at plan time are proto-specs that can be formalized into full SDD specs with minimal additional effort. For teams starting SDD, a lightweight spec format — behavior, constraints, acceptance tests, exclusions — in a SPECS.md file alongside CLAUDE.md prevents the tooling-fragmentation problem without committing to a heavyweight framework. The loop engineering pattern (2026-06-21-am) maps cleanly onto SDD: the spec's acceptance criteria become the loop's exit criteria, so the two patterns compose without redundancy.

**Verifier Agent Pattern: Agent Checks Agent**

**What it is:** A second, independent agent reviews the primary coding agent's output before it enters the codebase — checking for security flaws, behavior consistency against a spec, test coverage gaps, or factual correctness of documentation. The pattern emerged from multi-agent orchestration research and is now a first-class recommendation in the Anthropic agentic coding trends report for 2026: hierarchical architectures where a "reviewer" or "fact-checker" agent operates as a quality gate in the pipeline, separate from the agent that wrote the code. The key architectural insight is that the verifier agent's context is intentionally narrower than the coder's — it receives the diff, the spec, and the test results, not the full session history — which produces more reliable verification than asking the same agent to review its own output. Some teams run the verifier on a different model tier (e.g., Opus for verification even when Haiku wrote the initial draft) to catch reasoning errors the cheaper model might have glossed over.

**Pros:** Independent context produces more reliable reviews — a verifier agent that has never seen the coder's reasoning cannot rationalize the coder's mistakes. The pattern scales: one verifier can review the output of multiple parallel coding agents simultaneously, turning verification into a fixed overhead rather than a linear cost. Running the verifier on a different model or at a different temperature diversifies the failure modes checked, catching different classes of error than a single-model self-review would. Verifier output is structured and targetable — "failed criterion 3 of the spec" is more actionable feedback than a human reviewer's "this doesn't feel right."

**Cons:** The verifier adds latency and cost to every task, making it a poor fit for trivial edits where the verification overhead exceeds the implementation cost. A verifier is only as good as its instructions — a verifier told to "check for correctness" without a spec to check against produces vague feedback that doesn't improve on a human skimming the diff. Verifier agents can develop their own hallucination patterns, flagging false positives that slow down the pipeline or building false confidence when they miss real issues. Teams sometimes over-invest in the verifier architecture while under-investing in the spec it's checking against, which inverts the actual value chain.

**Suggestions:** The multi-agent orchestration entries in CATALOG.md (Agent Orchestration category) — fleet, multi-claude, KARIMO — all support multi-agent pipelines where a verifier role can be assigned to a dedicated agent. For teams with existing CI infrastructure, the lightest verifier implementation is a CI-stage agent that runs on every PR diff: it receives diff + test results + CLAUDE.md and outputs a structured review comment, adding verification without a separate orchestration layer. Pair with the SDD pattern above: the verifier agent's job becomes mechanically simple when the spec is explicit — "does the diff satisfy these criteria?" is a well-scoped prompt. code-review-graph (CATALOG.md → Code Review & Quality) is the closest catalog tool for blast-radius analysis, which complements spec-checking with dependency-aware review.

**Parallel Worktree Orchestration: Human as Traffic Controller**

**What it is:** Multiple agents run simultaneously in isolated git worktrees — each agent owns one branch, one task, one context window — while the human developer acts as orchestrator: assigning tasks, reviewing diffs, and deciding what to merge. The pattern is now the emerging consensus for high-throughput agentic development: rather than a single long-running agent session that blocks on each subtask, a developer fans out 3–8 agents across parallel worktrees and reviews the converging diffs as tasks complete. The git worktree primitive is the enabling mechanism — each agent gets a clean directory state without the interference, merge conflict risk, or shared-file contention of a single working tree. Addy Osmani's "Code Agent Orchestra" framing describes the developer's role explicitly as conductor rather than instrumentalist: setting tempo, adjusting priorities, integrating results, not writing the notes.

**Pros:** Wall-clock throughput scales with the number of parallel agents up to the developer's review bandwidth — a developer can supervise 5 agents writing 5 features simultaneously, compressing sequential development time by a large factor. Worktree isolation eliminates the class of agent failures caused by one task's changes contaminating another's context, making parallel runs more reliable than a single agent multitasking across files. Human review becomes the throughput bottleneck, which is the right bottleneck: the developer's judgment gates what enters the codebase, not the agent's availability. The pattern composes naturally with verification: each worktree's output can be checked by a verifier agent before the human even sees it, so the human's review queue contains only pre-screened diffs.

**Cons:** Managing 5+ simultaneous agent sessions requires a monitoring layer — without a dashboard showing which agents are running, stuck, or complete, the orchestration overhead falls back on the developer's attention. Context divergence accumulates over a long session: agents that started from the same commit progressively diverge as they each modify files, making late-stage merges increasingly conflict-prone. Task decomposition is a hard prerequisite — fanning out 5 agents on a poorly decomposed task produces 5 implementations of the same undefined thing. Developers new to the pattern frequently underestimate the review bandwidth required and end up with a queue of completed diffs they can't process fast enough to maintain the parallel sessions.

**Suggestions:** fleet and multi-claude (CATALOG.md → Agent Orchestration) are the catalog's primary tools for parallel agent visibility — both provide dashboards or status views across running agents. The parallel worktree pattern (introduced in 2026-06-19-pm and referenced in 2026-06-21-am) is the infrastructure layer this pattern sits on; the new contribution here is the explicit traffic-controller framing of the human role. Task decomposition at the Plan stage (WORKFLOW.md) is the critical upstream input: a well-decomposed task list with clear interface boundaries maps directly onto a well-isolated set of worktree assignments. For teams with GitHub Actions, a workflow that automatically runs the verifier agent on each worktree's PR as it's opened converts the parallel pipeline into a fully automated pre-review stage.

**Context Engineering: Configuration Files as First-Class Artifacts**

**What it is:** Context engineering is the discipline of treating the files that configure AI agents — CLAUDE.md, AGENTS.md, .cursor/rules/*.mdc, .github/copilot-instructions.md — as first-class code artifacts: versioned, reviewed, tested against agent behavior, and maintained with the same discipline as application code. By mid-2026, the consensus from practitioner accounts (Addy Osmani, GoGloby, Virtido) is that the primary variable separating teams with reliable agentic output from teams with inconsistent output is not model choice or prompt quality — it is the quality of the structured context those agents consume at session start. The discipline includes: maintaining a single source of truth for team conventions that all agents read (typically CLAUDE.md or AGENTS.md), writing context files in a format proven to work with the target agent (declarative rules > prose descriptions), and running context files through a regression test whenever they're modified — checking that changes don't degrade agent behavior on a known-good task set.

**Pros:** Well-engineered context files multiply model capability: an agent given precise style rules, lint constraints, and architectural invariants produces output that matches team standards without per-session prompting. Context files are the durable layer of an agentic workflow — they survive model upgrades, tool switches, and team member churn, providing continuity that prompt libraries don't. Versioning context files creates an audit trail for agent behavior changes: if agent output quality degrades after a CLAUDE.md edit, the regression is attributable and reversible. The discipline also forces teams to make implicit conventions explicit, which improves human code review independently of AI tooling.

**Cons:** Context files drift if not actively maintained — a CLAUDE.md that accurately described the codebase six months ago silently misleads agents with stale assumptions. Optimizing context files for one agent can accidentally degrade another's behavior: a rule written for Claude Code's parsing format may be ineffective or confusing to Cursor's rules engine. Teams frequently over-specify context files with every convention they can think of, producing files so long that agents either truncate them or weight all rules equally regardless of importance. Testing agent behavior against a context file change requires a structured benchmark — without one, "does this edit help?" is answered by impression rather than evidence.

**Suggestions:** This repo's CLAUDE.md is the canonical example of the pattern for Claude Code sessions. AGENTS.md (introduced in 2026-06-20-pm) is the cross-agent layer that both Claude Code and Cursor consume, preventing convention drift between tools. The Memory & Context category in CATALOG.md (claude-mem, SimpleMem, mem0) addresses the dynamic complement to static context files — per-session state that supplements the static conventions. For teams wanting to regression-test context changes, the eval harness pattern (TEMPLATE.md in this repo) applies: define a task set, run it against the old and new context file, compare output quality on the correctness and maintainability signals from WORKFLOW.md.

**PR Governance for AI-Generated Changes: Closing the 4.6× Gap**

**What it is:** A structured pipeline of ownership, coverage gates, and review tooling that prevents AI-generated pull requests from stalling in review or slipping into production without adequate scrutiny. The Opsera AI Coding Impact Benchmark (cited in mid-2026 practitioner discourse) found that AI-generated PRs wait 4.6× longer in review than human-authored PRs in teams without explicit governance — reviewers are unsure what to trust, uncertain who owns the intent, and lack the tooling to assess blast radius efficiently. The governance pattern addresses each bottleneck: named human ownership (one engineer explicitly responsible for the intent and correctness of each AI-generated diff), strict test coverage gates (automated enforcement that AI-generated code meets or exceeds the codebase's coverage threshold before review is requested), and structured PR descriptions (templates requiring the human owner to articulate what the change does, why, and the rollback plan). Together these collapse review time by giving reviewers a clear starting point and reducing the "I don't know where to begin" friction that inflates the wait.

**Pros:** Named ownership shifts the review dynamic: the reviewer is validating the owner's judgment rather than trying to reconstruct intent from code they know was AI-generated with no human in the loop. Coverage gates catch the most common failure mode of AI-generated code — plausible implementations with no tests, or tests that pass assertions without testing real behavior — before the diff reaches a human reviewer. Structured PR templates create a searchable record of intent: six months later, the PR description answers "why does this exist?" without requiring access to the original agent session. Teams that implement the full pattern consistently report review latency returning to near human-authored PR baselines, even as the volume of AI-generated PRs increases.

**Cons:** The 4.6× wait time is a symptom of process gaps, not a property of AI-generated code itself — governance closes the gap but also adds mandatory steps that slow individual PRs when applied uniformly to trivial changes that don't need them. Coverage gates can be gamed: an agent that knows the gate threshold can generate tests that satisfy it mechanically without providing real behavioral coverage, creating false confidence. PR description templates fill up with boilerplate when developers treat them as a form to complete rather than a communication tool, which defeats the intent. Teams sometimes implement governance as an overlay on an existing PR process rather than integrating it, producing duplicated review steps that add overhead without adding value.

**Suggestions:** The Human Accountability Chain pattern (2026-06-21-am) is the cultural complement to this structural pattern — governance enforces the chain at the process level while accountability enforces it at the norm level; both are needed. code-review-graph (CATALOG.md → Code Review & Quality) gives reviewers blast-radius visibility that directly reduces the "I don't know where to begin" friction. PR templates that require the three-sentence ownership statement (what, why, rollback) are the minimal viable governance implementation and can be added to any GitHub repo via .github/pull_request_template.md without additional tooling. For teams with CI infrastructure, WORKFLOW.md's Verify stage maps directly onto the coverage gate: the gate is the automated layer of verification that precedes human review in the outer loop.
