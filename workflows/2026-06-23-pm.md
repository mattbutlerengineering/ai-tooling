**Task-as-Issue Delegation: Writing Tasks Like Well-Formed GitHub Issues**

**What it is:** The most consistent operational pattern observed in high-performing agentic development teams in 2026 is not a framework or a tool — it is a discipline: writing tasks for coding agents exactly as you would write a well-scoped GitHub issue, with explicit background, goal, acceptance criteria, constraints, and test commands. This replaces the dominant practice of issuing ad hoc natural-language requests, which produces inconsistent results because agents are forced to infer scope, success criteria, and constraints rather than receiving them explicitly. The three-trait rule that has emerged from practitioner observation captures the structure: tasks are written with the specificity of a junior-engineer onboarding document (what is the context and why does this matter), permissions are managed with the discipline of a production system (least privilege, auditable, scoped to the task), and results are reviewed with the rigor of a pull request from a capable but inexperienced contributor (diff inspection, test coverage check, edge case identification). Organizations that adopt this operational discipline report more consistent agent output than those that invest in better prompting, because the issue-format imposes a structure the agent can reliably consume rather than depending on the agent to correctly infer what the developer intended.

**Pros:** The issue-format imposes a forcing function on the human side of the workflow: a developer who must articulate background, goal, acceptance criteria, constraints, and test commands before submitting a task to an agent will discover ambiguity in the task before the agent does, catching the most common source of bad agentic output (underspecified tasks) at the point where it is cheapest to fix. The least-privilege permission discipline ensures that an agent operating outside its expected scope — running a destructive command, modifying files outside its assigned directory, opening network connections — produces an observable permission error rather than a silent unintended side effect. Review-like-a-junior-PR keeps human oversight proportional to agent capability: agents in 2026 are fast and often correct, but the failure modes (plausible-but-wrong implementations, incomplete coverage, silent omissions) are exactly the failure modes junior engineers exhibit, and the code-review habits that catch junior-engineer mistakes transfer directly.

**Cons:** The issue-format discipline adds overhead to every task submission: writing explicit acceptance criteria and test commands for a small change that would have taken ten seconds to prompt and two minutes to execute takes longer than the informal approach on a per-task basis. Teams that adopt the discipline inconsistently — applying it to large tasks but not small ones — often find that the small tasks are where context loss, scope creep, and unintended side effects actually occur, because small tasks feel low-stakes enough to skip the structure. The least-privilege permission model requires upfront configuration that many teams skip, defaulting to the broadest permission set for convenience; the cost of this shortcut appears not when the agent succeeds but when it runs out of expected scope and the permission audit trail is unavailable to diagnose what happened.

**Suggestions:** The WORKFLOW.md Plan stage is the entry point for issue-format task writing — the discipline of explicit background, goal, acceptance criteria, constraints, and test commands is the operational implementation of the Plan→Implement handoff. The Security & Safety category in CATALOG.md provides the permission audit tooling that makes least-privilege enforcement observable rather than aspirational; pairing agent permission management with static analysis on agent-generated diffs is the two-layer safety net that survives agent scope drift. For teams starting out: before submitting any task to an agent, write the acceptance criteria first — if you cannot write three concrete criteria, the task is underspecified and the agent will produce an underspecified result regardless of how capable the underlying model is.

**Spec-Driven Development: Industry Convergence on Specification as Source of Truth**

**What it is:** By mid-2026, spec-driven development has ceased to be a niche methodology and become the default structural assumption of every major AI coding platform: GitHub Spec Kit, AWS Kiro, Claude Code (via CLAUDE.md and spec templates), Cursor, OpenSpec, BMAD, and Tessl have all shipped their own implementation of the same core principle — the specification, not the code, is the primary artifact of software delivery. The principle is operationally specific: a spec defines six elements — outcomes (what state the system should reach), scope boundaries (what the agent is and is not permitted to touch), constraints (technical or business rules the implementation must satisfy), prior decisions (architectural choices already made that the agent must not reverse), task breakdown (the ordered sequence of verifiable subtasks), and verification criteria (the machine-executable tests that confirm each subtask completed correctly). The convergence is significant not because any single tool's implementation is correct but because the field has agreed on the artifact: before generation begins, a structured document must exist that answers "what should exist when this is done?" in terms an agent can evaluate against its own output. This is the operational antidote to "vibe coding" — unstructured generation guided by approximate natural-language intent — which produces high initial velocity and high subsequent maintenance cost.

**Pros:** A well-formed spec with verification criteria turns every agentic session into a verifiable unit of work: the agent either satisfies the acceptance criteria or it does not, and the answer is machine-determinable rather than requiring human judgment after every run. The task breakdown element enables a Plan-Implement-Verify cycle at the subtask level — the agent delivers one subtask, the developer or a verifier agent checks it against the verification criteria, and the next subtask begins from a confirmed baseline rather than from accumulated uncertainty. Spec convergence across tools also means that a spec written in one platform is increasingly portable: a spec written for a Claude Code session can be adapted to a Cursor session because the underlying six-element structure is the same, reducing the switching cost of multi-tool workflows.

**Cons:** Spec quality is highly variable and poorly tooled: a spec that lists outcomes without verification criteria, or that omits scope boundaries, is worse than no spec at all because it creates a false sense of structure without the machine-evaluability that makes specs valuable. The deeplearning.ai course on spec-driven development with coding agents had to ship a dedicated curriculum on "how to write a good spec" because the field has converged on the artifact before converging on the craft — teams that adopt SDD without investment in spec quality quickly discover that their agents are as inconsistent as they were without specs, because the spec is too vague to anchor agent behavior. The six-element structure also imposes scope discipline that conflicts with exploratory work: a developer building something genuinely novel does not know the full scope boundary or all prior decisions before they start, and forcing a complete six-element spec before generation begins blocks the discovery work that should happen before the spec is written.

**Suggestions:** WORKFLOW.md's Plan stage is the natural home for spec authoring — the spec is the concrete artifact that the Plan stage produces before the Implement stage begins. The setup-workflow skill (skills/setup-workflow/) includes a spec template for bootstrapping the six-element structure in a new repository; the evaluate-tool skill (skills/evaluate-tool/) follows a structurally parallel pattern (outcomes, evidence criteria, verification) and illustrates how the same spec discipline can anchor evaluation work, not just implementation work. For teams evaluating SDD tools: the useful comparison point is not feature count but spec portability — does the tool's spec format produce machine-executable verification criteria, or does it produce human-readable descriptions that require judgment to evaluate? The former supports agentic verification loops; the latter does not.

**Loop Engineering: Designing Loops That Prompt Agents**

**What it is:** The most significant craft shift reported by experienced agentic developers in mid-2026 is the move from prompt engineering to loop engineering — from writing better prompts for individual agent interactions to designing the loops that govern how agents operate continuously over multi-step tasks. A loop is a structured execution context with defined inputs (what the agent starts with), defined tools (what the agent can call), defined observation format (what the agent reads back after each action), defined termination conditions (when the loop stops), and defined escalation criteria (when the agent pauses and asks for human input rather than continuing). The framing is explicit in practitioner literature: "stop prompting coding agents and focus instead on designing loops that prompt those agents" — the developer's role shifts from author of individual instructions to architect of the execution context in which the agent generates its own instructions at each step. This is a meaningful abstraction level change: prompt engineering operates at the granularity of a single interaction, while loop engineering operates at the granularity of an autonomous workflow that may run for minutes or hours with no human in the loop.

**Pros:** A well-designed loop is reusable across tasks in a way that individual prompts are not: a loop that implements Plan→Act→Observe→Correct for test-driven implementation is the same loop whether the task is adding an endpoint or fixing a type error; the task changes, the loop stays constant. Termination conditions and escalation criteria also make loops safer than unconstrained ReAct agents: a loop that halts when it has made ten consecutive Act steps without progress, or when an error rate exceeds a threshold, limits the blast radius of a misdiagnosed task in a way that an open-ended agent cannot. "Loopmaxxing" — extending loops indefinitely in the hope that the agent will eventually self-correct — is identified in mid-2026 practitioner writing as the most common failure pattern, and explicitly scoped termination conditions are the structural answer to it.

**Cons:** Loop design is a new skill with no established curriculum: most developer education is organized around prompt writing, code review, and system design, none of which directly maps to the skills of defining observation formats, calibrating escalation criteria, or reasoning about token consumption across a 30-step loop. Teams that attempt loop engineering without sufficient expertise often produce loops that are too permissive (no meaningful termination condition, no escalation path) or too restrictive (step budgets so tight that agents terminate before completing the task), and debugging a badly designed loop is harder than debugging a bad prompt because the failure surface is distributed across many steps. Loop engineering also conflates two distinct problems — what the agent does and how the developer observes what the agent did — and teams that design good loops but poor observability cannot diagnose loop failures because the execution trace is not legible.

**Suggestions:** WORKFLOW.md's Implement stage is where loop execution lives, but the loop architecture decision belongs at the Plan stage — the loop design (inputs, tools, termination, escalation) should be an explicit artifact of planning, not an implicit consequence of tooling defaults. The Agent Harnesses category in CATALOG.md (claude-code harness tooling, hook infrastructure) provides the implementation layer for loop design decisions: hooks on tool calls, step counters, and error-rate monitors translate loop design into enforcement. A minimum viable loop has four defined elements: a starting context (what the agent reads before its first action), a tool allowlist (what it can call), a step budget (how many Act steps before forced termination), and one escalation condition (what observation triggers a human checkpoint); adding more loop structure before the four-element baseline is operational adds complexity without proportional safety gain.

**Context Engineering: Active Codebase Investigation as the New Core Skill**

**What it is:** The consensus from mid-2026 practitioner accounts is that the primary skill differentiating high-performing agentic developers from low-performing ones is not prompt writing but context engineering — the discipline of ensuring agents have the right information, in the right structure, at the right point in the execution loop. The shift is architectural: older static indexing approaches gave agents a pre-built summary of the codebase that the agent consumed passively; 2026 agents are designed to actively investigate the repository at the start of each session, building a working understanding of structure, patterns, conventions, and dependencies before generating any code. The practical implication is that a developer's most leveraged investment is not in writing better prompts but in making the codebase more legible to investigation: explicit CLAUDE.md or AGENTS.md guidelines files, consistent naming conventions, self-documenting module boundaries, and comprehensive README files that a context-gathering phase can read in seconds and correctly interpret. Context engineering operates on both sides: the agent side (what the agent reads and how) and the codebase side (how the codebase is organized to be readable by agents under time and token constraints).

**Pros:** Active codebase investigation at the session start front-loads the context cost: an agent that spends 30 seconds reading the relevant files, tests, and conventions before generating code produces more accurate first drafts than an agent that starts generating immediately and discovers relevant context mid-loop through error messages. Investing in codebase legibility — guidelines files, naming conventions, module boundaries — pays compounding dividends: every improvement to codebase structure makes every future agentic session more accurate, so the return on investment scales with agent usage rather than capping out. Context engineering also exposes codebase health problems that manual development tolerates: an agent that struggles to understand the module structure is surfacing a legibility problem that affects human developers too, but which human developers compensate for through accumulated familiarity that agents cannot replicate.

**Cons:** Active investigation consumes tokens and wall-clock time before any productive work begins: a session that starts with a thorough context-gathering phase on a large codebase can exhaust a meaningful fraction of the token budget before writing a single line of code. There is no established standard for "how much context is enough" — agents that gather too little miss relevant conventions and produce inconsistent implementations; agents that gather too much run out of context window before completing the task. Codebase legibility investment is also asymmetric in its returns: a well-organized, well-documented codebase benefits enormously from active investigation; a degraded codebase with inconsistent conventions and poor documentation sees the agent's context-gathering phase surface a chaotic picture that produces inconsistent rather than coherent generation. The teams that most need better context engineering are the ones whose codebases are hardest to engineer context from.

**Suggestions:** WORKFLOW.md identifies code health as an AI readiness signal — the connection between codebase legibility and agentic session quality is direct and measurable. The Memory & Context category in CATALOG.md provides tools for structuring the context that agents read at session start; the Code Review & Quality category provides the codebase health tooling that makes the legibility investment computable. Guidelines files (CLAUDE.md, AGENTS.md) are the highest-leverage single codebase artifact for context engineering: a well-maintained guidelines file that explains conventions, architecture decisions, and anti-patterns can replace minutes of active investigation with seconds. For teams starting the context engineering discipline: audit one failing agentic session per sprint for context failures — "the agent didn't know about X convention" or "the agent missed the Y module" — and fix each failure with a codebase change rather than a prompt change; within three sprints, the most common context failures disappear.

**Role-Specialized Agent Teams: Planner → Architect → Implementer → Tester → Reviewer**

**What it is:** The most ambitious multi-agent pattern emerging in mid-2026 is the role-specialized agent team — a pipeline of agents each assigned a specific role that mirrors a real engineering team, rather than a single general-purpose agent performing all roles in sequence. The canonical five-role pipeline observed across multiple platforms is: a Planner agent that decomposes the feature request into a sequenced task list with acceptance criteria; an Architect agent that reviews the task list against the existing codebase and produces implementation constraints; an Implementer agent that generates code against the constraints; a Tester agent that writes and executes tests against the implementation and reports failures; and a Reviewer agent that evaluates the implementation against the spec and flags spec drift, coverage gaps, and style violations before the PR is opened. Each agent receives only the output relevant to its role: the Implementer does not see the Planner's raw decomposition, only the Architect's constrained task definition; the Tester does not see the Planner or Architect output, only the implementation plus the acceptance criteria. Role specialization reduces context bloat (each agent's context window contains only what its role requires) and enables parallel execution where roles are independent.

**Pros:** Role specialization enforces the separation of concerns that is difficult to maintain in a single general-purpose agent: an Implementer agent that cannot make architectural decisions (because it never sees the full task decomposition, only its scoped subtask) cannot make the architectural mistakes that general-purpose agents make when they exceed their competence boundary. The Tester and Reviewer roles make the verification phase explicit and structurally guaranteed rather than optional: a pipeline with a Tester agent always runs tests; a general-purpose agent may or may not run tests depending on whether the prompt asked it to. Role specialization also maps naturally onto existing engineering team communication: the outputs of each role (task decomposition, implementation constraints, code, test results, review notes) are artifacts that human developers can review and intervene on at the role boundary, rather than needing to inspect an opaque multi-step execution trace.

**Cons:** Five-agent pipelines are expensive — each role transition incurs model invocation cost and latency, and a five-role pipeline on a medium-sized feature can cost more than a single capable agent completing the same feature, particularly if the task is simple enough that role specialization adds overhead without adding quality. Pipeline rigidity is also a failure mode: the Planner → Architect → Implementer → Tester → Reviewer sequence assumes each role completes cleanly before the next begins, but in practice the Implementer often discovers constraints the Architect missed, the Tester often discovers acceptance criteria the Planner left ambiguous, and a rigid pipeline has no mechanism for these discoveries to propagate backward without a full pipeline restart. Teams that implement the five-role pattern before achieving reliable two-role (Planner + Implementer) operation frequently find that pipeline failures accumulate before any role produces reviewable output.

**Suggestions:** CATALOG.md's Agent Orchestration category (fleet, multi-claude) provides the scaffolding for role-specialized pipelines in Claude Code-based workflows. WORKFLOW.md's outer loop maps naturally onto the five roles: Plan stage → Planner + Architect agents; Implement stage → Implementer agent; Verify stage → Tester agent; Review stage → Reviewer agent; the Ship stage remains human-gated. The minimum viable role-specialized pipeline is two roles, not five: a Planner agent that produces a structured task list with acceptance criteria, followed by an Implementer agent that executes against that task list — this two-role separation delivers the most significant quality improvement (explicit planning before generation) at the lowest pipeline complexity and is a prerequisite for adding further roles. Add the Tester role only after the Planner + Implementer pipeline is stable; add the Architect and Reviewer roles only after the three-role pipeline has been validated.
