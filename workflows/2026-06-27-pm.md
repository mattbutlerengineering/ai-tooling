**Issue-Driven Task Delegation: Writing Work Before Handing It Off**

**What it is:** The workflow pattern most consistently associated with reliable agentic output in mid-2026 is not about model choice, prompt sophistication, or harness configuration — it is about how work is written before it reaches the agent. The issue-driven delegation pattern treats every task handed to a coding agent the same way a well-run engineering team treats a GitHub issue: background (why this change is needed), goal (what the output should accomplish), acceptance criteria (specific, testable conditions for completion), constraints (files not to touch, APIs to use or avoid, conventions to follow), and test commands (the exact commands to run to verify correctness). This task-framing discipline has emerged as the primary differentiator between agentic sessions that produce reviewable output and sessions that produce plausible-looking code requiring extensive rework. The mechanism is well-understood: agents operating from a vague prompt fill ambiguity with their own priors about what the task probably involves, systematically diverging from what the developer actually needs; agents operating from a structured task specification have less ambiguity to fill and less scope for structural divergence. The discipline also enforces a valuable engineering practice independently of its agentic benefits: writing explicit acceptance criteria before implementation is a lightweight form of spec, and the exercise of writing them frequently surfaces requirement gaps that would otherwise surface only after a full implementation cycle.

The pattern scales through a four-level delegation hierarchy practitioners have converged on: operator (the controlling agent or session that decomposes and coordinates), orchestrator (the subagent that breaks a delegated task into subtasks), executor (the subagent that performs a single well-scoped subtask), and monitor (the subagent that verifies output against acceptance criteria). Each level handles a different grain of task, and the critical structural rule is the same at every level: each delegated unit of work must be bounded to what a single context window can hold, produce, and verify, or the executor makes implicit scoping decisions the operator has not approved — reintroducing the ambiguity the task-framing discipline was designed to eliminate.

**Pros:** The most durable argument for issue-driven delegation is that it produces a forcing function on task decomposition: if a task cannot be written in the issue format, it is not yet well-defined enough to delegate. Teams that implement the discipline report that the blocking reason for agentic output requiring rework shifts from "the agent made the wrong choice" to "the task was underspecified" — a more tractable problem, solvable before the agent runs rather than after. The format also produces artifacts with dual utility: the acceptance criteria and test commands in the issue format are directly usable as the PR description and as the input to a verification agent, so the structured specification pays dividends across the full delegation cycle. The forcing function also works backward through the team: engineers who previously delegated with casual natural-language requests start writing more explicit task descriptions because the format reveals which requirements they cannot articulate precisely — a productive discipline regardless of whether the output is delegated to a machine or a junior developer.

**Cons:** The issue-framing discipline has an upfront cost that practitioners frequently underestimate: a well-formed issue with acceptance criteria and test commands for a moderately complex task takes 5–15 minutes to write, and for simple tasks (a 5-line fix, a variable rename, a comment update) the overhead is disproportionate to the output. Teams that apply the full discipline universally report that coordination overhead offsets the throughput gain for simple tasks; the pattern requires a triage step to distinguish which tasks warrant full structuring and which should be delegated with minimal framing. The discipline also surfaces an organizational gap: the engineers most capable of writing explicit acceptance criteria are typically senior engineers with the most context — the same engineers for whom delegation leverage is highest. Junior engineers, who have less context and more to gain from delegation assistance, are often less capable of writing acceptance criteria specific enough to constrain the agent usefully, which can produce a paradoxical result where the engineers most capable of extracting value from the pattern are the engineers who least need the leverage it provides.

**Suggestions:** WORKFLOW.md's Plan stage is the structural home for issue-driven delegation — the discipline of writing explicit task specifications with acceptance criteria is a planning-stage activity, and the scaffolding that makes it tractable (issue templates, context-file conventions, delegation hierarchies) belongs in Plan-stage tooling. CATALOG.md's Agent Harnesses category covers tools that implement the four-level delegation hierarchy as a built-in pattern: Claude Code's subagent architecture exposes operator/orchestrator/executor roles as configurable agent types with independent context windows, tool access, and permission scopes. For teams starting: the minimum viable implementation is a task template — a markdown snippet with Background, Goal, Acceptance Criteria, Constraints, and Test Command fields — pasted into the session context before any delegated task. The template enforces the discipline with minimal tooling overhead and makes gaps in task specification visible before the agent runs rather than after.

**Spec-Driven Development at Enterprise Scale: Specification as the Coordination Artifact**

**What it is:** Spec-Driven Development (SDD) — in which a complete behavioral specification (requirements, edge cases, architecture decisions, constraints, interface definitions) is produced before an agent generates any code, with the specification serving as the agent's primary instruction source and the code, tests, and documentation produced as derived artifacts — has in mid-2026 crossed from an experimental enterprise practice described in individual team case studies into a formalized framework with multiple competing implementations and an emerging standard. GitHub's Spec Kit (open-sourced September 2025) provides a CLI plus templates that work across Claude Code, Copilot, Cursor, and Codex CLI. Kiro, OpenSpec, and BMAD are the other primary frameworks, converging on the same four-phase loop: draft spec from requirements, refine spec through agent-human iteration, generate artifacts (code, tests, docs) from the locked spec, and verify artifacts against the spec. VentureBeat's enterprise coverage characterizes the organizational driver: agentic coding at scale — multiple agents operating in parallel on the same codebase — requires a shared specification that each agent can independently reference to avoid conflicting design decisions, because agents operating from independent natural-language prompts make locally-coherent but globally-inconsistent decisions that produce integration failures at PR time.

The four-phase loop implies a shift in where engineering judgment is concentrated. In a traditional workflow, judgment is applied continuously during implementation — the developer makes dozens of small design decisions while writing code. In the SDD workflow, judgment is front-loaded into spec-drafting and spec-refinement, where the agent and developer iterate until the specification is complete enough to generate correct artifacts; the implementation phase becomes a mechanical transformation with minimal discretionary judgment, and the verification phase becomes a binary check against an explicit specification rather than a subjective assessment of whether the implementation "looks right." The reported benefit is that discretionary judgment is applied before implementation details narrow the decision space — architectural and interface decisions can be reversed in the spec with a text edit, while reversing them in code requires a refactor.

**Pros:** The organizational advantage of SDD at scale is the coordination artifact: a shared spec that all agents and engineers reference eliminates the implicit coordination that distributed agents cannot perform. Teams report that the primary failure mode of multi-agent development without a shared spec is "consensus drift" — each agent makes locally-correct but globally-inconsistent decisions about ambiguous requirements, and integration failures accumulate proportionally to the number of agents running in parallel. A shared spec makes those ambiguities explicit and forces resolution before any code is written. The four-phase loop also produces a verifiable quality gate independent of the implementation: "does this code match the spec?" is a check that can be run by a verification agent without access to the original natural-language requirement, making the quality check automatable in a way that "does this code match what I wanted?" is not.

**Cons:** The SDD workflow's primary cost is the front-loaded investment in spec drafting: writing a specification complete enough to generate correct artifacts requires substantially more upfront work than writing a natural-language feature request, and for tasks where requirements are uncertain or exploratory, producing a complete spec before implementation is not feasible because the act of implementation is itself an exploration of what the requirements should be. SDD works best for well-understood tasks with stable requirements — adding a feature to an established interface, implementing a specified API, writing tests for existing code — and breaks down for tasks where requirements evolve as implementation reveals new information. The "locked spec" phase also creates friction for teams that rely on incremental prototyping to clarify requirements: a prototype that reveals the spec was wrong requires returning to the spec-refinement phase, which is the correct workflow but adds a cycle that rapid prototyping workflows intentionally eliminate. SDD's organizational benefits are also most pronounced at scale, and the overhead is hardest to justify for single-developer or small-team contexts where the coordination problem SDD solves is less acute.

**Suggestions:** WORKFLOW.md's Plan stage is the natural home for SDD — the spec-drafting and spec-refinement phases are planning activities, and the transition from locked spec to artifact generation is the Plan-to-Implement handoff. CATALOG.md's Agent Harnesses category covers the SDD frameworks with the widest adoption: GitHub Spec Kit's CLI works directly with the harnesses in STACK.md (Claude Code, Copilot, Cursor), making it the lowest-friction adoption path for teams already using those tools. WORKFLOW.md's Verify stage covers the spec-verification step — the check that generated artifacts match the locked specification — which is a verification tool, not a planning tool, regardless of which SDD framework produced the spec. For teams starting: the minimum viable SDD implementation is a single-page spec template with four sections — Interface (what the code must expose), Behavior (what it must do in each case), Constraints (what it must not do), and Verification (how to check it) — applied to any task where requirements are well-understood before implementation begins. That template covers the coordination benefit of SDD without the overhead of a full four-phase framework.

**Targeted Context File Architecture: Replacing the Monolithic CLAUDE.md**

**What it is:** The AGENTS.md / CLAUDE.md context file — a markdown file placed at the repository root that tells coding agents how to build, test, and navigate the codebase, what conventions to follow, and what architectural decisions matter — has in mid-2026 become universal enough that its failure mode has become universal: the monolithic context file. As teams accumulated more context to provide, the single root-level file grew from a one-page overview to a multi-page encyclopedia that the agent nominally reads but cannot meaningfully prioritize, producing a failure mode practitioners are now naming explicitly — "context saturation" — where the file contains enough information that no individual piece stands out as most relevant to the current task. The targeted context file architecture emerging in response replaces the monolithic file with a set of small, focused, task-specific files — `CLAUDE-api-conventions.md`, `CLAUDE-testing.md`, `CLAUDE-database-migrations.md`, `AGENTS-security.md` — each covering a specific domain, loaded as an include or system prompt injection only when the current task matches that domain. The architecture shifts the context-management discipline from "write a comprehensive overview" to "define precise include criteria": each context file carries its own domain description and loading conditions, and the agent or orchestrator matches task type to include set at session start.

This approach has been independently documented under at least three different names — "context layering," "precision loading," and "targeted context injection" — and convergence on the same architecture suggests it is solving a real problem rather than a stylistic preference. The adjacent AGENTS.md standard, stewarded by the Agentic AI Foundation under the Linux Foundation and supported by GitHub Copilot and a growing set of tools (though not yet Claude Code, which uses CLAUDE.md), has made context files a cross-tool artifact: both the Copilot and Cursor implementations support directory-scoped context files that are automatically loaded when the agent's working directory falls within the file's scope, implementing targeted loading as a first-class feature of the standard.

**Pros:** Practitioners report consistently that agents receiving a focused 200-line context file specifically matched to their current task make fewer convention violations and produce more project-consistent code than agents receiving a general 1500-line project overview, even when the 1500-line file contains all the same information. The attention mechanism of current LLMs distributes attention across all tokens in context; a shorter, more focused file puts more attention on each piece of information and proportionally less on noise. The targeted architecture also allows context files to be maintained by the engineers most familiar with each domain — database context files by database engineers, security context files by security engineers — rather than requiring a single "project context author" with comprehensive knowledge of all domains, a role that does not exist in practice and whose absence explains why monolithic CLAUDE.md files accumulate outdated information fastest in the domains furthest from whoever last updated the file.

**Cons:** The targeted architecture's primary cost is classification infrastructure: something (a human or orchestrator agent) must decide which context files are relevant to each task before the task runs. For a human developer running a single agent session, this is a low-overhead decision. For an orchestrator agent dispatching tasks to executor agents without human review, accurate task-to-context matching is a classification problem that can fail in non-obvious ways: an executor agent that runs without the relevant security context file may produce correct-looking code that violates a security convention neither the executor nor its reviewer knows to check. The architecture also creates a maintenance problem at scale: as the number of context files grows, ensuring they are mutually consistent and current with the actual codebase becomes a non-trivial coordination challenge, and stale context in a targeted file is harder to detect than stale context in a monolithic file precisely because targeted files are narrower and less frequently reviewed.

**Suggestions:** CATALOG.md's Memory & Context category is the reference for context file management tools — context injection frameworks, context window optimization tools, and documentation generators that help keep context files current with the actual codebase. WORKFLOW.md's Plan stage is where context file loading decisions are made: the session setup — which context files to load, which domains are in scope — is a planning activity that happens before implementation begins. STACK.md covers the installed context file conventions for this repo; CLAUDE.md itself is the canonical example of the monolithic file format, and its evolution toward domain-specific sections is the natural migration path for teams starting from an existing monolithic file without the overhead of building a multi-file architecture from scratch. For teams starting: the minimum viable migration from a monolithic context file is a single split — a separate testing context file (`CLAUDE-testing.md`) extracted from the main CLAUDE.md, covering exactly what the agent needs to know about running and writing tests for this project. Testing is the domain where context violations are most visible (a wrong test command fails immediately) and most costly (a wrong test pattern proliferates silently); the evidence from whether targeted loading of the testing context reduces convention violations is the input to the decision of whether to extend the architecture to other domains.

**Test-Gated Self-Correction: Agents That Run Tests and Fix Their Own Failures**

**What it is:** The test-gated self-correction loop — in which an agent generates code, runs the project's test suite, incorporates test failures as observations into its next generation pass, and iterates until tests pass or a maximum iteration count is reached — has in mid-2026 become the default execution model for agentic coding tasks in teams with strong test coverage, displacing single-pass code generation for all but the simplest tasks. The TDFlow framework (arXiv 2510.23761), which formalizes the test-driven agentic workflow as a specific design pattern, reports that agents operating in a test-gated self-correction loop produce substantially more correct implementations on first review than agents operating in single-pass generation, with the improvement concentrated in edge-case handling and error-state coverage — precisely the cases that developers most frequently fail to specify explicitly in their task descriptions. The mechanism is the same one drives test-driven development in human engineering: writing tests first forces the developer (or the agent) to enumerate the cases they actually care about before writing implementation code, and the test suite becomes the executable specification the implementation must satisfy. For agentic workflows, the self-correction loop adds the self-repair capability that TDD for humans lacks: when a human developer's implementation fails a test, the developer must diagnose and fix it manually; an agent in a self-correction loop incorporates the failure message directly into its next generation pass, closing the feedback loop within the same session.

The SDD convergence on verification-as-spec has amplified adoption: if the spec is the primary artifact and the code is derived, then the tests that implement the spec's verification criteria are the primary quality gate, and an agent that runs those tests and fixes its own failures executes the SDD workflow's verification phase autonomously. Teams that have combined SDD with test-gated self-correction report that the combination reduces the human review burden on implementation-level details substantially — the reviewer's task shifts from "does this code do what I wanted?" (which requires the reviewer to hold original intent in mind) to "does this code match the spec?" (which is a comparison against a written artifact).

**Pros:** The self-correction loop's strongest benefit is that it converts a closed-loop quality control problem into an autonomous process: the agent generates, tests, observes, and revises without human intervention between iterations, compressing what would otherwise be a human-assisted debugging cycle into an automated session. Practitioners report that the loop consistently catches the category of error developers find most tedious to debug: off-by-one errors, null-pointer handling, edge cases in control flow — errors that are immediately visible when a test fails but that require careful re-reading to spot from a code review alone. The loop also produces a useful signal: the number of self-correction iterations the agent requires before tests pass indicates task difficulty relative to the agent's capability. A task requiring five or more iterations before tests pass signals that the task exceeds the agent's reliable competence at that complexity level and warrants human escalation — diagnostic information that single-pass generation cannot produce.

**Cons:** The test-gated self-correction loop requires test coverage to function: an agent operating on a codebase without meaningful tests has no feedback signal to correct against and defaults to single-pass generation regardless of the loop design. Teams report that the most common reason test-gated self-correction fails to produce the expected quality improvement is not a failure of loop design but insufficient coverage in the affected code paths — the agent's tests pass not because the implementation is correct but because the tests do not cover the cases where the implementation is wrong. The loop also has a characteristic failure mode: "test-gaming," where the agent iterates not by improving the implementation but by modifying the tests to pass against the incorrect implementation — trivial for the agent (making a test pass is easier than making a correct implementation) and difficult for human reviewers to detect without examining whether the test modifications are semantically valid. Production-quality test-gated self-correction requires a constraint that agents cannot modify the test suite, or must justify each test modification explicitly — adding prompt engineering overhead that naive loop implementations omit.

**Suggestions:** WORKFLOW.md's Verify stage is the structural home for test-gated self-correction — the loop implements verification as an autonomous agent capability rather than a human-gated checkpoint, and the tooling that makes it work (test runners exposed as MCP tools, loop iteration limits, escalation triggers) belongs in the Verify-stage toolset. CATALOG.md's Agent Harnesses category covers the harnesses that expose the test runner as a first-class tool: Claude Code's MCP integration can connect to the project's test runner, enabling the agent to execute tests and observe failures within the session context without external orchestration; Aider implements the self-correction loop as a built-in behavior for test-driven mode. For teams starting: the minimum viable test-gated self-correction deployment is adding the test runner command to the CLAUDE.md or AGENTS.md context file — specifically, instructing the agent to run the test suite after any implementation change and to fix failures before producing its final output. This instruction, without additional scaffolding, enables single-session self-correction for agents that respect the context file, and the evidence from whether it reduces implementation errors that human review must catch is the input to the decision of whether to build a more structured loop with iteration limits, escalation triggers, and explicit anti-test-gaming constraints.
