**Workflow Mode Selection: Matching the AI Interface to the Task**

**What it is:** As of mid-2026, developers operating with AI coding agents have four distinct workflow interfaces available — IDE (in-editor assistant, inline completions and chat), terminal (CLI harnesses like Claude Code, aider, or opencode that operate directly on the working tree), pull request (PR-level agents that review, annotate, and implement changes against existing diffs), and cloud (long-running autonomous agents triggered by issues or webhooks, operating in remote ephemeral environments). The choice of interface is not cosmetic: each mode has a different context surface, latency profile, feedback loop, and appropriate task type. IDE mode excels at rapid, high-context completions and small refactors where the developer is actively steering; terminal mode excels at bounded, longer-horizon tasks where the agent needs full filesystem access and tool execution; PR mode excels at review and fix-up of already-scoped changes; cloud mode excels at overnight or background tasks that are too slow or expensive to run interactively. The practitioner failure mode identified by Real Python's 2026 guide is workflow mode mismatch: developers reaching for IDE mode for tasks that require filesystem traversal and test execution (where terminal mode is the correct interface), or triggering cloud agents on tasks that require tight feedback loops (where IDE or terminal mode is more appropriate). Mode selection is now treated as the first design decision in an agentic workflow, not a default.

**Pros:** Matching mode to task eliminates a category of failure that is invisible when developers use only one mode. A task that requires reading 40 files, running tests, and iterating on output is structurally mismatched to IDE mode — the IDE agent has no persistent context across tool calls, no filesystem traversal, and no CI integration — and the mismatch produces output that looks plausible in the editor but fails in the test suite. Terminal mode eliminates this class of failure for that task type by giving the agent the full working tree, a persistent session, and direct tool execution. PR mode similarly excels at its specific task type — code review, annotation, fix-up — because the diff surface is a natural context boundary that keeps the agent's context focused on exactly the change under review, rather than the entire codebase. Deliberate mode selection also enables cost optimization: cloud agents running in ephemeral environments on low-traffic overnight tasks cost less than holding an interactive IDE session open for the same work, and the appropriate mode choice reduces token waste from context accumulation in long interactive sessions.

**Cons:** Mode selection introduces a meta-decision cost: developers must now evaluate not just "what should the agent do" but "which interface is appropriate for this task," and the boundaries between modes are not always obvious for tasks of intermediate complexity. Multi-step tasks often span mode boundaries — a task that starts as a PR review (PR mode) and discovers that a non-trivial refactor is needed (terminal mode) requires a mode switch mid-task, which currently requires manual handoff rather than automatic transition. The four-mode taxonomy also does not account for hybrid interfaces (IDE plugins that spawn terminal sessions, PR agents that trigger cloud runs) that are increasingly common in 2026 tooling, so the taxonomy is a conceptual framework rather than a mutually exclusive classification. Cloud mode agents introduce a new class of operational overhead: monitoring, cost controls, and escalation paths for agents that run unattended for hours, which requires infrastructure investment before the mode is safe to use at scale.

**Suggestions:** CATALOG.md's Dev Workflow and Agent Harnesses categories map the available tools per mode: IDE agents (Cursor, Windsurf, Copilot), terminal harnesses (Claude Code, aider, opencode), PR agents (CodeRabbit, Graphite, custom review bots), and cloud orchestration platforms (Devin, SWE-agent derivatives, GitHub Copilot Workspace) are all inventoried there. WORKFLOW.md's mode-selection guidance belongs in the Plan stage — the first decision in any agentic task framing should be which mode is appropriate for the task's scope, feedback requirements, and latency constraints. For teams starting: map your three most common task types to the four modes and identify which mode each task is currently using versus which mode is structurally appropriate — the gap analysis often reveals that a majority of tasks are running in IDE mode that would perform better and cost less in terminal or cloud mode.

**Spec-Driven Development: Writing the Specification Before the Code**

**What it is:** Spec-Driven Development (SDD) is a structured delivery discipline that treats the specification as the primary artifact of software work — the document from which implementation is derived, verified, and governed — rather than as a precursor to "real" coding. In the agentic context, SDD addresses the specific failure mode that the 2026 practitioner consensus identifies as the most common source of rework: confident, plausible-but-wrong code that solves the wrong problem because the agent was never grounded in a precise specification of the right one. The SDD workflow produces a spec with six defined elements before any implementation begins: (1) outcomes — what the system must do, expressed in verifiable terms; (2) scope boundaries — what is explicitly out of scope; (3) constraints — technical, regulatory, or architectural limits the implementation must honor; (4) prior decisions — architectural choices that are settled and should not be revisited; (5) task breakdown — a sequenced decomposition of the implementation into sub-tasks with explicit acceptance criteria per sub-task; (6) verification criteria — the mechanical tests, linter rules, or integration checks that constitute proof of completion. With this spec in place, the agent's role is reduced from "figure out what to build and build it" to "build exactly this, verified by exactly this" — a much narrower task that produces less hallucination and less review-rework. SDD pairs with a three-role agent pattern — Coordinator (owns the spec and task graph), Implementor (executes sub-tasks), Verifier (checks output against acceptance criteria) — that operationalizes the discipline at multi-agent scale.

**Pros:** Verification criteria are the most important structural element of a well-formed spec because they convert the exit condition from a human judgment call ("does this look right?") into a mechanical check ("do all these tests pass?") — the same transition that loop engineering makes, but at the spec level rather than the execution level. A spec with explicit verification criteria functions as an oracle: the Verifier agent does not need to exercise judgment about whether the Implementor's output is correct, because the spec tells it exactly what "correct" looks like. SDD also prevents a common failure mode in multi-agent systems: architectural drift, where each sub-task's Implementor makes locally rational but globally inconsistent decisions about architecture, API contracts, and data shapes. The spec's constraints and prior-decisions elements establish the global invariants that each sub-task's Implementor is required to honor, and the Verifier catches violations before they accumulate into a refactor-requiring inconsistency. The 2026 Zeroshot and Medium analyses both note that the bottleneck in agentic development is no longer generation speed but review-and-rework cycles — and SDD's verification criteria directly target rework: output that passes the Verifier's criteria does not require human rework for the failure modes the spec anticipated.

**Cons:** Writing a complete SDD spec requires more upfront investment than writing a task description or an issue ticket — and the investment cost is paid before any implementation output is visible, which creates an adoption barrier for developers under delivery pressure. The spec's value is proportional to its completeness: a spec that omits scope boundaries produces Implementors that expand scope; a spec that omits prior decisions produces Implementors that relitigate settled architecture; a spec that omits verification criteria produces Verifiers with nothing to verify against. Incomplete specs are arguably worse than no spec at all because they give the appearance of rigor without the substance, and developers relying on them discover the gap only at verification time. SDD also cannot catch the failure modes it did not anticipate: verification criteria catch what the spec author thought to specify, not what they forgot. Architectural violations and API contract drift that the spec did not anticipate are invisible to the Verifier. The three-role Coordinator/Implementor/Verifier pattern adds the same coordination overhead as any multi-agent architecture — justified for complex, parallelizable tasks but inefficient for simple sequential ones — and the Coordinator's task-decomposition quality determines the entire pipeline's output quality.

WORKFLOW.md's Plan stage is structurally aligned with SDD: the issue-style task description that the Plan stage produces is the entry point for full SDD spec authoring, and teams already practicing structured task framing are one step from adopting SDD by adding scope boundaries, constraints, prior decisions, and verification criteria to their existing task description template. The `skills/setup-workflow/` bootstrap skill in this repo produces a minimum-viable SDD starting point: a CLAUDE.md that establishes the codebase's constraints and prior decisions, and a task-framing template that prompts for acceptance criteria. **Two-Layer Code Review: Mechanical Agent Pass Before Human Judgment**

**What it is:** In the agentic SDLC of mid-2026, code review has bifurcated into two structurally distinct layers — an agent-first mechanical pass and a human judgment pass — and teams that run both layers in sequence consistently outperform teams that run either layer alone. The agent review layer handles the mechanical aspects of code quality: style consistency, pattern compliance, security scanning for known vulnerability classes (injection, auth bypass, secrets in code), performance anti-patterns, test coverage gaps, and spec-conformance checking. The agent review layer is fast, deterministic, and exhaustive within its scope: it checks every line in every file in every PR, at a consistency that human reviewers — who are fatigued, time-constrained, and subject to familiarity bias — structurally cannot match. The human review layer handles the judgment calls: architectural fit (does this change belong in the architecture the team has chosen?), design trade-offs (is this the right abstraction for the problem, or the convenient one?), business logic correctness (does this implement what the product team actually intended?), and emergent risk (does this change, combined with three other recent changes, create a surface the agent did not flag?). The two layers are not redundant — they catch different failure modes — and teams that attempt to use agent review as a substitute for human review (rather than a pre-filter for it) discover empirically that the judgment-call failure modes the agent cannot catch are the ones that produce the most expensive rework.

**Pros:** The agent review pre-filter substantially reduces human reviewer load on the mechanical class of issues: a PR that has already passed an agent review arrives at the human reviewer with style, security, and pattern issues already resolved, which means the human reviewer can spend their entire review budget on the judgment-call issues where human expertise is irreplaceable. This directly addresses the PR review bandwidth gap that Anthropic's 2026 agentic coding data quantifies: AI-generated PRs waiting 4.6× longer in review than human-authored PRs is a symptom of reviewers spending their limited time on mechanical issues in agent-authored code rather than on judgment calls. The agent review layer also introduces a consistency property that human review alone cannot provide: the same security scanning and style enforcement applies to every PR, regardless of which human reviewer is assigned, which time of day it arrives, and how many other PRs are in the queue. This consistency is particularly valuable for security-sensitive code paths, where human reviewer fatigue is the most common source of missed vulnerabilities. For the human reviewer, the practical benefit is focus: reviewing a PR that has already passed mechanical checks is a qualitatively different (and more intellectually engaging) task than reviewing a raw PR with mixed mechanical and judgment issues.

**Cons:** Agent review adds latency to the PR process: a PR that must wait for an agent review pass before human review can begin has a longer total cycle time than a PR sent directly to human review. For small, time-sensitive changes (hotfixes, urgent security patches), the agent review latency may exceed its benefit. Agent reviewers also have a consistent false-positive profile: they flag patterns that match a vulnerability or anti-pattern template but are context-specifically correct, producing review noise that human reviewers must triage before acting on. A high false-positive rate degrades the two-layer model by training reviewers to dismiss agent findings as noise — the opposite of the intended pre-filtering effect — and requires ongoing calibration of the agent reviewer's ruleset to maintain a useful signal-to-noise ratio. The two-layer model also does not eliminate human reviewer judgment calls; it relocates them. Human reviewers still make the architectural, design, and business-logic judgments — and those judgments can still be wrong. An agent review that catches 100% of the mechanical issues while the human review misses an architectural violation produces a codebase that is mechanically clean but structurally wrong, which may be harder to diagnose because the clean surface suppresses the usual heuristic signals that something is amiss.

**Suggestions:** CATALOG.md's Code Review & Quality category inventories both automated review tools and the harness-integrated review skills that implement agent review as a CI step: CodeRabbit, custom review skills (including the `/code-review` skill in this repo), and security scanning tools are all catalogued there. WORKFLOW.md's Review stage maps the two-layer structure: the agent review layer belongs at the automated CI step (gating merge), and the human review layer belongs at the judgment checkpoint before merge approval. For teams starting: the minimum viable two-layer adoption is to install a single automated review agent (CodeRabbit or equivalent) as a required CI check on all PRs, with the human reviewer's mandate explicitly scoped to judgment calls only — architectural fit, business logic, design trade-offs — rather than re-checking the mechanical issues the agent already covered. The explicit scope restriction is important: human reviewers default to re-checking mechanical issues out of habit, which duplicates rather than complements the agent review and does not reduce the review bandwidth gap.

**From Prompt Engineering to Orchestration Design: Interaction Protocols as the Primary Technical Challenge**

**What it is:** The 2026 consensus across the CIO, Backslash, and Augment Code agentic SDLC analyses is that the primary technical challenge in AI-assisted development has shifted from prompt engineering (how to phrase a single instruction to maximize one model's output quality) to orchestration design (how to structure the interaction protocols between multiple specialized agents across a workflow). The shift is architectural: when a development workflow involves a single model responding to a single user, prompt engineering is the primary technical lever — the model's output quality is directly controlled by the input quality. When the workflow involves a Coordinator routing tasks to specialized Implementors, a Verifier evaluating their outputs, and a Review agent checking their PRs, the interaction protocols between those agents — how they share context, how the Coordinator sequences sub-tasks, how the Verifier's verdicts propagate back to the Implementor, how escalation to a human checkpoint is triggered — become the primary determinant of overall workflow output quality. A poorly-structured orchestration with well-crafted individual prompts produces worse outcomes than a well-structured orchestration with average prompts, because the interaction protocol determines which information reaches which agent at which time, and the wrong information at the wrong time causes failures no individual prompt can recover from. The practical manifestation: teams that have invested heavily in prompt engineering for individual agent interactions but not in orchestration architecture are discovering that their workflows do not scale — adding more agents to a poorly-architected orchestration produces coordination failures faster than it produces output gains.

**Pros:** The shift to orchestration design as the primary technical challenge is epistemically accurate — it reflects the actual leverage point in multi-agent workflows — and teams that internalize it make better investment decisions. A team that spends its next 10 engineering hours on prompt refinement for individual agents improves the output of each individual interaction; a team that spends the same 10 hours on orchestration architecture improves the output of every interaction in the workflow by improving the context, sequencing, and feedback loops that govern all of them. Orchestration design is also more durable than prompt engineering: a well-designed orchestration remains effective as models improve (the interaction protocols do not depend on specific model capabilities), while a carefully-optimized prompt for one model version often needs revision when the model is updated. The Google Cloud agentic design pattern documentation identifies the specific architectural choices that determine orchestration quality — how context is preserved and pruned across agent transitions, how task boundaries are defined to prevent scope bleed, how escalation conditions are expressed so agents know when to invoke human judgment — and each of these is a design decision that applies across models and tasks, not a prompt-specific optimization.

**Cons:** Orchestration design is a new discipline without the decade of accumulated practitioner knowledge that prompt engineering has developed, and the tooling for it is significantly less mature. Prompting an individual agent is a one-step operation with immediate feedback; designing an orchestration is a multi-session engineering project whose failure modes emerge only at scale (with multiple parallel agents, complex task graphs, or long-running sessions) and are difficult to diagnose because they manifest as coordination failures rather than individual agent errors. The shift in primary technical challenge also creates a skills gap: developers who have invested heavily in prompt engineering skills find that those skills are less transferable to orchestration design than expected, and orchestration design requires new competencies (workflow architecture, context lifecycle management, escalation protocol design) that most developers have not yet formally studied. The orchestration design discipline is also not independent of prompt quality: a well-designed orchestration with poor individual agent prompts still produces poor output for the individual task each agent handles — orchestration design sets the ceiling on workflow quality, prompt quality sets the floor, and both matter.

Mapping the handoff gaps typically reveals that 80% of coordination failures trace to three or four missing context fields, which are cheaper to fix in the protocol than in individual prompts.
