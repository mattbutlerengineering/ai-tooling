**Loop Engineering: Designing the Feedback Cycle as a First-Class Artifact**

**What it is:** Loop engineering is the 2026 successor to prompt engineering — the discipline of designing the loop an agent runs inside rather than hand-crafting the prompt for a single request. The canonical loop structure that the 2026 practitioner corpus has settled on is observe-decide-act-verify: the agent inspects state (reads files, calls tools, reads test output), selects an action (code change, file write, shell command), executes it, then checks the outcome against a verifiable exit condition before deciding whether to iterate or halt. The build-test-fix cycle is the most common instance in coding contexts: the agent makes a code change, runs the test suite, reads the failure output, patches the most probable cause, and iterates until the suite passes or a maximum-round guard fires — a recursive goal rather than a single completion. By June 2026, a single post describing this framing crossed 6.5 million views in under 24 hours, signaling that the "loop engineering" label crystallized something practitioners had been groping toward: the insight that agentic reliability is a loop design problem, not a prompt quality problem. The key shift is that the loop structure is a designed artifact — teams that define the termination condition, the verification step, the maximum iteration count, and the fallback behavior when the exit condition is not reached in N rounds produce substantially more consistent agent behavior than teams that leave these decisions implicit in the model's emergent judgment. From ReAct (the Reason-Act primitive where the agent alternates between a reasoning trace and a single tool call) to full loop engineering (the enclosing architecture that defines what constitutes done, what constitutes failure, and how the loop communicates with the developer at checkpoints), the shift is from controlling the agent's individual steps to controlling the structure in which those steps execute.

**Pros:** A designed loop is debuggable in a way that a single long agent session is not. When a loop fails or exits with a wrong result, the failure is localized to a specific iteration: the agent's state at the start of the failing round, the action it chose, and the verification step's assessment of the outcome. This localization is absent from monolithic sessions where failure is only visible in the final output and tracing it back to an intermediate cause requires reading the full context window. Loop engineering also separates the iteration logic from the model's judgment: the maximum-round guard, the exit condition, and the checkpoint notification are deterministic constraints that hold regardless of what the model decides during its reasoning trace — which means they can be tuned independently of model selection or prompt content. The self-correction property of well-designed loops is also structurally superior to human mid-session correction: rather than the developer checking in at arbitrary intervals to redirect an errant session, the loop's verification step catches the same failures at defined checkpoints using the same criteria every time, producing a consistent signal rather than a human-judgment-dependent one. By mid-2026, loop design is shipping inside the products rather than requiring custom infrastructure: Claude Code's `/loop` skill, harness tools, and cloud agent platforms expose loop configuration as a first-class parameter, which means teams can adopt loop discipline without building the scaffolding themselves.

**Cons:** Loop engineering introduces a class of failure that single-session work does not have: runaway iteration. A loop with a poorly specified exit condition, a weak maximum-round guard, or an exit condition that can be satisfied by low-quality output that technically passes the test will consume tokens and wall-clock time indefinitely or terminate prematurely with a wrong result. The build-test-fix loop specifically can enter a pathological state where the agent keeps patching the same code in circles — each fix makes the immediately failing test pass but breaks a passing test, and the loop oscillates without converging — a failure mode that requires either a human interrupt or a sophisticated loop design that detects oscillation (e.g., comparing the state hash at the start of each round and exiting if it repeats). The verification step is the most critical design decision and also the most expensive to get right: an exit condition that is too strict will never terminate a loop that should succeed; one that is too lenient will exit early with output that does not meet the actual goal. For tasks where the exit condition is qualitative rather than binary (e.g., "the code is idiomatic and maintainable" rather than "the tests pass"), loop engineering provides less leverage than for tasks with binary verifiable exit conditions, because the verification step requires a model call rather than a deterministic check.

WORKFLOW.md's Verify stage is where the loop's exit condition and verification step live: defining what constitutes a passing Verify step is equivalent to defining the loop's termination condition, and teams that invest in the Verify stage produce loops that terminate reliably at the right condition. This repo's `/loop` skill (`.claude/skills/loop-engineering`) is directly relevant: it implements the scheduled iteration pattern for recurring tasks, and its design — prompt, interval, and termination via omission — is a minimal instantiation of the loop-as-artifact principle applied to workflow automation. For teams starting: the minimum viable loop is a two-phase structure applied to the team's most common agentic coding task — a build-test phase that runs until tests pass (or a round limit fires) and a review-gate phase where the developer inspects the output before accepting it — with the exit conditions written down before the first run and treated as constraints on the session rather than as aspirational outcomes.

**Delegation Scope Calibration: The 0–20% Full Delegation Finding**

**What it is:** Anthropic's 2026 Agentic Coding Trends Report produces a finding that has become the most-cited statistic in the 2026 developer practitioner corpus: developers now use AI for roughly 60% of their work, yet can only fully delegate 0–20% of tasks to agents without active oversight during execution. The gap — 60% involvement, 20% full delegation — is the defining tension in AI-assisted development in mid-2026. The tasks that can be fully delegated share an observable cluster of properties: they are verifiable by automated tests (so the agent's exit condition is deterministic), low-stakes in failure consequence (so a wrong output can be discarded without cost), and self-contained with no unresolved external dependencies or open architectural questions. The tasks that cannot be fully delegated share a complementary cluster: they require design judgment about tradeoffs that no spec has fully articulated, domain knowledge about the failure modes the code must handle, or decisions about API contract changes that affect other parts of the system beyond the agent's context window. The L2→L3→L4 autonomy ladder that the 2026 practitioner corpus has converged on maps this calibration into a structured adoption path: L2 (single-step tasks with human-reviewed output — the current reliable baseline for most teams); L3 (multi-step tasks with checkpoint review — achievable with SDD and loop engineering investment); L4 (full autonomous execution with deferred review — viable only for a well-defined subset of task types where evaluation infrastructure is in place). Incremental laddering — starting at L2, building evaluation confidence by tracking failure rates across repeated delegation, and expanding to L3 only when the L2 failure rate is consistently below the team's acceptable threshold — is the pattern that consistently produces durable results. Teams that attempt L4 autonomy before they have built the task taxonomy and evaluation infrastructure to distinguish L4-safe tasks from L4-unsafe ones accumulate rework and review debt faster than they accumulate delegation speed.

**Pros:** The 0–20% finding reframes the question teams ask about AI delegation: instead of "can we delegate this?" (binary), the right question is "where does this task sit on the L2–L4 ladder, and do we have the evaluation infrastructure to support it at that level?" This reframing produces more operational clarity than either optimism ("agents can handle anything") or pessimism ("agents always need supervision"): it identifies a specific investment — evaluation infrastructure — that unlocks delegation at higher autonomy levels for specific task types, rather than making delegation a general capability that either works or doesn't. The cluster of properties that characterize L4-safe tasks — deterministic verifiability, low blast radius, bounded context — also doubles as a quality signal for task framing: engineers who write tasks with explicit acceptance tests (making them verifiable), bounded scope (limiting blast radius), and explicit context provided up front (eliminating external dependencies) are both better-scoped tasks for human engineers and better-delegated tasks for agents. The iterative calibration across L2→L3→L4 also produces the evaluation data that teams need to make model routing decisions: once a team knows which task types succeed at L2 with a given model, they have the empirical basis for deciding whether to route those tasks to a cheaper model tier without quality regression.

**Cons:** The 0–20% full delegation ceiling creates a difficult planning problem: the 80% of AI-assisted work that still requires oversight during execution consumes more developer attention than naive adoption projections assume. Teams that plan their capacity model around "60% of work delegated to agents = 60% of developer time freed" systematically underestimate the oversight load that accompanies partial delegation — monitoring sessions, intervening at decision points, reviewing intermediate outputs, and correcting course mid-session are all developer-time costs that do not appear in the delegation percentage but do appear in the time budget. The L2→L3→L4 ladder also obscures the fact that the delegation ceiling for any given task type is not fixed: it is a function of the team's evaluation infrastructure, the clarity of the task specification, and the quality of the verification tooling in place. A task that is L4-safe with a comprehensive test suite and explicit architectural constraints may be L2-only without them — which means the investment required to move a task class up the ladder may be larger than the delegation throughput gain justifies for low-frequency task types. Teams evaluating the economics of investing in L3/L4 infrastructure for a specific task class should weigh frequency, task duration, and blast-radius-of-failure against the investment required to build verifiable exit conditions at the appropriate level.

**Suggestions:** WORKFLOW.md's Plan stage is where delegation scope decisions belong: the question "is this task L2, L3, or L4?" is a Plan-stage decision that should be made before the agent session begins, based on the task's verifiability, scope, and the team's current evaluation confidence. CATALOG.md's Dev Workflow category inventories tools that support task framing with explicit acceptance criteria and scope bounds — issue trackers with structured templates and lightweight project management tools that enforce the task-scoping discipline that L3/L4 delegation requires. The 2026 Agentic Coding Trends Report itself (linked in Sources) is the primary reference for the 0–20% finding and the L2–L4 taxonomy; teams calibrating their own delegation ceiling should use its task-classification framework as a starting taxonomy and adapt it to their codebase's properties. For teams starting: the minimum viable delegation calibration is a two-week logging exercise — for every AI delegation during a sprint, record the task type, the autonomy level attempted, and whether the output required human correction before use — then use the failure rate by task type to identify which task classes are ready for L3 and which need more evaluation investment before moving up the ladder.

**Procedural Memory and Cross-Session Context Persistence**

**What it is:** The 2026 benchmark finding from the context engineering practitioner corpus is that over 70% of errors in production LLM applications stem not from model capability gaps but from incomplete, irrelevant, or poorly structured context — which means context engineering (the discipline of architecting the full information environment an agent operates in, not just its prompt) is the primary quality leverage point ahead of model selection or prompt refinement. The specific context gap that production coding workflows expose most acutely is procedural memory: the agent needs to know how things are done in this codebase — the team's coding conventions, tool-use habits, review patterns, deployment procedures, and the failure modes that have caused rework in prior sessions — not just what the current task requires. In 2026, two mechanisms address procedural memory at different granularities. The first is static: convention files (CLAUDE.md, AGENTS.md, .cursorrules) that the team maintains by hand, providing a fixed procedural context document that every agent session reads at startup. The second is dynamic: memory systems that persist agent state across sessions, updating the procedural context as the team's conventions evolve and as new violation patterns are observed. The memory infrastructure in 2026 has expanded substantially: 21 frameworks (including LangGraph, which ships with built-in checkpointing that creates an effective "scratchpad" persisting across all steps of a complex workflow), 20 vector stores, and three hosting models (managed cloud, open-source self-hosted, and local MCP) now support cross-session persistent memory. The practical distinction is significant: an agent with no cross-session memory starts every session knowing only what the convention file and the current context window provide; an agent with a persistent memory system can start already knowing the team's most common review objections, which tool calls tend to fail in this environment, and what prior sessions produced — context that would otherwise require either a long and continuously-maintained system prompt or repeated human re-explanation at session start.

**Pros:** Cross-session procedural memory converts agent improvement from a manual update cycle into an adaptive feedback loop. The current baseline — developer observes a violation pattern in an agent's output, adds a rule to CLAUDE.md, and hopes the agent follows it in the next session — requires the developer to both notice the pattern and translate it into the right kind of rule without knowing in advance which rule formulation the agent will actually follow. A memory system that records the violation pattern, the correction applied, and the outcome creates a feedback signal that can inform the next session's context automatically, shifting the update burden from developer-at-rule-time to system-at-session-start. The practical impact on onboarding is also significant: a new team member (human or agent) starting work on a codebase with rich procedural memory has access to the accumulated convention knowledge from prior sessions, which substantially reduces the time spent discovering conventions by violating them. For teams running parallel agent fleets, shared procedural memory across agents eliminates the convention drift that single-session-isolated agents accumulate: each agent in the fleet reads the same up-to-date procedural context, rather than each running against a convention file that may be stale relative to what yesterday's fleet discovered.

**Cons:** Persistent memory systems introduce a failure mode absent from static convention files: memory poisoning, where an incorrect decision made in a prior session is recorded and propagated to subsequent sessions. A static CLAUDE.md can be corrected with a single file edit; an incorrect memory that has been reinforced across multiple sessions may require active identification and explicit correction of the stored state. The volume problem is also real: a memory system that records everything accumulates a retrieval problem as the corpus grows — relevant procedural memories must be retrieved efficiently from an expanding store, and retrieval quality degrades if the memory organization does not scale. Teams using vector-store-based memory for procedural context also face a specificity trade-off: the same semantic similarity that retrieves relevant memories can retrieve superficially similar but contextually wrong memories ("use async/await" retrieving a memory about Python async patterns when the current task is JavaScript async), producing false context confidence. Convention files have the advantage of being human-auditable: every rule a developer adds to CLAUDE.md is visible, intentional, and removable — properties that memory systems must work to preserve with explicit audit tooling. The infrastructure cost of self-hosted memory systems is also non-trivial for small teams: managed cloud options reduce this, but they introduce data residency considerations for teams working on proprietary codebases.

**Suggestions:** CATALOG.md's Memory & Context category is directly relevant: it inventories the tools that provide cross-session persistence, vector store integration, and session state management — the infrastructure layer that enables the procedural memory pattern at production scale. WORKFLOW.md's Reflect stage is where procedural memory updates should be triggered: the pattern is Implement → Verify → Review (PR catches violation patterns) → Reflect (update the procedural memory — either the convention file or the memory system — with the new pattern and the correction). This repo's convention files (CLAUDE.md itself, the `skills/setup-workflow/` bootstrap skill) instantiate the static end of the spectrum: they are the manually-maintained procedural memory that agent sessions in this repo read at startup. For teams starting: the minimum viable procedural memory investment is a two-part convention file update discipline — after every sprint, extract the three most common reviewer corrections on AI-generated PRs, add them to CLAUDE.md with explicit correct/incorrect examples, and delete any rules that have not been triggered in 60 days. This maintains the file's signal density (avoiding the accumulation of stale rules that dilute the relevant ones) and keeps the static memory layer as effective as a manual process can be, before the team has evidence to justify a dynamic memory system.

**The Three-Layer Dev Stack: Routing AI Assistance to the Right Execution Environment**

**What it is:** The 2026 practitioner consensus on the operational infrastructure for AI-assisted development has converged on a three-layer stack that allocates AI assistance to the execution environment matched to each task's required interactivity and time horizon, rather than routing all AI work through a single interface. The three layers as the 2026 corpus describes them: Layer 1 is the IDE — real-time, in-context collaboration where the developer is fully present, providing immediate feedback, redirecting the agent at every step, and integrating suggestions at the line or function level; this is the mode for exploratory work, complex design decisions, and anything where the developer's judgment must be continuously in the loop. Layer 2 is the CLI — local execution of bounded, scoped tasks against the full repository, where the developer delegates a well-framed task (typically matching the L2/L3 autonomy classification described above) and returns to review the output; this is the mode for implementation tasks with clear acceptance criteria, refactors against a spec, and test writing. Layer 3 is cloud agents — asynchronous execution of longer-horizon tasks in dedicated cloud environments with persistent state, tool access, and cross-session memory, typically assigned via GitHub Issues and reviewed the next morning; this is the mode for overnight backlog drain, cross-repo changes, documentation updates, and tasks that would occupy a developer's machine for hours. The routing decision between layers is made at task-framing time based on two properties: required interactivity (continuous → Layer 1; bounded checkpoint → Layer 2; deferred → Layer 3) and time horizon (minutes → Layer 1; tens of minutes → Layer 2; hours to overnight → Layer 3). The three layers are complementary, not alternatives: the 2026 developer's AI workflow typically uses all three in a single workday, with the routing discipline determining which tasks go where.

**Pros:** The three-layer framing converts the common question "which AI tool should I use?" into the operationally clearer question "which layer does this task belong to?" — a routing decision the developer can make quickly at task-framing time without evaluating tools against each other. Layer assignment also creates natural accountability separation: Layer 1 output is reviewed in real time (the developer is watching), Layer 2 output is reviewed at session end (PR review), and Layer 3 output is reviewed the next morning (asynchronous PR audit) — each layer has a defined review moment, and the review bandwidth required scales with the layer number rather than with the total volume of AI work being produced. The three layers also have natural cost structures that align with the three-tier model routing pattern covered in prior runs: Layer 1 benefits from the frontier model (the developer is present to guide correction if the reasoning is wrong), Layer 2 is well-served by mid-tier models (the output has a deterministic verification step, so model reasoning quality is less critical than throughput), and Layer 3 is the natural fit for overnight fleet execution at scale with cheaper model tiers for the high-volume mechanical steps. Cloud agents specifically enable a task category that single-session local work cannot: tasks that require more wall-clock time than a developer can leave a session unattended, or that require coordinating changes across multiple repositories in sequence.

**Cons:** The three-layer stack requires more tooling investment than single-layer IDE-only workflows: teams using all three layers must configure and maintain IDE integrations (Cursor, VS Code extensions), CLI agent tools (Claude Code, Aider), and cloud agent platforms (GitHub Copilot Workspace, Claude for GitHub, Devin, or similar), with authentication, permission scoping, and audit logging configured across all three. The tooling proliferation also creates context fragmentation: a task that starts in Layer 2 and requires a design decision that the developer wants to make interactively in Layer 1 requires either an explicit handoff protocol or the developer re-importing the Layer 2 context into their Layer 1 session, which is not always seamless. Layer 3's asynchronous review model — output reviewed the next morning — creates a specific failure mode: tasks that have unresolved architectural questions at delegation time produce overnight output that requires substantial rework, with the developer not discovering the problem until the next morning after the agent has spent hours on the wrong approach. The overnight-backlog-drain pattern specifically requires a high-quality task queue: pre-scoped, self-contained, L3-or-L4-safe tasks that can execute without decision-point interruption. Teams that populate the Layer 3 queue with under-scoped tasks discover the failure mode the hard way, wasting cloud agent compute on output they discard entirely.

CATALOG.md's Dev Workflow category covers the IDE integration layer and the task management tools that feed the Layer 3 queue. WORKFLOW.md's inner loop maps to Layers 1 and 2, and the outer loop — the PR review, merge, and reflection cycle — maps to the review step that follows Layer 3 execution. The `skills/audit-workflow/` skill in this repo audits the team's installed tool stack against the recommended configuration, and the three-layer structure is a useful lens for identifying gaps: a team using only Layer 1 tools is missing the bounded-task and overnight-delegation modes that produce the highest-leverage AI throughput, while a team invested in Layer 3 infrastructure without Layer 2 CLI discipline is skipping the evaluation feedback loop that makes Layer 3 task quality reliable. For teams starting: the minimum viable three-layer adoption is to add Layer 2 (a CLI agent tool for bounded local tasks) to an existing Layer 1 (IDE) workflow and treat it as a separate tool for a different task class, rather than as a replacement for the IDE workflow — the discipline of choosing one over the other for a given task is itself the skill being built, and it develops faster if the two layers are kept visibly distinct in the team's mental model.
