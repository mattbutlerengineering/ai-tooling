**Spec-Driven Development (SDD)**

**What it is:** Write a markdown spec first — requirements, acceptance criteria, edge cases — then hand it to the agent as its north star. The agent generates a plan from the spec, implements against the plan, and a validation pass compares the output against the original spec. Spec Kit (90k GitHub stars by mid-2026, 30+ agent integrations) is the dominant toolchain; Claude Code's own tutorial documents the four-phase loop: Specify → Plan → Implement → Validate.

**Pros:** The spec becomes the long-lived artifact even as the code evolves, so "re-implement with a better model" is a one-command upgrade path. Acceptance criteria written before implementation double as test cases, closing the loop between intent and verification. Reduces mid-implementation drift — the agent has something to argue against, not just a vague prompt. Works well for overnight runs where you need a defined success criterion the agent can check itself against.

**Cons:** Writing a good spec is as hard as writing good requirements — thin specs produce the same vague output as a bare prompt session. No tool yet automatically verifies that implementation matches spec without human review; the validation step still relies on the same agent that did the implementation. Adds ceremony that doesn't pay off for one-off or throwaway tasks. SDD papers (arXiv 2602.00180) note that "spec quality variance" is the single biggest predictor of implementation quality — a bad spec is worse than no spec.

**Suggestions:** Run graphify or codegraph (CATALOG.md → Code Understanding) over the codebase before writing the spec — a spec written blind often targets the wrong abstractions. Use context7 (WORKFLOW.md → Implement) to keep API references in the spec current rather than training-data stale. The superpowers plugin (CATALOG.md → Agent Harnesses) ships a specify→plan skill that automates the handoff. For overnight runs, pair with the verification pattern below: the spec becomes the checklist the autonomous agent validates against before exiting.

**Parallel Agents via Git Worktrees**

**What it is:** Each Claude Code session gets its own `git worktree` — an isolated checkout of the same repo in a separate directory — so multiple agents work on independent tasks without touching the same files. Claude Code's built-in agent view (`claude agents`) auto-moves dispatched sessions into worktrees. For manual setups, dmux or the parallel-worktrees skill handle the plumbing. The `/batch` command goes further: it splits a single large change into up to 30 worktree-isolated subagents, each opening its own PR.

**Pros:** Reported 30-50% acceleration in development cycles among teams with agentic coding infrastructure. Natural isolation means no in-flight merge conflicts during execution — each agent owns a clean slice. Agent view gives a single dispatch-and-monitor screen so you don't lose track of which session is doing what. Sequential bottlenecks (waiting for one PR to land before starting the next) disappear for tasks that don't depend on each other.

**Cons:** Token usage multiplies with every parallel session, and rate limits bite hard if you underestimate the concurrent load. Anthropic's own 2026 Agentic Coding Trends Report says multi-agent "doesn't make sense for 95% of agent-assisted development tasks" — parallel overhead exceeds the benefit for most day-to-day work. Review burden scales with output: four PRs from four agents means four code reviews, not one. Worktrees require careful upfront partitioning — if two agents share a schema or interface definition, one will step on the other.

**Suggestions:** claude-squad and gastown (CATALOG.md → Agent Orchestration) add visibility across sessions. dmux handles lightweight worktree isolation for solo multi-task runs without a full orchestration layer. For automated task dispatch with conflict resolution, agent-orchestrator (CATALOG.md → Agent Orchestration) plans tasks, spawns parallel agents, and handles CI failures autonomously. Reserve parallel runs for tasks that are genuinely independent — anything sharing an interface or data model should be serialized.

**Evaluator-Optimizer (Reflection) Loop**

**What it is:** A two-agent quality loop: the generator agent produces an implementation, response, or analysis; a second evaluator agent critiques it against an explicit checklist (test coverage, correctness, security, style); the generator revises based on the critique. The loop repeats until the evaluator passes or a configured round limit is reached. Now a named primitive in LangGraph, Flowise, and most 2026 agentic frameworks. A key finding from the search results: wrapping a weaker model in a well-designed reflection loop can outperform a stronger model used in zero-shot mode.

**Pros:** Catches a consistent class of errors that generator agents miss about their own output — particularly missing edge cases and under-specified behavior. The evaluator prompt is version-controllable and auditable, making quality standards explicit rather than tacit. Each iteration produces a log of critiques that becomes a record of what the generator changed and why, useful for PR review. The pattern composes: you can chain multiple evaluators (correctness, security, style) without changing the generator.

**Cons:** Cost roughly doubles per loop iteration — three rounds triples the bill. Loops can oscillate between two wrong answers if the evaluator isn't specific enough to break the tie. Evaluator quality is as important as generator quality; a vague evaluator produces generic feedback that the generator ignores. Hard to know in advance how many iterations are enough for a given task type — budget overruns sneak up.

**Suggestions:** superpowers (CATALOG.md → Agent Harnesses) ships a code review skill that serves as a drop-in evaluator. ECC and ruflo both include reflection-style critique loops in their harness configurations. The harness skill (CATALOG.md → Agent Harnesses) generates domain-specific evaluator agents tailored to a codebase's conventions. CodeScene's agentic patterns guide recommends capping reflection at three iterations and requiring the evaluator to cite specific file/line locations in its critique — vague criticism produces vague revisions.

**AGENTS.md as Executable Workflow Policy**

**What it is:** A markdown file (typically named AGENTS.md, living alongside CLAUDE.md) that documents agent policies, MCP tool contracts, and workflow expectations in plain language the agent reads at session start. Where CLAUDE.md covers project setup and code conventions, AGENTS.md covers which MCP tools to invoke in what order, how to combine them into a coherent workflow, and what the agent should and shouldn't do autonomously. The pattern emerged from teams finding that individual MCP tools are discoverable but the workflow connecting them isn't — agents improvise and get it wrong.

**Pros:** Makes agent behavior predictable and reviewable by humans on the team. Turns engineering principles into executable guidance rather than tribal knowledge that lives in senior developers' heads. Survives model updates — the policy is in the file, not the system prompt. Accelerates onboarding a new agent version onto an established project without a re-prompting session. Pairs naturally with SDD: AGENTS.md describes the workflow, the spec describes the task.

**Cons:** No formal schema means agents interpret the file differently across providers and versions. Easy to let it go stale as the MCP toolchain evolves — a stale AGENTS.md is worse than none because agents follow outdated instructions confidently. Can conflict with or duplicate CLAUDE.md if teams don't maintain both deliberately. Writing it well requires mapping the full MCP tool graph upfront, which most teams skip.

**Suggestions:** compound-engineering (CATALOG.md → Agent Harnesses) provides a cross-harness methodology that AGENTS.md is well-suited to document. gstack and ECC (CATALOG.md → Agent Harnesses) both include opinionated starter configurations that can serve as AGENTS.md templates. This repo's WORKFLOW.md already functions as a high-level AGENTS.md — a project-level AGENTS.md would translate it into specific tool invocations for a given codebase's MCP setup.

**Autonomous Overnight / Long-Running Runs**

**What it is:** Send an agent off to execute a large batch of work — codebase-wide migration, bulk refactor, research sweep, test generation — while the developer is away. The agent runs with a pre-defined spec and success criteria, and either halts on unexpected failure or posts a summary to a notification channel on completion. Claude Code routines (scheduled sessions in Anthropic's cloud) and ralph-claude-code (autonomous loop with intelligent exit detection) are the two primary approaches; dynamic workflows (`/workflows`) can fan out into hundreds of cross-checking subagents for large-scale jobs.

**Pros:** Parallelizes human sleep time with agent execution — work that would take days of context-switching can land as a draft PR by morning. Dynamic workflows support massive scale: a 500-file migration or codebase-wide audit that would take a human days can complete in a single overnight session with cross-validation between subagents. Routine scheduling (see Claude Code docs) removes the "I need to remember to kick this off" overhead.

**Cons:** Errors compound without human oversight — a wrong assumption in step 3 propagates through 200 files before anyone sees it. Cost visibility is poor until the session ends; overnight runs can be expensive surprises. Autonomous agents carry elevated permissions and are the top attack surface for 2026 — 48% of cybersecurity professionals now rank agentic AI as their top threat vector, up from a footnote in 2024. Verification checkpoints are hard to define upfront for open-ended tasks, which means "did it work?" often requires the same human effort you were trying to avoid.

**Suggestions:** ralph-claude-code (CATALOG.md → Agent Harnesses) handles autonomous exit detection — the agent knows when to stop rather than running until timeout. CowAgent (CATALOG.md → Agent Orchestration) adds self-evolving memory so overnight runs accumulate knowledge across sessions. For cost observability, pair with the cost-observability tooling evaluated in evaluations/cost-observability.md. The single highest-leverage practice: define success criteria in a spec file (SDD pattern above) before starting the run — without a checkable spec, there's no way to verify success without human review, negating the autonomous benefit.
