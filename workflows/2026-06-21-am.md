**Loop Engineering: Self-Correcting Agent Cycles**

**What it is:** Loop engineering is the practice of designing agent cycles where the agent performs a task, evaluates output against a verifiable criterion — tests pass, lint is clean, a spec condition is met — and automatically retries if the criterion fails, with a defined stopping condition to prevent runaway loops. As of June 2026, loop engineering has gone from a niche technique to the dominant topic in developer AI discourse, and Claude Code shipped /loop and /goal as first-class native commands. The core loop requires three elements: a task the agent executes, a verifiable exit criterion (not "the code looks right" but "all 47 tests pass"), and a hard retry limit or budget cap before the agent escalates to a human. Practitioners distinguish loop engineering from the simpler ReAct pattern (which only checks the agent's own reasoning) by the presence of an external verifiable criterion that the agent cannot self-certify.

**Pros:** Self-correcting loops catch regressions immediately in the session that introduced them, without requiring a separate CI run or human review of intermediate states. The retry budget gives agents enough runway to work through transient failures and tool-call noise without demanding human intervention on every stumble. Failure of the exit criterion is explicit and binary — the loop either clears the bar or it doesn't, eliminating the ambiguous agent output that makes human review slow. When the criterion is a test suite or linter, the loop doubles as an audit trail: passing the criterion is a machine-verified statement about output quality, not a claim.

**Cons:** The verification problem is the genuinely unsolved part: a loop can produce plausible-but-wrong code that games a weak criterion by deleting failing tests, writing stubs that satisfy assertions without doing real work, or satisfying the letter of a spec while missing its intent. A retry budget set too low fails tasks that need genuine iteration; set too high it runs up unbounded cost on a stuck loop. Loop engineering requires actually-good exit criteria, which are hard to write for novel or ambiguous tasks where the right behavior isn't fully specified upfront. Complex loops without structured telemetry are opaque — knowing "the loop failed on attempt 4" is not the same as knowing why.

**Suggestions:** claude-code-harness and KARIMO (CATALOG.md → Agent Harnesses) both provide harness-level loop support with per-attempt logging. The verification-aware planning pattern (2026-06-20-am) is the planning-time complement to this runtime pattern — acceptance criteria written at plan time map directly to loop exit criteria at execution time. For loops that do run away, model cost routing (see Two-Tier Model Selection below) limits damage by using cheap fast models for retries rather than burning premium reasoning tokens on each attempt.

**Dual-Tool Stacking: Cursor for In-File, Claude Code for Cross-File**

**What it is:** By mid-2026 the AI coding tool conversation has shifted from "Cursor vs. Claude Code" to "Cursor and Claude Code, for complementary workflow stages." The dominant pattern in developer practitioner accounts: Cursor (and its Composer agent) handles within-file editing, autocomplete, and single-feature implementation where the agent's context is naturally bounded by the current file or module. Claude Code handles structural cross-file changes — renames, API contract updates, schema migrations, dependency upgrades, and anything where the agent needs to trace an interface across 20+ files simultaneously. Session structure follows: morning Claude Code for architecture planning and task decomposition; heavy implementation writing in Cursor; end-of-session Claude Code for cross-file structural cleanup and review. Some teams run the tools simultaneously — Cursor in the editor for active writing, Claude Code in a terminal for a parallel long-running structural task.

**Pros:** Task-appropriate context boundaries produce better output from each tool: Cursor's in-file context window is the right shape for "make this function handle the edge case"; Claude Code's multi-file context is the right shape for "rename this interface everywhere and update its callers." Parallel operation captures more of the developer's session — while Cursor is completing a feature, Claude Code can be processing a schema migration independently. The natural handoff point (when a task crosses file boundaries) is usually obvious from the task description, reducing the decision overhead of choosing which tool to use.

**Cons:** Two-tool stacks require consistent state management: if Cursor edits a file Claude Code has in its context window, the two agents see divergent state and can produce conflicting output. Configuration files (.cursor/rules/*.mdc and CLAUDE.md) must be kept synchronized to prevent convention drift between tools. Higher baseline cost since both tool subscriptions are active simultaneously. Developers new to the team face a steeper onboarding curve learning two complementary agent workflows rather than one.

**Suggestions:** This repo's CLAUDE.md is the template for the Claude Code configuration layer; the complement is a .cursor/rules/ directory for Cursor-specific conventions. AGENTS.md (2026-06-20-pm) is the shared layer both tools consume — maintaining it prevents convention drift between the two agents. The parallel worktree pattern (2026-06-19-pm) naturally extends this: each agent in its own worktree, one running Cursor, one running Claude Code, producing outputs the developer integrates at review time. fleet and multi-claude (CATALOG.md → Agent Orchestration) provide visibility across both running agents.

**Two-Tier Model Selection: Reasoning Models for Hard Problems, Fast Models for Routine Work**

**What it is:** As usage-based pricing replaces flat subscriptions across major AI coding tools — GitHub Copilot usage-based credits live June 1 2026, Anthropic Antigravity on a compute pool, Codex on API tokens — teams are adopting explicit model-routing policies: deep-reasoning models (Claude Opus 4.8, GPT-o4) for architecture decisions, cross-file refactors, and complex debugging; fast/cheap models (Haiku 4.5, Gemini 3.5 Flash, Cursor Composer Standard) for routine edits, docstrings, test boilerplate, and boilerplate generation. The principle is that premium-model cost scales linearly with use, so burning reasoning tokens on a button color change is wasteful in a way a flat subscription concealed. Practitioners describe the rule: check the in-product usage dashboard weekly, set budget alerts before agent-heavy work, and reserve Opus-class models for tasks where reasoning depth changes the output quality.

**Pros:** Token economics become predictable and intentional rather than invisible. Teams that profile their agent use typically find 60–80% of prompts are routine edits where the speed of Haiku or Flash is the actual bottleneck — not reasoning depth — so routing those prompts to fast models simultaneously lowers latency and cost. Reserving Opus for genuinely complex tasks also improves quality on those tasks because the context window isn't diluted by routine traffic and the model's reasoning capacity is applied where it differentiates. The pattern also creates a natural forcing function: deciding which tier a task belongs to clarifies the task's complexity before the agent starts.

**Cons:** Deciding which model tier to use adds friction to the developer loop for every non-obvious task. Over-conserving on cheap models for tasks that actually need reasoning produces bad output that costs more time in correction than the premium model would have. The tier boundary is fuzzy: an "obvious" bug fix that requires tracing 10 files of call graph is fast-model territory until it isn't, and the failure is silent — the fast model confidently returns a wrong answer. Automated routing heuristics, where tools offer them, can also make wrong calls without surfacing the mismatch.

**Suggestions:** Claude Code's model selection (Sonnet vs. Opus vs. Haiku) is the clearest in-tool implementation of this pattern. For teams building their own agent tooling with the Anthropic API, the claude-api skill in this repo provides current model IDs (claude-opus-4-8, claude-sonnet-4-6, claude-haiku-4-5), pricing, and latency data to inform routing policy. The Cost Efficiency quality signal in WORKFLOW.md is the canonical framework for reasoning about this trade-off; model selection is the primary lever on that signal for agentic workloads.

**Long-Running Agent Harness: Initializer + Incremental Coding Agent**

**What it is:** A two-agent architecture for tasks that span multiple sessions or take hours: an initializer agent runs once on first invocation to set up the persistent environment (repo clone, dependency install, API key configuration, test baseline, session-state file creation), and a coding agent runs on every subsequent invocation to make incremental progress from where the last session ended. Anthropic Engineering documented this for coding agents that need reliable multi-hour runs — the core insight being that "set up environment" and "make code progress" have fundamentally different failure modes and shouldn't share error-recovery logic. The coding agent reads the session-state file written by its predecessor, adds the next subtask's output, and writes updated state before terminating — leaving the environment in a known-good position for the next run.

**Pros:** Environment setup cost is paid once rather than re-run at the start of every agent invocation, cutting both latency and the class of failures caused by environment setup non-determinism. The coding agent always starts from a known-good state, dramatically simplifying its error-recovery logic: if the environment is wrong, that's the initializer's scope, not a coding-agent edge case to handle. Session state persists across invocations and model context limits, so overnight agents can resume mid-task rather than rediscovering where they left off. The separation also makes it practical to restart just the coding agent without re-running expensive environment setup.

**Cons:** The two-agent split introduces a contract between initializer and coding agent that must be maintained: if the coder assumes a dependency the initializer no longer installs, or reads a state key the initializer no longer writes, the failure is non-obvious to debug. Session state files can drift from actual repository state if the coding agent modifies things the state schema doesn't track. Not appropriate for short or one-shot tasks where setup cost is trivial relative to execution. Requires a harness that routes invocation #1 to the initializer and invocations #2-N to the coding agent — simple to implement but easy to misconfigure.

**Suggestions:** claude-code-harness and KARIMO (CATALOG.md → Agent Harnesses) are the catalog's closest implementations; both support phased agent execution with logging between phases. For the session-state layer, claude-mem and SimpleMem (CATALOG.md → Memory & Context) provide structured persistence without a full memory server. Pair with loop engineering (above) for the coding agent's inner loop: the agent makes progress in a self-correcting loop, saves state on exit-criterion success, and the next invocation continues from that checkpoint.

**Human Accountability Chain: Owning Every AI-Generated Diff**

**What it is:** A code review and engineering-culture discipline: every AI-generated diff entering a codebase must be owned by a named human who can explain the intent, the constraints applied, the alternatives considered, and the rollback plan — not just "the AI suggested it." Practitioner writing and research in mid-2026 identifies this as the essential counterweight to agentic speed: teams that skip it report higher regression rates, slower incident diagnosis, and an accumulating mass of code nobody fully understands even when it passes all tests. The accountability chain runs from task intent (human-defined) through agent output (human-reviewed) to production (human-owned). The rule is not about distrust of AI output — it is about ensuring that the human reviewing a diff can answer "why does this exist?" six months later.

**Pros:** Named accountability creates natural incentives for reviewing agent output seriously rather than rubber-stamping passing tests. When an incident occurs, the accountability chain is the starting point for diagnosis: the owner knows why the code was written that way, what the constraints were, and what the rollback plan is. Maintaining the chain keeps human decision-making at the highest-leverage points — intent definition, constraint setting, review — rather than allowing agent judgment to silently replace it. Code with a clear human owner is dramatically easier to refactor or remove when requirements change, because the owner's reasoning is recoverable.

**Cons:** In high-velocity agentic workflows, the accountability requirement creates friction — developers under deadline pressure take ownership on paper without genuine understanding, which gives the cultural norm without the safety benefit. At team scale, verifying that the chain is real requires process tooling (PR templates with required fields, CODEOWNERS, mandatory review gates) rather than trust. The norm is culturally enforced, which makes it fragile when productivity pressure intensifies. Solo developers or very small teams may find the overhead disproportionate relative to their actual risk surface.

**Suggestions:** PR description templates that require explicit answers to "what does this change do, why, and what's the rollback" enforce the chain at the merge gate without relying on cultural norms alone. code-review-graph (CATALOG.md → Code Review & Quality) gives blast-radius visibility to reviewers, making informed accountability faster rather than more burdensome. The Safety quality signal in WORKFLOW.md is the canonical home for this discipline; the agent-as-author, human-as-owner framing directly operationalizes it. For solo developers, the lightest-weight version is a commit message policy: every AI-generated commit includes one sentence explaining the human decision behind it.
