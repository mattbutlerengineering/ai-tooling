**Always/Ask/Never: The Agent Permission Tier Model**

**What it is:** The Always/Ask/Never model classifies every agent action into one of three permission tiers before deployment begins: Always (the agent acts autonomously without confirmation — e.g., creating branches, running tests, reading files), Ask (the agent must pause and request explicit human approval — e.g., pushing to protected branches, deleting files, making API calls that incur cost), and Never (permanently prohibited regardless of instruction, context, or apparent necessity — e.g., dropping production tables, publishing credentials, bypassing security checks). The model emerged in 2026 as teams discovered that the binary "human in the loop / human out of the loop" framing was too coarse for complex, multi-step agentic workflows: some actions in a given task are genuinely low-risk and safe to automate, while others in the same task require a human decision even when everything else runs autonomously. Without a permission tier, developers face a false choice — block the agent at every action (too slow) or grant full autonomy (too risky). The tier encodes the answer to "which actions, specifically, need a human?" as a durable policy rather than an ad-hoc session instruction, making it reproducible across runs and auditable after the fact.

**Pros:** Permission tiers eliminate the most common source of agentic trust failure: developers approving dangerous actions reflexively because they are fatigued by approval prompts for safe operations. When "create a branch" and "force-push to main" both require the same approval gesture, developers stop reading what they are approving; separating Always from Ask means confirmations are reserved for actions that warrant actual attention. The Never tier acts as a circuit breaker that persists even when the agent is mid-task and confident a prohibited action would complete its goal — it prevents the failure mode where in-session reasoning overrides safety constraints established outside the session. Encoding permission tiers in the agent harness configuration (rather than in prompts) makes them visible to auditors and resistant to prompt injection attacks that attempt to elevate the agent's effective permissions mid-run. Teams also report that building the tier list forces the kind of pre-deployment risk analysis that surfaces unanticipated dangerous actions before any code runs.

**Cons:** The classification decision itself is harder than it looks: "write a file" sounds safe (Always), but "write to .github/workflows/" is high-risk (Ask) — drawing the right boundary requires domain knowledge the permission model does not itself provide. Tiers defined at fine action granularity can grow excessively long for agents that use many tools, and overly long permission lists become unmaintained in practice, reverting to the same approval fatigue they were designed to prevent. The model handles uncertainty poorly: an agent that is 80% confident an action is safe has no mechanism to signal that confidence — it can only ask or act, with no gradation between them. Teams new to the framework frequently discover that their Never tier is aspirational rather than enforced — prohibited actions are blocked by policy statement but not by the harness itself, leaving a gap between stated governance and runtime behavior.

**Suggestions:** WORKFLOW.md's inner-loop Verify stage is the natural place to define tier boundaries — actions at the Plan and Implement stages tend to be Always, while Ship-stage actions (push, deploy, merge) tend to be Ask or Never. The agent harness tools in CATALOG.md (Agent Harnesses category) are the implementation layer where tiers become runtime constraints rather than policy documents: hooks and permission settings in `.claude/settings.json` and `CLAUDE.md` encode the tier for each action type. For teams starting the exercise, the minimum viable tier list is three categories of git operations — read-only (Always), branch-level writes (Ask), and protected-branch or force-push operations (Never) — which covers the most consequential agentic risks without requiring a comprehensive action inventory upfront. The audit-workflow skill (skills/audit-workflow/) can surface gaps between stated permission policy and actual harness configuration during periodic reviews.

**Spec as Source of Truth: The Upstream Quality Shift**

**What it is:** In spec-driven development, the specification is not a planning artifact that the code supersedes — it is the authoritative source of truth from which code, tests, and documentation are generated. The practical consequence for quality control is that the highest-leverage review point moves upstream: instead of reviewing AI-generated code for correctness and convention, teams review the spec for completeness, precision, and hidden assumptions before any code is generated. This represents a fundamental reorientation in where developers spend their review attention — a developer who catches a missing edge case in the spec prevents the agent from generating a whole class of buggy code, whereas the same developer catching that gap in a code review is already downstream of the problem: the code exists, tests have been written for the wrong behavior, and fixing it requires re-running the generation step from scratch. The distinction from the planning step in the plan→execute→verify loop is specificity: a planning prompt describes what to build, while a spec describes the full envelope of required behavior — input ranges, error states, interface contracts, performance constraints, and explicitly out-of-scope behaviors — to the level of precision an agent can execute without ambiguity.

**Pros:** Review cycles are shorter when the spec is the artifact under review: a human reading a structured 500-word spec can evaluate completeness in minutes; the same human reviewing 2,000 lines of generated code is doing a lower-value exercise — confirming that the code mechanically implements a spec neither they nor the agent stepped back to verify was correct. Spec review also catches a failure mode unique to AI-generated code: the agent that satisfies the literal prompt while violating an unstated assumption — an assumption that would have been explicit in a well-written spec. Teams that adopt spec-as-source-of-truth report that their review process shifts from "is this code correct?" to "does this spec accurately capture the requirement?" — a question domain experts (PMs, architects, security leads) can answer without reading implementation code, expanding the reviewer population beyond engineers. Versioned specs provide a durable record of intent that survives context loss between sessions and across team members, making the codebase's behavior auditable against documented intent rather than inferred from code that happens to pass tests.

**Cons:** Writing a complete, precise spec is harder than writing a planning prompt, and teams that underspec — producing a document that looks complete but contains ambiguity that lets the agent make consequential choices — get the overhead of the spec-driven process without the quality benefit. Spec review requires a different skill than code review: evaluating whether a spec is *complete* (does it cover all relevant behaviors?) is harder than evaluating whether code is *correct* (does it match observable behavior?) — the former requires anticipation of edge cases that have not yet been tried, not just pattern-matching against known defect classes. Specs also age: a spec written against a stable interface can be rendered incorrect by a dependency change, and code generated from a stale spec passes review while silently implementing outdated behavior. Teams using spec-driven development with parallel agents (separate author and tester agents, for instance) discover that different agents interpret the same ambiguous spec language differently — meaning ambiguity that seemed minor at spec review time produces divergent implementations that must be reconciled in code.

**Suggestions:** WORKFLOW.md's Plan stage is where spec authoring belongs — the transition from spec to code generation is the Plan→Implement handoff, and committing the spec to version control before the agent runs makes that handoff durable. The Code Understanding category in CATALOG.md (CodeSee, ctags-mcp, repomix) surfaces the existing codebase structure that should inform the spec before an agent runs — a spec written without an accurate model of the existing architecture routinely generates code that conflicts with undocumented conventions. For teams starting spec-driven development, the minimum viable spec has five sections: (1) what the component does, (2) what it explicitly does not do, (3) the interface contract (inputs, outputs, error states), (4) performance and security constraints, and (5) acceptance criteria the agent can evaluate programmatically. The evaluate-tool skill (skills/evaluate-tool/) follows a structurally similar template and serves as a working example of how disciplined spec authorship makes an agent task reproducible and auditable.

**A2A Protocol: Agent-to-Agent Interoperability**

**What it is:** The Agent-to-Agent (A2A) protocol is an industry standard that enables AI agents built on different frameworks and models to coordinate directly without a shared orchestrator or bespoke integration layer. Where MCP standardizes how an agent connects to tools and data sources, A2A standardizes how one agent hands off work to another — passing task context, capability declarations, and result expectations in a format any A2A-compliant agent can receive regardless of the underlying framework. By mid-2026, as multi-agent systems moved from research into production (57% of organizations now report deploying multi-step agent workflows), the absence of a coordination standard was the primary barrier to composing specialized agents: a Claude Code review agent and a LangGraph test agent running in the same pipeline required a custom handoff layer written for the specific pair, which did not generalize when a third agent was added. A2A resolves this by making the handoff protocol a standard rather than a per-integration negotiation — an orchestrator agent that speaks A2A can hand off to any receiving agent without knowing its implementation details, and can add new specialized agents to a pipeline without rewriting coordination logic.

**Pros:** Specialized agents from different vendors — a CodeRabbit review agent, a Claude Code implementation agent, a dedicated security scanner — can participate in the same pipeline without a monolithic orchestrator that understands all three. Teams can swap out individual agents in a pipeline (e.g., replacing one code-review agent with a better one) without rewriting the handoff logic, because the A2A protocol abstracts the interface. A2A-compliant agents expose capability declarations — what kinds of tasks they accept, what context formats they expect — so an orchestrator can route tasks dynamically rather than by fixed assignment, enabling load-balancing and fallback across equivalent agents. The protocol also establishes a standard for task provenance: each A2A handoff carries a chain of context showing which agent produced which output, giving multi-agent pipelines the auditability that monolithic single-agent runs have by default.

**Cons:** A2A compliance requires explicit adoption: agents built before the protocol existed need wrapper layers or refactoring to expose A2A interfaces, and many actively-used agents in 2026 predate the standard with no upgrade path on the published roadmap. The protocol standardizes the handoff format, not the semantics — an A2A-compliant agent can receive a task in a well-formed envelope and misinterpret its intent without either agent detecting the misunderstanding; A2A enables coordination but does not guarantee alignment. Interoperability also surfaces divergent capability assumptions: an orchestrator that assumes the receiving agent can execute bash commands will fail silently if the receiving agent is sandboxed, and A2A provides no capability negotiation to catch this mismatch before the task runs. Teams building A2A pipelines for the first time also report unexpected costs from over-granular handoffs — splitting a task into too many specialized agents creates coordination overhead that exceeds the value of specialization for work that a single capable agent could handle end-to-end.

**Suggestions:** The Agent Orchestration category in CATALOG.md (fleet, multi-claude) provides orchestration infrastructure that can serve as the A2A coordination layer for Claude Code-based multi-agent pipelines. The AGENTS.md convention (covered in 2026-06-22-am) is the context file that an A2A-compliant agent reads on pickup — giving it repository conventions and constraints it needs to produce aligned output without relying on the handoff envelope to carry all that context. For teams evaluating A2A adoption, the minimum viable multi-agent pipeline is two agents: an implementation agent and a review agent, with A2A used only for the single handoff between them; adding a third agent before the two-agent pipeline is stable adds coordination complexity without a validated base to build from. The bounded execution discipline (2026-06-22-am) applies inside each agent in an A2A pipeline exactly as it does in a single-agent workflow — A2A standardizes how agents exchange work, not how each agent manages its own execution loop.

**The 4.6x PR Review Problem: Governing Agent-Generated Code**

**What it is:** A 2026 benchmark study (Opsera AI Coding Impact Benchmark Report) found that AI-generated pull requests wait 4.6x longer in code review than human-generated PRs when no governance changes accompany AI adoption — not because the code is lower quality, but because reviewers apply the same per-line scrutiny to agent-generated PRs as to hand-written ones despite those PRs often being larger, spanning more files, and containing more boilerplate-level changes that don't warrant close inspection. The 4.6x delay compounds directly in organizations where code review is already the rate-limiting step in the dev loop: agentic workflows can produce draft implementations in minutes, but if each PR waits two to three days for a human reviewer to start, the productivity gains disappear entirely in the queue. The governance response emerging in 2026 is not to reduce scrutiny but to restructure *where* scrutiny is applied — agent-generated PRs should be smaller (one spec → one PR), labeled with provenance (which agent, from which spec), accompanied by a machine-generated test coverage report, and routed through an automated verification gate before entering the human review queue, so reviewers receive a pre-triaged submission rather than starting from scratch on every diff.

**Pros:** Structuring agent-generated PRs around the spec→implementation→test cycle gives reviewers a fixed artifact to review against — the spec — rather than inferring intent from the diff. Labeling PRs with agent provenance enables calibrated trust: a PR from an agent that has a good track record on similar tasks in a well-understood module warrants a lighter review than one from an agent encountering an unfamiliar codebase area for the first time, and reviewers can apply that calibration explicitly rather than by instinct. Automated verification gates (test runs, static analysis, code health checks, security scans) before human review eliminate a large fraction of the defects that currently consume reviewer attention — reviewers see the PR only after it has been verified, not as an iterative draft to be improved in place. Teams that restructure review around spec provenance and verification gates consistently reduce average review time as PR volume increases, because reviewers are doing higher-leverage work on a more targeted portion of the diff rather than a full re-evaluation.

**Cons:** Restructuring PR review requires changing team workflows that have inertia proportional to team size — the governance changes are straightforward to describe but require sustained engineering leadership buy-in to sustain against the path of least resistance (reviewing AI PRs the same way as human PRs). Smaller, spec-scoped PRs increase merge frequency, which can exacerbate merge conflict rates in high-concurrency development without corresponding improvements to branch strategy. Pre-review verification gates add latency between PR submission and reviewer visibility, which can create perception problems: reviewers waiting for gate results may begin pulling reviews forward to avoid blocking the agent's next iteration, undermining the triage intent of the gate. Teams that over-rely on automated verification also risk developing false confidence that the gate catches everything — verification confirms code correctness against the spec, but does not catch spec-level errors, architectural misfits, or security issues that only experienced human reviewers recognize.

**Suggestions:** WORKFLOW.md's Review and Ship stages are where PR governance changes land — the pre-review verification gate is the transition between them, and spec-scoped PR sizing is a policy enforced at the Review entry point. The CATALOG.md Code Review & Quality category (CodeRabbit, Sourcery, Bito, Codacy) is the agent-checks-agent verification layer that pre-triages PRs before human review queues see them; these are the tools that reduce the 4.6x review delay by eliminating the most common defect classes before reviewer attention is spent on them. The parallel worktree pattern (2026-06-19-pm, 2026-06-21-pm) produces the spec-scoped, single-concern PRs that make governance tractable at scale: each worktree implements one spec, producing one reviewable PR with clear provenance. For teams measuring the impact of governance changes, time from PR open to merge is the leading indicator — tracking it before and after governance adoption is the simplest way to demonstrate ROI from agentic workflow restructuring, and the 4.6x reduction is the benchmark to measure against.
