**Spec-Driven Development as Full-Lifecycle Discipline**

**What it is:** Spec-driven development (SDD) is a workflow structure that replaces ad-hoc prompting with a formal four-phase loop applied across the entire feature lifecycle: Spec → Plan → Tasks → Implement. In the Spec phase, a developer writes a complete behavioral specification — the target behavior, architecture decisions, edge cases, constraints, and what done looks like — before any agent produces code. In the Plan phase, an agent decomposes the spec into a hierarchical execution plan and the developer reviews and approves it. In the Tasks phase, the plan is broken into atomic, independently verifiable units of work. In the Implement phase, agents execute tasks in sequence or parallel, self-correcting by checking each output against the spec rather than against a heuristic sense of "done." The key structural property of the Spec phase is that it is written by a human and reviewed by a human before delegation begins — it is the document that anchors all subsequent agent behavior. Every major SDD framework in 2026 — GitHub Spec Kit, AWS Kiro, OpenSpec, and BMAD — converges on this same four-phase loop under different names, which signals the loop has stabilized as a production-validated structure rather than a vendor preference. The 2026 VentureBeat analysis of enterprise agentic coding practice frames SDD as the prerequisite for scaling agentic development across teams: the spec is the shared contract that allows multiple agents, multiple developers, and multiple sessions to work on the same feature without accumulating behavioral divergence, because every agent checks its output against the same written specification rather than each developer's implicit mental model of what the feature should do.

The self-correction dynamic that SDD enables is distinct from the self-correction in simpler agentic loops. In a loop without a spec, an agent self-corrects against build failures and test failures — it fixes what the tests catch. In a spec-driven loop, an agent self-corrects against the spec — it fixes not just what the tests catch but what the specification describes. The difference matters when tests are incomplete (which they always are): a loop that self-corrects against tests produces code that passes the tests; a loop that self-corrects against the spec produces code that matches the intended behavior, with tests and code both generated to satisfy the spec.

**Pros:** SDD's core advantage is alignment: the spec is written in natural language by a human before any code is produced, which means human intent is injected at the highest-leverage point in the loop — before the loop begins — rather than as corrective feedback after the loop produces misaligned output. The review cost of spec-misaligned code is high because it requires understanding what the agent did, why it diverged from intent, and how to correct the divergence across potentially many files; the cost of writing a clear spec before delegation is lower and paid once. SDD also scales across sessions and team members in a way that session-level prompting does not: the spec file persists, can be updated, and is available to any agent or developer picking up the feature — providing continuity that chat history cannot.

**Cons:** The Spec phase front-loads effort that session-level prompting avoids, and the cost is paid before any implementation output is visible. For exploratory work — where the goal cannot be fully specified in advance because understanding what is possible requires investigating the codebase — the full SDD loop imposes specification overhead that exceeds the benefit: a developer who cannot write acceptance criteria until the exploration is complete cannot benefit from spec-driven self-correction until the exploration is complete. SDD is most valuable when the feature is well-understood in advance and the difficulty is in correct implementation; it is least valuable when the feature is being discovered iteratively. Teams that apply SDD uniformly across all feature types find the loop adds friction on exploratory spikes and misunderstood problems, and calibrating when to use full SDD versus lightweight task framing requires experimentation. The framework convergence (GitHub Spec Kit, Kiro, OpenSpec, BMAD) is also not yet complete: different SDD frameworks use different file formats, different agent integrations, and different Plan-phase conventions, so a team adopting one framework's spec files cannot easily switch to another without reformatting.

CATALOG.md's Agent Harnesses category covers the frameworks that implement SDD's four-phase loop as tool-native infrastructure — the evaluations there record which harnesses consume spec files natively versus requiring prompt-level workarounds. The `skills/setup-workflow/` bootstrap skill generates a CLAUDE.md and initial spec template that captures the minimum SDD structure for new projects. For teams starting: the minimum viable SDD investment is adding one artifact to the existing planning process — a one-page spec file that answers four questions before any implementation prompt is written: (1) what should the feature do, stated in behavioral terms; (2) what architectural constraints apply; (3) what must not change; (4) what are the exact test commands that constitute acceptance. That document, written before any agent prompt is issued, converts the session from direction-based to destination-based and provides the anchor for self-correction loops.

**Multi-Agent Topology Selection and the Orchestration Role Shift**

**What it is:** The 2026 practitioner consensus, articulated across multiple analyses including the Fungies.io orchestration guide and the Augment Code build-vs-buy comparison, is that the primary role transition for senior developers is from code author to agent orchestrator. The 2026 multi-agent landscape has matured to the point where developers are not choosing whether to use multi-agent systems but which topology to use and whether to build it or buy it. The topology choices that matter most in practice are: sequential (agents execute in a chain, each receiving the prior agent's output, suited to pipeline tasks with clear handoff points), concurrent (agents execute in parallel over independent subtasks and outputs are synthesized, suited to fan-out analysis and parallel research), handoff (an orchestrator routes tasks to specialist agents based on task type, suited to workflows where task classification is cheap and specialist quality exceeds generalist quality), and group-chat (agents share a conversation space and self-coordinate on task assignment, suited to collaborative design and review workflows where the emergent coordination is the value). The frameworks shipping these topologies as first-class primitives — LangGraph, CrewAI, AutoGen (with its Magentic-One pattern), and the Claude Code SDK — have converged on the same four-topology vocabulary, which signals the topology space has stabilized for production use. The build-vs-buy decision governing adoption of these frameworks turns on three factors: team technical depth (custom frameworks give more control but require orchestration infrastructure investment), compliance requirements (regulated industries need audit trails and permission boundaries that off-the-shelf platforms provide more readily than custom builds), and time-to-ship (orchestration platforms reduce time to production at the cost of framework lock-in and customization ceiling).

The 5,000+ MCP servers now available as of mid-2026 have materially changed the topology selection calculus. MCP provides a standardized tool-calling interface that allows agents to invoke external capabilities without bespoke integrations — which means the handoff and concurrent topologies that previously required custom routing code can now be assembled from MCP server compositions. An orchestrator directing specialist agents that are themselves MCP-tool consumers is the emerging production architecture, because the MCP layer decouples capability from coordination logic: new capabilities can be added by registering an MCP server without changing the orchestration topology.

**Pros:** Topology selection as an explicit design decision produces systems that are easier to debug, extend, and cost-optimize than systems where topology is an emergent property of an undifferentiated multi-agent configuration. Sequential topologies have bounded failure modes: if agent N fails, the chain terminates with a clear failure point rather than producing partial output from a subset of concurrently running agents. Concurrent topologies have predictable cost profiles: each subtask's cost is estimable in advance, and the total cost is bounded by the sum of subtask costs rather than the open-ended cost of a single agent operating over the full task scope. Handoff topologies enable specialist quality at scale: a routing orchestrator that directs test-generation tasks to a test-specialist agent and security-review tasks to a security-specialist agent produces better output than a generalist agent handling all task types, at the cost of one routing call per task.

**Cons:** Topology selection introduces upfront architectural decisions that are costly to reverse after agent infrastructure is built around them. A team that builds a sequential topology for a pipeline that later requires concurrent execution must refactor the orchestration layer — not the agents themselves, which can usually be reused, but the coordination logic. The concurrent topology's cost efficiency advantage depends on subtask independence: a concurrent topology applied to subtasks with hidden dependencies (where agent A's output must inform agent B's execution, though the dependency is not explicit in the task decomposition) produces output that requires reconciliation passes that are more expensive than a sequential topology would have been. Group-chat topologies are the most opaque: the emergent coordination that makes them valuable for design workflows also makes their behavior non-reproducible across sessions and non-diagnosable when they produce poor output, because there is no explicit routing log to inspect. MCP server composition introduces its own failure modes: a topology built from MCP server calls is only as reliable as the most unreliable MCP server in the composition, and error handling across a multi-server composition requires defensive patterns that are not always surfaced in framework tutorials.

**Suggestions:** CATALOG.md's Agent Orchestration and Agent Harnesses categories inventory the frameworks implementing each topology with their evaluation status — the evaluations distinguish frameworks by their topology support, not just their marketing claims. WORKFLOW.md's inner loop architecture (Plan → Implement → Verify) maps naturally onto the sequential topology for single-agent use and onto the concurrent or handoff topology for multi-agent use — the topology selection decision belongs at the same stage as the loop architecture decision described in the Data Science Dojo loop engineering guide (referenced in CATALOG.md's Agent Harnesses evaluations). For teams starting: the minimum viable topology selection framework is a two-question pre-delegation checklist — (1) are the subtasks independent or do they have data dependencies? (independent → concurrent is safe; dependent → sequential is safer); (2) does task quality vary significantly by task type? (yes → handoff to specialists; no → generalist is sufficient) — which captures the most consequential topology choice in two decisions without requiring full orchestration architecture specification upfront.

**Agentic Design Pattern Selection: Matching Pattern to Task Type**

**What it is:** The production practitioner literature in 2026 has converged on a vocabulary of five foundational agentic design patterns, documented comprehensively in the VDF.ai pattern catalog and the Augment Code pattern guide. Reflection: the agent generates an output, then evaluates its own output against a rubric, and repeats until the output passes self-evaluation — suited to generation tasks where quality criteria are specifiable in natural language but not easily formalized as tests. Tool-use: the agent invokes external tools (file systems, APIs, code execution environments) to gather information or take actions, with the output grounded in real system state rather than model inference — suited to information-gathering and state-modification tasks. Planning: the agent generates a complete action plan before taking any action, enabling human review of the plan before side effects occur — suited to well-specified tasks where human checkpoint before execution is valuable. ReAct: the agent alternates between reasoning about what to do next and executing a step, adapting its action sequence to intermediate observations — suited to exploratory tasks where the full action sequence cannot be determined before the first step. Multi-agent: tasks are decomposed across specialized agents with distinct roles, with coordination handled by an orchestrator — suited to tasks that exceed a single context window, require genuinely parallel execution, or benefit from specialist quality in distinct subtask types. The critical insight from the 2026 production analyses is that these patterns are not alternatives to each other but composable primitives: a practical workflow typically combines multiple patterns (a planning agent that uses tool-use to gather context before generating its plan, a multi-agent system where each specialist agent uses ReAct internally, a critic-refine loop implemented as two agents in a reflection pattern).

The matching failure mode that the 2026 literature most consistently diagnoses is applying a complex pattern to a task that does not require it: a developer who uses a full multi-agent system for a task that a single agent with ReAct would complete more reliably and cheaply. The cost of over-engineering the pattern is real: multi-agent systems have coordination overhead (context passing between agents, synthesis of parallel outputs, error handling across the agent boundary) that a single-agent solution does not; planning agents have plan-generation latency that a ReAct agent does not; reflection loops have iteration overhead that a single-pass generation does not. The matching heuristic that the 2026 analyses surface consistently: start with the simplest pattern that can plausibly satisfy the task quality requirement, and escalate only when the simpler pattern demonstrably fails to reach the quality threshold.

**Pros:** Explicit pattern selection reduces the most common agentic over-engineering failure mode: reaching for a full multi-agent orchestration system when a single ReAct agent is sufficient. Teams that explicitly match pattern to task type develop a calibrated intuition for pattern escalation — they know from prior runs that generation tasks below a certain quality threshold benefit from reflection, that information-gathering tasks require tool-use, and that exploratory tasks require ReAct — which allows them to make the pattern decision in seconds rather than configuring the wrong pattern and discovering its unsuitability from failed runs. The composability of patterns is architecturally significant: a team that understands the five primitives can assemble novel workflows for task types they have never encountered before by composing patterns whose individual behaviors they understand, rather than being limited to workflow configurations they have previously tested.

**Cons:** Pattern selection requires experience with each pattern's failure modes to be calibrated correctly, and the failure modes are not always predictable from first principles. A reflection loop whose self-evaluation rubric is poorly specified produces an agent that iterates indefinitely because the rubric never triggers acceptance — the same failure mode as a poorly specified verifiable goal in a ReAct loop, but with reflection's iteration overhead amplifying the cost. Planning agents' upfront plan quality depends on the completeness of the context available at plan-generation time, which is often incomplete for tasks requiring codebase exploration — the plan is generated from incomplete context and the human review point comes before the agent has the information that would make the plan reviewable. The multi-agent pattern's specialist quality advantage depends on the orchestrator correctly classifying task types, and misclassification — routing a security-sensitive task to a generalist agent instead of a security specialist — produces lower-quality output than a single well-prompted generalist agent would have produced without the routing overhead. The 2026 literature notes that pattern matching is among the harder skills for developers to develop because the failure modes are often only visible in hindsight: the wrong pattern may produce acceptable output on simple task instances and fail only on edge cases that appear at scale.

**Suggestions:** CATALOG.md's Agent Orchestration category inventories the frameworks that implement each design pattern as a configurable primitive, with evaluations distinguishing which patterns each framework supports robustly versus in beta. WORKFLOW.md maps the inner loop stages (Plan, Implement, Verify) to the pattern types that fit each stage: the Plan stage is the natural home for planning-pattern agents, the Implement stage for tool-use and ReAct agents, and the Verify stage for reflection-pattern agents. The VDF.ai pattern catalog (linked in Sources) is the most comprehensive published reference for pattern selection criteria, with 15 named patterns organized by task type and failure mode; the Augment Code pattern guide covers the five foundational patterns with more accessible decision trees. For teams starting: the minimum viable pattern selection practice is a two-level escalation rule applied before each agentic task — (1) default to ReAct with tool-use (the general-purpose pattern that handles most coding tasks); (2) escalate to planning when human plan review is required before side effects; (3) escalate to reflection when output quality is below threshold and the quality criteria are articulable; (4) escalate to multi-agent only when the task demonstrably exceeds a single context window or requires parallel execution — which prevents premature escalation to complex patterns without requiring pattern expertise to apply.

**Security and Permission Architecture in Agentic Workflows**

**What it is:** A Dark Reading poll finding that 48% of cybersecurity professionals consider agentic AI the top attack vector for 2026 has elevated permission architecture from an afterthought to a first-class design concern in production agentic workflows. The structural reason is straightforward: AI coding agents carry elevated permissions by design. An agent that can read and write files, execute code, call APIs, and push to version control has a much larger blast radius than a human developer using the same tools, because an agent can execute many operations in the time a human performs one and because agents make authorization decisions based on model inference rather than the tacit security judgment that human developers apply. The 2026 practitioner consensus across the GoGloby and Virtido workflow analyses applies three principles from production systems engineering to agentic permission design: least privilege (agents receive only the permissions required for the specific task scope, not a superset of permissions that would make any task easier), audit logging (every tool call, file write, and API request is logged with the task context that triggered it, enabling forensic reconstruction of any agentic action), and human-supervised escalation (permission requests that exceed the pre-authorized scope trigger a human checkpoint rather than being auto-approved or auto-denied). The operational patterns 2026 teams are deploying: permission tiers encoded in AGENTS.md (which file paths agents may read vs. write vs. not touch; which APIs agents may call; which shell commands are pre-authorized and which require confirmation), MCP server permission scoping (each MCP server receives a minimal set of credentials rather than full system access), and per-task permission contexts (a security-review agent runs with read-only access; an implementation agent runs with write access to the feature branch but not to main; a deployment agent runs with the specific deployment credentials and nothing else).

The attack surface that 2026 security analyses are most focused on is prompt injection through external data: an agent that fetches a web page, reads a file, or calls an API can receive content crafted to modify the agent's subsequent behavior. A security-unaware agentic pipeline that reads a malicious markdown file could execute embedded instructions that redirect the agent to exfiltrate credentials, modify files outside the task scope, or make API calls the developer did not authorize. The defense is defense in depth: permission restrictions that limit what an injected instruction can actually do (an agent with read-only filesystem access cannot exfiltrate files by writing them to a new location even if injected instructions attempt it), tool-call audit logs that surface anomalous action sequences for human review, and explicit policy rules in AGENTS.md that prohibit specific high-risk actions regardless of instruction source.

**Pros:** Treating agentic permission architecture as a first-class engineering concern — at the same level as test coverage or code review — converts the security risk of agentic development from a residual exposure to a designed-down bounded surface. The least-privilege principle has a direct productivity payoff in addition to its security payoff: a per-task permission context that scopes an agent to the specific files and APIs relevant to the task produces a smaller action space that the agent navigates more reliably, because the agent cannot accidentally make side-affecting calls outside the task scope that complicate debugging or produce unexpected system state. Audit logs have a debugging payoff that matches their forensic value: when an agentic task produces unexpected results, the tool-call log is the primary diagnostic — it answers what the agent actually did, in what order, and what responses it received, which is far more informative than inspecting the final output alone.

**Cons:** Implementing per-task permission contexts adds configuration overhead that frictionless agentic workflows avoid: specifying which files and APIs each task scope can access requires upfront analysis of the task's resource requirements, which adds to the task-framing cost. Permission systems that are too restrictive produce agents that cannot complete legitimate tasks — an agent that encounters a permission boundary while executing a valid task must escalate to a human checkpoint, which adds latency that erodes the throughput advantage of autonomous execution. Calibrating permission tiers requires experimentation: a team that starts with maximally restrictive permissions will encounter frequent legitimate escalations and must iteratively expand the pre-authorized permission set to match real task requirements; a team that starts too permissive gets the productivity benefit but accepts the security exposure. Prompt injection defenses are imperfect: defense-in-depth slows attackers but does not eliminate the risk, and an agent operating with legitimately broad permissions (a deployment agent with production credentials) has a large enough blast radius that even a bounded injection is a significant incident.

**Suggestions:** CATALOG.md's Security & Safety category inventories the security tools that integrate with agentic workflows: static analyzers, permission scanners, and audit-logging infrastructure that operationalize the permission architecture patterns described here. WORKFLOW.md's Ship stage is the natural checkpoint for permission architecture review: before any agent is authorized to push, deploy, or call external APIs, the permission context should be reviewed against the task scope. This repo's CLAUDE.md is a live example of permission-tier encoding in an instruction file: the "Executing actions with care" section specifies which actions require human confirmation, which are pre-authorized, and which are never permitted without explicit instruction — the same structure teams should encode in their own AGENTS.md or CLAUDE.md. For teams starting: the minimum viable agentic security posture is a four-rule AGENTS.md policy — (1) never push to main without human confirmation; (2) never call external APIs not listed in the task description; (3) never read or write files outside the project directory; (4) log every tool call to a session log — which eliminates the highest-impact blast radii without requiring a full permission-tier architecture. These four rules, added to every project's instruction file, capture most of the security benefit of formal permission architecture for the vast majority of agentic coding tasks.
