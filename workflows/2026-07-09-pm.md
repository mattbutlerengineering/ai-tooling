**ReAct + Reflection Self-Correction Loop**

**What it is:** Two agentic execution patterns — ReAct (Reason + Act) and Reflection — are receiving the most practitioner attention in mid-2026 as the mechanisms that close the gap between what an AI agent generates in a single pass and what it needs to produce to meet a behavioral acceptance criterion. They operate at different levels: ReAct governs the micro-cycle of a single execution step, while Reflection governs the macro-cycle of an entire generation attempt. A ReAct agent does not treat code generation as a one-shot operation. Instead it runs a Reason-Act-Observe cycle: it reasons about what action to take next, executes that action (a file edit, a terminal command, a tool call), reads the observable result, and then reasons again about what the result implies before the next step. The cycle continues until the agent concludes the task is complete. The critical property of ReAct that distinguishes it from pure code generation is the Observe step: by reading objective feedback from the environment (test output, compiler errors, linter results) at each cycle, the agent can detect and respond to failures before they compound across the full implementation. Reflection adds a second-order critique layer on top of ReAct: after completing a sequence of ReAct cycles, the agent explicitly evaluates its own output against the original task specification — asking what the acceptance criteria require, whether the current output satisfies them, and what it would change in a revision — then uses that critique as input for a revision pass. The Towards AI design-pattern analysis documents the accuracy impact: Reflection alone can push coding benchmark accuracy from 80% to 91%, and combining Reflection with external verification tooling (a test runner or linter that provides an objective oracle the agent did not author) can deliver accuracy gains exceeding 30 percentage points over a no-Reflection baseline on specific benchmark task sets. The structural reason is that a self-critiquing agent with external verification operates on two distinct feedback channels — the agent's subjective assessment of quality (Reflection) and the environment's objective assessment of correctness (external verification) — which catch non-overlapping classes of errors. A single-pass generation agent has neither channel.

**Pros:** ReAct loops provide a concrete mechanism for agent error recovery without human intervention: an agent that encounters a failing test, a syntax error, or an unexpected file state mid-implementation can reason about the discrepancy, adjust its plan, and continue — rather than producing an output that encodes the failure without acknowledging it. Reflection adds a qualitative review pass that ReAct alone cannot perform: the Observe step in a ReAct cycle reads objective signals (pass/fail, line numbers), while Reflection evaluates subjective quality (whether the behavioral intent of the task is met, even if all observable signals are green). The combination maps cleanly onto the Implement-to-Verify boundary: ReAct during implementation corrects observable errors in the execution loop; Reflection at the boundary catches behavioral intent violations before the implementation reaches code review. Accuracy improvements in the 11-to-30-plus percentage point range across benchmark task sets quantify what is otherwise a qualitative argument — the compounding effect of two complementary feedback channels applied at different loop levels.

**Cons:** ReAct loops lack a natural convergence guarantee: an agent pursuing a structurally wrong implementation approach can loop indefinitely on a persistent test failure, burning token budget without converging, unless a hard iteration limit or a budget cap provides a forced exit. The Reflection step is only as reliable as the agent's ability to evaluate its own output relative to the task specification — an agent that generated a wrong implementation under a systematic misunderstanding of the task will also generate a self-critique that does not identify the misunderstanding, because both the implementation and the critique share the same context. External verification tooling is a necessary complement to Reflection for exactly this reason (the oracle is independent), but teams with thin test coverage cannot realize the full benefit: the external oracle verifies only that existing tests pass, not that behavioral intent is met, which limits the Reflection + verification combination to the quality of the test suite it exercises. Nested Reflection-within-ReAct architectures also introduce latency: each Reflect step is an additional LLM call, and for tasks with many implementation steps the cumulative Reflection overhead can be significant relative to the implementation cost.

**Suggestions:** CATALOG.md's Agent Harnesses and Code Review & Quality categories both include tools that implement ReAct and Reflection natively. WORKFLOW.md's Implement stage is the home for ReAct loop design — the agent's execution cycle during Implement should include an Observe step that reads test and linter output as objective feedback before the next action, not just the agent's subjective assessment. Reflection belongs at the Implement-to-Verify boundary: a self-critique pass against the task's acceptance criteria (from the spec or issue) before the draft is submitted for code review. For teams starting: configure the agent to run the test suite after each implementation pass and return the full test output (not a summary) to the agent as context for the next Reason step. One test run per Reason cycle is the minimum Observe configuration, and it operationalizes the ReAct pattern's core advantage — objective external feedback on each step — without requiring additional infrastructure.

**Planner-Executor-Reviewer Three-Role Architecture**

**What it is:** The CodeRabbit agentic SDLC guide and the VentureBeat enterprise-scale analysis both document the same emergent multi-agent architecture as the dominant production pattern in 2026: a three-role system where the Planner decomposes the task into a specification and execution plan, the Executor implements against the specification, and the Reviewer validates the implementation against the specification before it moves downstream. The architecture's structural insight is role-separation by epistemic function: the Planner operates on uncertain information (what needs to be built, what the correct approach is) and produces a structured artifact that resolves the uncertainty before implementation begins; the Executor operates on resolved information (the Planner's specification) and focuses entirely on producing correct code against a known acceptance criterion; the Reviewer operates on completed information (the Executor's implementation plus the Planner's specification) and catches the gap between what was specified and what was produced. Each agent is specialized for a narrow epistemic function rather than attempting all three, which eliminates the failure mode that afflicts generalist agents: an agent simultaneously planning, implementing, and reviewing its own work cannot maintain the independence between roles that makes the review step useful. The Reviewer's specific contribution distinguishes this architecture from simpler coordinator-worker patterns: unlike a coordinator that assesses whether a worker's output is "good enough," a Reviewer operates against the original specification as an explicit acceptance oracle, which means it can catch architectural violations and API contract drift that unit tests structurally cannot test (because unit tests verify isolated function behavior, not cross-component contracts). The BCMS SDD analysis identifies this as the quality mechanism that makes spec-driven development scale past individual features: the Reviewer's specification-based review catches the integration failures that emerge when multiple Executors implement independently against their sub-task specifications, before those failures compound downstream.

**Pros:** Role specialization produces quality gains that are orthogonal to model size or prompt quality: a three-role architecture with mid-tier models in each role can outperform a single frontier model across all three roles, because the specialization eliminates the context-switching overhead that degrades generalist agent quality at each role boundary. The Reviewer's independence from the Planner and Executor also provides an audit trail: the Reviewer's assessment is a specification-referenced document that records not just whether the implementation passed but which specification requirements it satisfied and which it violated — a record that becomes the basis for the next Planner's refinement if the implementation requires revision. The architecture also scales gracefully: adding a second Executor (for a parallel sub-task) does not require a new Planner or Reviewer — both can handle multiple Executor outputs against the same specification, which is the mechanism that makes parallel agentic implementation coordination tractable without multiplying coordination overhead linearly with the number of Executors.

**Cons:** The Planner-Executor-Reviewer architecture requires a specification quality that most teams do not produce by default: a Planner that receives an under-specified task produces an under-specified execution plan, and the Reviewer's specification-based validation is only as useful as the specification's acceptance criteria. A Reviewer operating against a weak specification can confirm that the Executor's output is consistent with the specification without confirming that either the specification or the output is correct — the review validates the internal consistency of the three-role system, not its alignment with the actual business requirement. The architecture also adds coordination overhead: each role transition involves a structured handoff artifact (Planner → specification document → Executor; Executor → implementation → Reviewer → review report → Planner for revision), and designing these handoff formats is non-trivial engineering work that is not required for single-agent approaches. Teams that adopt the three-role architecture without engineering the handoff formats find that the coordination overhead consumes the quality gains: a Reviewer that receives a poorly-structured implementation artifact cannot produce a structured review report that the next Planner can act on.

**Suggestions:** CATALOG.md's Agent Orchestration and Agent Harnesses categories inventory the frameworks that implement role-specialized multi-agent architectures — LangGraph and CrewAI both support explicit role assignment and structured handoff artifacts, which are the infrastructure primitives this pattern requires. WORKFLOW.md's Plan, Implement, and Review stages map directly onto the three roles: the Plan stage is the Planner's output, the Implement stage is the Executor's output, and the Review stage is the Reviewer's validation — which means the three-role architecture is a formalization of the WORKFLOW.md stage sequence into an agent role assignment rather than a human-managed sequence. For teams starting: the minimum viable three-role implementation is a human-authored specification (Plan stage output, written as a numbered list of acceptance criteria) plus two sequential agents — an Executor that implements against the specification and a Reviewer that evaluates the implementation against each criterion in turn, producing a pass/fail assessment per criterion rather than a holistic code review. One human-authored spec plus two agents eliminates the Planner complexity while providing the Reviewer independence that is the architecture's primary quality contribution.

**Long-Running Session Architecture and Context Lifecycle**

**What it is:** The O-mega.ai long-running agent guide and Zylos Research's context window management analysis both document a structural shift in how practitioners are designing AI coding sessions in mid-2026: from short, bounded interactions (four-minute average session duration circa 2025) toward multi-hour or multi-day autonomous execution (52-hour sessions documented after Claude Code's /goal command shipped in May 2026). The shift is not simply a duration increase — it is a qualitative change in the design problems the practitioner must solve. A four-minute session fits comfortably in a single context window, and the developer reviews the complete output before it drifts from the original intent. A 52-hour session requires the agent to maintain coherent state across a problem space that is orders of magnitude larger than any current context window, across likely multiple context resets, with a developer review boundary that is hours rather than seconds from the moment the agent began accumulating context. Three distinct context management architectures have emerged for handling this: manual explicit context (@mention syntax, as in Cursor's model), where the developer specifies which files, folders, and documentation should be in the context at each step — providing fine-grained control at the cost of requiring the developer to anticipate what context the agent needs before each action; automatic semantic retrieval (as in Windsurf's Cascade Flows model), where the agent indexes the codebase semantically and pulls relevant snippets automatically based on the current task state — reducing developer overhead at the cost of opacity about what the agent is actually using; and checkpoint-based session persistence (as in LangGraph's first-class checkpointing primitive), where the agent's full state is serialized at each step, enabling time-travel debugging, human-in-the-loop interruption at any checkpoint, fault recovery from any persisted state, and long-term memory that persists across session boundaries and is shareable across parallel threads. The arXiv analysis of the shift to agentic AI (drawing on Codex usage data) corroborates the aggregate session-length trend as a structural feature of the 2026 AI coding landscape rather than a platform-specific artifact: the shift to longer sessions and multi-file edits is visible across the agent platform category, not just in early-adopter usage.

**Pros:** Long-running sessions eliminate the principal source of productivity loss in short-session workflows: the context-rebuild overhead at the start of each session. A developer who resumes a four-minute session after an interruption must reconstruct the agent's context from scratch — re-loading files, re-providing the task specification, re-establishing the constraints the previous session had established. A long-running session with checkpoint-based persistence eliminates this overhead: the developer can interrupt the session, switch to another task, and resume from the exact state the session was in at the interruption point. The 52-hour completion claim from the /goal command data also demonstrates that the capability class has shifted from "a long coding session" to "a coordinated multi-day implementation" — the kind of work that previously required a developer to maintain context across multiple days is now executable as a single agent session with persistence, which changes the economic calculation for large feature implementations. Automatic semantic context retrieval (Windsurf's RAG-based approach) has a compounding advantage for long sessions: as the session progresses, the semantic index becomes more precisely informative about which parts of the codebase are relevant to the current implementation state, because the implementation itself changes the relevance landscape.

**Cons:** Long-running sessions accumulate context rot at a rate that short sessions do not: an agent that has been running for 40 hours has filled its context window multiple times, which means the earliest context (often the most important: the original task specification, the initial architectural decisions, the constraints established at session start) is the most degraded. Context window management architectures that do not explicitly address this degradation — particularly manual explicit context approaches that were designed for short sessions — produce implementations that drift from the original specification as the session lengthens, because the specification is no longer reliably in the agent's active context. The /goal command's 52-hour completion claim also represents an outlier case rather than a typical workflow: the failure mode for a 52-hour session that diverges from its specification 30 hours in is dramatically more expensive in terms of token budget, time, and rework than the same divergence in a 4-minute session. Human oversight tiering — the practice of scheduling periodic review checkpoints during long-running sessions — is the structural mitigation, but it requires the developer to interrupt the session at the right moments rather than treating long-running sessions as fully autonomous. Checkpoint-based persistence (LangGraph's model) adds infrastructure complexity that is inappropriate for teams who are not already operating at the scale where that complexity pays for itself.

**Suggestions:** CATALOG.md's Memory & Context and Agent Harnesses categories both include tools that address long-running session architecture — the evaluations distinguish between tools that provide session state persistence (checkpoint-based), tools that provide automatic semantic retrieval (RAG-based), and tools that require manual context management, which is the distinction that determines which architecture is appropriate for a given session duration and team oversight capacity. WORKFLOW.md's outer loop Reflect stage is the correct home for long-session architecture design: the question "how will the agent maintain coherent state over the lifetime of this task" is a Reflect-stage architectural decision, made before the session begins, not a mid-session adjustment. For teams starting with long-running sessions: establish an explicit context lifecycle — an initial context load (task specification, relevant files, architectural constraints), a mid-session human checkpoint (a review of the agent's progress against the specification at a planned interval, not ad hoc), and a context compaction strategy (a summary of completed steps and remaining work written to a persistent file the agent loads at each context reset). The lifecycle does not require persistence infrastructure; it requires discipline about what the agent carries across the session and when a human reviews whether the session is still on track.

**Specs as Executable Validation Gates**

**What it is:** The 2026 practitioner discourse on spec-driven development has converged on a framing that is qualitatively distinct from the traditional understanding of specifications as design documents: the VentureBeat enterprise-scale analysis and the CodeRabbit agentic SDLC guide both characterize the production-grade SDD workflow as one in which the specification is not read by the agent as context but executed against the implementation as a validation gate. The distinction matters operationally: a specification that the agent reads as context shapes the implementation by providing information; a specification that executes as a validation gate shapes the implementation by providing a rejection criterion. In the executable-spec model, the agent does not generate code against a specification and then hope the implementation is correct — it generates code, runs the specification as a validation suite, observes which specification requirements the implementation satisfies and which it violates, and uses the violation report to direct the next implementation pass. This is structurally analogous to TDD but operates at a higher level: a TDD test suite validates function-level behavior (does this function return the expected output for these inputs?); an executable specification validates system-level contracts (does this implementation respect the API contracts, architectural constraints, and component boundaries specified for this feature?). The BCMS SDD analysis identifies two classes of failures that unit tests structurally cannot catch but executable specifications can: API contract drift (an implementation that changes a function's interface in a way that breaks callers, which no unit test for that function will detect) and architectural violations (an implementation that introduces a dependency that the codebase's architecture explicitly prohibits, which no unit test for any individual component will detect). As agentic implementations become larger and multi-file — the 2026 trend toward 78% of sessions involving multi-file edits creates this surface area — both failure classes become more frequent, making the executable specification's cross-component validation increasingly valuable relative to function-level unit tests alone.

**Pros:** Executable specifications provide a validation surface that scales with the agentic implementation scope: a specification that encodes the API contracts and architectural constraints of a feature validates those contracts no matter how many files the agent edits across the implementation, whereas a unit test suite validates the behavior of each edited file in isolation. For multi-file agentic implementations — the dominant session structure in 2026 — this cross-component scope is the relevant validation surface, and the executable specification is the tool that operates at that scope. Enterprise-scale teams benefit additionally from the governance dimension: an executable specification is a versioned artifact that encodes the behavioral contract for a feature, against which any future agentic implementation of that feature (a refactor, a migration, a port) can be validated. The specification becomes a regression gate for the behavioral intent, independent of the specific implementation that produced the initial passing result. Teams running parallel agentic implementations of different sub-tasks benefit from the specification as a coordination mechanism: each Executor implements against its sub-task specification, and the specification's cross-component validation catches integration failures across sub-tasks before they compound.

**Cons:** Writing executable specifications requires a level of precision about API contracts and architectural constraints that most specification authoring practices do not produce. A specification that reads well as a design document (high-level descriptions of component responsibilities, informal examples of expected behavior) is not executable as a validation gate; an executable specification requires formal descriptions of interface contracts, explicit component boundary rules, and deterministic acceptance criteria that can be evaluated mechanically. The gap between what developers typically produce as specifications and what executable specifications require is the adoption barrier that enterprise teams encounter: teams that attempt executable SDD without a dedicated specification authoring practice find that their specifications cannot be executed (because they're ambiguous) or catch only trivial violations (because their constraints aren't precise enough to detect the failures that matter). The VentureBeat enterprise-scale analysis frames this as the primary reason that enterprise agentic coding pilots fail at scale: the specification quality bottleneck emerges when the agentic implementation scope expands past what a single developer can mentally validate, and teams that did not invest in specification rigor before scaling their agentic footprint cannot retrofit it without pausing the expansion. There is also a tooling maturity gap: the tools that execute specifications against implementations are less mature and less standardized than unit test frameworks, which means teams must either build custom validation harnesses or adopt platform-specific SDD tooling that may not integrate with their existing CI infrastructure.

**Suggestions:** CATALOG.md's Code Review & Quality and Dev Workflow categories both include tools at the specification-execution boundary — the evaluations distinguish between tools that implement specification-as-context (agent reads the spec) and tools that implement specification-as-validation-gate (spec executes against the implementation). WORKFLOW.md's Plan and Verify stages are the correct homes for executable spec design: the Plan stage produces the specification as an artifact, and the Verify stage executes the specification as an acceptance gate before code review. The `/verify` skill in this repo operationalizes the Verify stage gate — "drive the affected flow end-to-end" is the informal equivalent of running the executable specification against the implementation, and teams that formalize this with a written specification reduce the skill's reliance on the developer's judgment about what constitutes a complete flow. For teams starting with executable specifications: begin with a single constraint type — API contract drift detection — and write one specification per feature that enumerates the public interfaces the feature must maintain unchanged. Run a script that validates those interfaces against the implementation before any PR is merged. This is a narrower scope than full SDD, but it catches the most damaging agentic failure mode (cross-component contract violations) with a specification artifact that is precise enough to execute (interface signatures are deterministic) and constrained enough to author without a dedicated SDD practice.
