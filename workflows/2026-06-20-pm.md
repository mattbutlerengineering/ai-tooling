**AGENTS.md as Universal Agent Configuration**

**What it is:** A version-controlled markdown file at the repo root that gives AI coding agents their operating context: coding standards, architecture decisions, review checklists, test commands, and codebase conventions. In mid-2026 AGENTS.md has converged into a cross-tool standard read by Claude Code, OpenAI Codex CLI, Cursor, Aider, Devin, Sourcegraph Amp, Google Jules, Zed AI, Continue, Roo Code, Factory Droids, GitHub Copilot, Gemini CLI, Windsurf, and Amazon Q — most tools that previously used proprietary files (CLAUDE.md, .cursorrules) now also consume AGENTS.md. The pattern separates three layers: AGENTS.md for the durable shared layer (what every agent should know about this codebase), session context for the ephemeral task layer (what this agent needs right now), and auto-memory for accumulated session history.

**Pros:** A single well-maintained AGENTS.md file cascades consistent behavior across every AI tool the team uses, eliminating per-tool prompt drift. Configuration-as-code: the guidelines live in version control, change through PRs, and apply to every contributor. Teams can codify not just style rules but architectural invariants ("never use raw SQL outside the data layer"), which AI agents actually respect when trained on the file. Stack Overflow's research found teams with shared AI guidelines produce more consistent output and spend less time correcting tool drift than those relying on per-developer system prompts.

**Cons:** AGENTS.md is only as good as its maintenance discipline — a stale file actively misleads agents about deprecated conventions. Cross-tool compatibility is imperfect: tools layer proprietary extensions on top of the shared standard, so the same file behaves differently in Cursor vs. Claude Code. Verbose AGENTS.md files hit the front-of-context tax: the more they try to cover, the more they displace the actual task context. Most teams start with guidelines but don't close the loop by reviewing whether agents actually followed them.

**Suggestions:** This repo's own CLAUDE.md is a working example of the pattern. For team-wide adoption, pair AGENTS.md with an audit skill that checks whether recent AI outputs violated the stated conventions — WORKFLOW.md's Review stage is the natural home for that gate. claude-mem and claude-subconscious (CATALOG.md → Memory & Context) can inject task-session state on top of the AGENTS.md foundation without bloating the shared file. context7 (WORKFLOW.md → Implement) keeps library-specific conventions current so the AGENTS.md reference stays accurate as dependencies update.

**SDD at Scale: The 30-Framework Ecosystem**

**What it is:** Spec-Driven Development has matured from an experimental pattern into an ecosystem with 30+ named frameworks by mid-2026. Every major AI coding tool has shipped its own SDD flavor: GitHub Spec Kit, AWS Kiro, Claude Code's native plan mode, Cursor, OpenSpec, BMAD, Tessl, Google Antigravity. The core SDD loop is unchanged — write a version-controlled spec that describes what the system should do, derive a phased implementation plan, decompose into atomic tasks, generate code only then — but 2026's key development is the breadth of tooling that now implements it. Andrej Karpathy, who coined "vibe coding" in early 2025, publicly acknowledged a year later that the vibe coding era is ending and that agentic engineering against detailed specs is the successor. Early adopter data from GitHub and AWS reports ~3–10× higher first-pass success rates for AI agents working against a formal spec versus an informal prompt on non-trivial tasks.

**Pros:** The spec is a forcing function for precision: developers who write a weak spec discover the ambiguity before the agent ships wrong code, not after. Spec-first separates "what should this do" (human decision) from "how should this be built" (agent decision), keeping humans at the highest-leverage point in the loop. The spec becomes durable project documentation that outlasts any individual session or model. The 3–10× first-pass improvement compounds: faster first pass means fewer correction loops, lower cost, and less context consumed on repair.

**Cons:** SDD shifts the cognitive work to the front of the loop — writing a good spec is harder than writing a prompt. Teams under time pressure cut spec quality to save time and then pay for it downstream, erasing the first-pass benefit. With 30+ frameworks all named "SDD," there is no canonical toolchain: teams pick one, invest in its idioms, and then face migration cost if it loses momentum. Acceptance criteria in specs are often underspecified — "validate the form" is a spec entry that every agent will satisfy and no agent will handle correctly.

**Suggestions:** The 2026 workflow log from 2026-06-19-am covers SDD foundations; this entry adds the ecosystem-scale framing. KARIMO and claude-code-harness (CATALOG.md → Agent Harnesses) are the catalog's two closest implementations of the spec → plan → execute → review cycle. For generating acceptance criteria that are actually checkable, pair SDD with the verification-aware planning pattern (2026-06-20-am): spec entries become checkpoint criteria directly. design-council (CATALOG.md → Skills & Plugins) can run 11 role-specialized agents over the spec before execution starts, catching ambiguity before it reaches code.

**Developer-as-Orchestrator**

**What it is:** A shift in the senior engineering role from writing code to coordinating teams of AI agents — delegating implementation tasks, defining agent roles, managing inter-agent dependencies, and quality-gating outputs. The pattern is explicit in multi-agent research: "the most effective developers in 2026 are the ones who have made the transition from coding to orchestration — not abandoning technical depth, but adding a coordination layer on top of it." In practice, this looks like a developer managing a dashboard of parallel agents, each in its own git worktree, working on decomposed subtasks: one agent per feature, one per test suite, one per documentation update. The developer's attention moves from the implementation itself to the task decomposition, handoff contracts between agents, and the review gate before any output merges.

**Pros:** A single developer can now advance multiple independent workstreams in parallel — what previously required a 3–5 person team can run with one orchestrator and several agents. Decomposing the work for agent delegation independently improves task design: the coordination overhead forces the developer to articulate dependencies and interfaces that would otherwise stay implicit. Agent outputs are reviewed rather than written, which is faster and often catches a different class of error than writing from scratch. Solo developers can now build and ship complete applications at team velocity without hiring, which is reshaping startup economics.

**Cons:** Orchestration skill requires a clear model of the codebase architecture — developers who skip the architecture step produce tangled task decompositions that agents can't resolve independently. Context sharing between parallel agents is the dominant failure mode: agents that aren't isolated correctly clobber each other's work or produce conflicting output. The coordination overhead itself isn't zero — managing 5 agents can cost more attention than writing 2 features yourself if the tasks aren't genuinely independent. Quality assurance becomes harder at scale: the orchestrator must review more output per cycle, and the failure mode is merging half-reviewed AI work.

**Suggestions:** The parallel worktree pattern (WORKFLOW.md, also 2026-06-19-pm) is the infrastructure layer for this workflow. fleet (CATALOG.md → Agent Orchestration) and multi-claude (CATALOG.md → Agent Orchestration) both provide dashboard views for managing parallel agents in separate worktrees. For quality at scale — the hardest part of orchestration — code-review-graph (CATALOG.md → Code Review & Quality) gives blast-radius visibility per PR so reviewers know exactly what each agent touched. The /workflows fan-out skill can dispatch specialized verification agents per agent output rather than requiring the orchestrator to review everything personally.

**Test Coverage as Agent Runway**

**What it is:** A workflow prerequisite, not a workflow step: before deploying agents at speed, invest in test suite coverage to the point where a failing test unambiguously signals a regression. The research framing from mid-2026 is that "agents fly through projects with a good test suite as a safety net" — a strong test suite is the substrate that converts an agent from a liability into a productivity multiplier. The pattern extends the code health insight (2026-06-20-am) from general code quality to the specific dimension of test coverage, with the causal mechanism made explicit: agents need an automated verdict on correctness or they operate on trust, which breaks at scale. Teams report that the marginal value of the next test doubles when an agent is writing the production code, because the agent will use every passing test as a green light to proceed.

**Pros:** Agents catch their own regressions rather than shipping them to CI — the test loop runs locally in seconds and the agent self-corrects before the output is reviewed. Coverage becomes a shared contract: the developer writes the test that defines correct behavior; the agent writes code until that test passes. Strong test coverage also unlocks faster review: a reviewer looking at agent output can trust the tests as a first-pass check rather than reading every line. The investment is self-reinforcing — agents are good at generating test boilerplate, so the agent can help build the runway itself.

**Cons:** Test coverage alone doesn't prevent specification-level failures: an agent can satisfy all tests while implementing the wrong behavior if the tests don't capture the intent. Coverage metrics (line, branch) don't directly measure "does this test actually catch meaningful regressions." Retrofitting tests onto a legacy codebase is expensive — teams without test discipline face the same chicken-and-egg as code health investment. Agents that write tests for their own code introduce a correlated failure risk: the same misunderstanding that produced wrong code tends to produce tests that pass that wrong code.

**Suggestions:** This directly operationalizes WORKFLOW.md's Verify stage. For coverage measurement, coverage-py and istanbul (referenced in CATALOG.md → Dev Workflow) quantify current coverage and surface gaps. For agent-generated test quality, have a separate reviewer agent check that newly generated tests would actually catch the regression types relevant to the feature, using the human-in-the-loop checkpoint pattern (2026-06-20-am) as the control mechanism. superpowers (CATALOG.md → Agent Harnesses) ships a test-generation skill that can bootstrap coverage in low-test codebases before agent deployment begins.

**Prompt Chaining as the Underrated Default**

**What it is:** Breaking a complex task into a fixed, sequential chain of simpler prompts — each prompt's output becomes the next prompt's input — instead of reaching for multi-agent or fully autonomous agentic loops. Agentic workflow research consistently names prompt chaining as "the simplest agentic pattern and often the right first choice" because it makes control flow explicit and inspectable. A typical chain: (1) a clarification prompt that resolves ambiguity in the task description, (2) a planning prompt that produces a numbered implementation plan, (3) one execution prompt per plan step, (4) a verification prompt that checks output against the original specification. The chain can be implemented in any coding harness as a sequence of tool calls, with no special multi-agent infrastructure required. The key property: each step is deterministic about what it receives and what it is expected to produce.

**Pros:** Chains are debuggable — when a task goes wrong you can isolate which step in the chain produced the bad output. Each step is individually reviewable, making human-in-the-loop checkpoints natural rather than intrusive. No coordination overhead, no inter-agent communication protocol, no shared context management — the simple sequence is the protocol. Faster to set up than multi-agent systems, easier to audit, and less expensive to run since each prompt is narrowly scoped. The pattern composes well with SDD: the spec drives the chain structure, each spec section becomes a chain step.

**Cons:** Fixed chains can't adapt when a step produces unexpected output — without a feedback loop, a wrong step N poisons every subsequent step. Longer chains accumulate errors without a correction mechanism unless verification steps are explicitly inserted. The pattern doesn't exploit parallelism — a chain is inherently sequential, so independent subtasks that could run in parallel must queue. For research or exploratory tasks where the right sequence of steps isn't known in advance, a static chain is the wrong shape; a planning agent is better.

**Suggestions:** This is the entry-level pattern in WORKFLOW.md's agentic progression before reaching for full multi-agent orchestration. claude-code-harness (CATALOG.md → Agent Harnesses) supports explicit chain definition with step-level logging. Pair with verification-aware planning (2026-06-20-am) to add pass/fail checkpoints between chain steps, converting the simple chain into a self-correcting loop without multi-agent overhead. The orchestration pattern (above) is the natural graduation from prompt chaining: once chains reliably produce good outputs, the orchestrator fans them out in parallel.
